GSD 2 深度解读

基于磁盘状态机的自主编程 Agent — TypeScript + Pi SDK
GSD（Get Shit Done）2 是一个运行在终端里的自主编程 Agent，它的核心创新是把整个开发工作流的状态存在磁盘文件而非内存中，从而实现了崩溃恢复、多终端协同和无限长任务执行。读完本文，你将理解它与其他 AI 编程工具的本质差异、如何在项目中落地使用，以及那些让它"聪明"的关键工程决策。

核心结论

  • 磁盘即状态机 — 所有工作流进度存在 .gsd/ 目录，进程崩溃后从磁盘恢复，而不是重头开始
  • 每单元独立上下文 — 每个子任务（plan / execute / verify）都在全新的 200k token 窗口执行，彻底消除上下文污染
  • 声明式调度取代条件分支 — 用规则表（dispatch table）替换 130 行 if-else，"下一步做什么"变得可测试、可扩展
  • 亚毫秒复杂度分级驱动模型路由 — 纯启发式算法在调用 LLM 前判断任务难度，动态分配 Opus / Sonnet / Haiku，平衡质量与成本
  • Rust N-API 加速关键路径 — grep / glob / AST 解析 / git 操作通过 N-API 调用 Rust，绕过 Node.js I/O 瓶颈

1 摘要

1.1 项目背景

随着 LLM 能力提升，"让 AI 写代码"从聊天补全演进为自主 Agent。但现有工具面临两个根本性问题：一是上下文污染——长任务中历史对话越积越多，模型注意力被稀释，代码质量在几十轮后急剧下滑；二是状态易失——所有进度存在内存中，一次网络中断或进程崩溃就意味着一切重来。GSD 2 针对这两个痛点做了系统性设计。

1.2 项目预期

1.2.1 预期一：长任务持续推进能力

传统对话型 AI 工具的有效窗口通常在几十轮内。GSD 的目标是让 Agent 能稳定推进"新增功能模块"这类需要多天工作量的任务，关键实现是：把一个大需求拆解为 Milestone → Slice → Task 三层结构，每个最小单元（Unit）都在独立的全新 LLM 会话中执行，完成后结果写回磁盘，下个单元读磁盘而非历史对话获取上下文。

1.2.2 预期二：进程级可靠性

GSD 的 auto 模式可能持续运行数小时。通过锁文件（crash-recovery）+ JSONL 取证（session-forensics）+ OS 互斥锁（session-lock）三层保护，即使 Node.js 进程崩溃，下次启动时也能精确定位到崩溃位置并提示用户是否继续。

1.2.3 预期三：成本感知的自适应质量

项目内置了一套无需 LLM 调用的复杂度分级器：在调度每个 Unit 前，通过任务元数据启发式地判断 light / standard / heavy，并结合当前 budget 使用率和历史失败率动态调整分配的模型。目标是在预算耗尽前优先完成更多工作，而不是用最贵的模型处理所有任务。

2 项目设计理念

2.1 磁盘驱动状态机（Disk-Driven State Machine）

GSD 最根本的设计选择：所有工作流状态不存在内存里，只存在磁盘上。.gsd/ 目录下的 Markdown 文件（ROADMAP / SLICE / TASK / SUMMARY）加上 SQLite 数据库就是整个系统的唯一真相来源。

这个选择带来了一系列连锁收益：进程可以随时崩溃然后重启；用户可以同时在多个终端查看状态；Git worktree 隔离每个 Milestone 的开发分支；而且整个状态是人类可读的 Markdown，用户随时可以手动编辑干预。代价是磁盘 I/O 变成热路径，这也是引入 Rust 原生引擎的直接原因。

2.2 每单元新鲜会话（Fresh Session Per Unit）

对话型 AI 工具的通病是"历史越长，质量越差"。GSD 的解法是彻底的：每个 Unit（Task / Slice / Milestone 级别）都在一个全新的 LLM 会话（新的 context window）里执行。上一个 Unit 的输出不是通过对话历史传递，而是写成 SUMMARY 文件，下一个 Unit 的 Prompt 里按需 inline 这个文件的内容。

这种设计的精妙之处在于"按需"两字——auto-prompts.ts 里有一套分层上下文注入机制，根据 inline_level（full / references / compressed）和当前 context budget 决定把多少历史内容塞进 Prompt，而不是无脑全部 inline。

2.3 声明式架构理念（竞品横向对比）

GSD 在多个核心模块都选择了声明式（declarative）而非命令式（imperative）的设计：dispatch table 替代 if-else 链、规则化的 TTSR guardrails、schema 驱动的状态类型。这反映了一个设计判断：可观察性和可测试性比"快速写完"更重要。

3 项目使用方式与执行链路

3.1 当前适合解决什么问题

✅ 适合的任务类型

  • 中大型功能开发 — 如"新增用户权限系统"，需要设计、实现、测试多个步骤
  • 无人值守批量推进 — 可以挂在后台运行，完成一个 Slice 后自动继续
  • 需要干净 Git 历史的场景 — 每个 Milestone 一个独立分支，squash-merge 保持主分支整洁
  • 跨终端协同 — 磁盘状态可跨终端查看，多人项目可共享 .gsd/ 目录状态
  • 高可靠性要求的自动化场景 — CI/CD headless 模式、定时任务驱动的自动维护

⚠️ 不太适合的任务类型

  • 五分钟内的小修改 — 创建 Milestone 的开销大于任务本身
  • 完全开放式的探索性研究 — GSD 需要明确的目标和可验证的完成标准
  • 纯设计 / 头脑风暴 — 需要人机高频交互的创意过程不适合自主循环
  • 需要即时视觉反馈的前端调试 — 当前版本对 UI 像素级调整的理解有限

3.2 最小可运行路径

从零开始，用 GSD 完成第一个自主任务只需 5 步：

# 第一步：安装
npm install -g gsd-pi
# 第二步：进入你的项目目录，首次运行会触发 onboarding（配置 LLM API Key）
cd my-project && gsd
# 第三步：创建一个 Milestone（描述你的目标）
/gsd discuss "为用户登录接口添加失败次数限制功能"
# 第四步：启动自主执行
/gsd auto
# 第五步：查看进度（另一个终端或 Web UI）
gsd web # 打开浏览器 Dashboard

4.2 关键模块实现细节

4.2.1 关键模块一：声明式调度管道（auto-dispatch.ts）

📁 src/resources/extensions/gsd/auto-dispatch.ts 🏷️ 规则表模式（Chain of Responsibility 变体）

要解决的问题：GSD 的工作流有 10+ 种 Unit 类型（discuss / plan / execute / uat / complete 等），决定"下一步该执行哪个"需要考虑当前 Milestone 状态、Slice 状态、是否有崩溃残留、预算是否超限……用命令式 if-else 实现会产生一个数百行的怪兽函数，无法单独测试每条规则，添加新规则会引入隐患。

解决方案：把所有调度逻辑提取为一张有序的 DispatchRule[] 数组，每条规则有 name 和 async match 函数。match 返回 DispatchAction（dispatch / stop / skip）或 null（不匹配，继续下一条）。调度引擎只需顺序评估规则表，第一条返回非 null 的规则获胜。

代价与局限：规则的顺序本身就是隐式的业务逻辑，乱序会导致静默错误（如崩溃检测放在预算检测后面会漏掉崩溃场景）。规则条数增多后，调试"为什么选了这条规则"需要靠日志中的 matchedRule 字段追溯。

4.2.2 关键模块二：复杂度分级器（complexity-classifier.ts）

📁 src/resources/extensions/gsd/complexity-classifier.ts 🏷️ 启发式分级 + 自适应学习

要解决的问题：模型路由需要知道"这个 Task 有多难"，但用 LLM 来判断难度有明显的自我矛盾（贵的 LLM 才能判准难度，用了就没省到）。需要一个零 LLM 调用的快速分级器。

解决方案：两阶段判断。第一阶段按 unit 类型给默认分级（plan-milestone = heavy, complete-slice = light）；第二阶段对 execute-task 类型做元数据分析：读取 TASK.md 的文件引用数、依赖数、代码块数量、复杂度关键词，并结合历史失败率做自适应上调；最后用当前 budget 使用率做一次强制降级（90%+ 时 heavy → light）。

代价与局限：启发式规则会有误判——一个关键词很少但实际上修改面很广的重构任务可能被错误分为 light。自适应学习的粒度是 unit 类型级别，无法区分同一类型内不同任务的差异。

4.2.3 关键模块三：崩溃恢复机制（crash-recovery.ts）

📁 src/resources/extensions/gsd/crash-recovery.ts + session-forensics.ts 🏷️ 心跳锁文件 + JSONL 取证

要解决的问题：auto 模式的单个 Unit 执行可能持续数分钟到数十分钟。网络中断、内存溢出、用户强制关闭都会导致进程非正常退出。如何在下次启动时让用户无缝感知到"上次崩溃了"并提供恢复选项？

解决方案：三层嵌套保护。第一层：每次 Unit 开始前在 .gsd/runtime/auto.lock 写入含 PID + sessionFile 路径的 JSON 锁文件，正常退出时删除；下次启动检测到锁文件且 PID 进程已不存在 → 判定崩溃。第二层：从崩溃时的 Pi session JSONL 文件读取已完成的工具调用，重建崩溃前状态摘要（session-forensics）。第三层：OS 级 flock 互斥锁防止并发执行。

sequenceDiagram participant Auto as auto.ts participant Lock as crash-recovery.ts participant OS as session-lock.ts (flock) participant Forensics as session-forensics.ts participant User as 用户终端 Auto->>OS: acquireLock(basePath) OS-->>Auto: 互斥锁获取成功 Auto->>Lock: writeLock(basePath, unitType, unitId, sessionFile) Note over Lock: 写入 .gsd/runtime/auto.lock Auto->>Auto: 执行 Unit（可能崩溃） Note over Auto,Lock: 进程崩溃，锁文件未删除 Auto->>Lock: detectCrash(basePath) Lock->>Lock: 读取锁文件，检查 PID 是否存活 Lock-->>Auto: {crashed: true, unitType, sessionFile} Auto->>Forensics: synthesizeCrashRecovery(sessionFile) Forensics->>Forensics: 解析 JSONL，提取已完成工具调用 Forensics-->>Auto: 崩溃摘要 Auto->>User: "上次在执行 execute-task 时崩溃，是否继续？"
图 4-4: 崩溃恢复三层保护机制

代价与局限：锁文件的 PID 检测在分布式环境（多台机器共享 .gsd/ 目录的场景）下不可靠，但 GSD 主要针对单机开发场景，这个限制可接受。Session JSONL 取证的质量依赖 Pi SDK 的日志完整性。

4.2.4 关键模块四：分层上下文注入（auto-prompts.ts）

📁 src/resources/extensions/gsd/auto-prompts.ts 🏷️ 分层预算分配 + 优雅降级

要解决的问题：每个 Unit 在独立新会话执行，但它需要了解当前任务的上下文——所在 Milestone 的目标、所属 Slice 的计划、关联 Task 的详情。把所有历史文档全部 inline 会超出 context window；什么都不 inline 则 Agent 缺乏背景会走弯路。

解决方案：三档 inline_level 配置（full / references / compressed）+ 30k 字符 preamble 上限的自动裁剪。full 模式把完整 Markdown 文件 inline；references 模式只 inline 文件路径（告诉 Agent "去读这个文件"）；compressed 模式只 inline 文件的关键摘要段落。超出 30k 上限时自动按章节边界截断，而不是硬截字符。

sequenceDiagram participant Dispatch as 调度器 participant Prompts as auto-prompts.ts participant Budget as context-budget.ts participant Files as .gsd/ 磁盘文件 participant Prefs as preferences.ts Dispatch->>Prompts: buildExecuteTaskPrompt(ctx) Prompts->>Prefs: loadEffectiveGSDPreferences() Prefs-->>Prompts: {inline_level: "full", context_window: 200000} Prompts->>Budget: computeBudgets(windowTokens) Budget-->>Prompts: {preambleChars: 30000, inlineContextChars: 60000} Prompts->>Files: loadFile(MILESTONE.md) Files-->>Prompts: milestone 内容 Prompts->>Files: loadFile(SLICE.md) Files-->>Prompts: slice 内容 Prompts->>Prompts: capPreamble(preamble, 30000) Note over Prompts: 按章节边界智能截断，不硬截字符 Prompts-->>Dispatch: 裁剪后的完整 Prompt
图 4-5: 分层上下文注入流程

代价与局限：inline_level = "references" 要求 Agent 有足够的工具调用能力主动去读文件；如果 Agent 不读或读错，任务质量会下降。compressed 模式的摘要质量依赖于 SUMMARY 文件是否准确，而 SUMMARY 本身是之前 Agent 生成的——存在"错误传播"的风险。

5 方案评价

5.1 优势分析

  • 磁盘状态机是真正的差异化：目前市场上很少有工具能做到跨会话、跨终端、跨崩溃的持久状态，这是 GSD 在"长任务可靠执行"场景的根本护城河
  • 声明式调度的工程严谨性：把调度逻辑变成可测试的规则表，每条规则独立，可以 unit test，这在同类工具中罕见
  • 成本感知是一等公民：复杂度分级 + 预算压力降级的组合，让 GSD 在有限预算下可以完成更多工作，不是简单地"永远用最贵的"
  • Rust 原生引擎的务实选择：没有为了"纯 TS" 而坚持，在性能关键路径引入 Rust 是正确的工程判断
  • 扩展架构良好：24 个扩展相互独立，主引擎不依赖具体扩展，添加新能力无需修改核心代码

5.2 局限分析

  • 依赖闭源 Pi SDK：核心 Agent 运行时（会话管理、工具执行、LLM 路由）封装在 @gsd/pi-coding-agent 里，这是一个闭源包，限制了社区贡献和问题排查深度
  • 初始学习曲线陡峭：Milestone / Slice / Task 三层结构对于"只想快速执行一个命令"的用户来说过重；onboarding 文档对新用户不够友好
  • 启发式分级的天花板：复杂度分级器无法感知代码库的全局状态，"修改一个被 50 个地方引用的核心接口"和"新增一个独立工具函数"可能被分到同一级别
  • SUMMARY 质量的链式风险：每个 Slice 的 SUMMARY 由 Agent 自己生成并作为下个 Slice 的上下文，一次质量差的 SUMMARY 会影响后续所有 Slice 的执行质量
  • Git worktree 的操作复杂性：每个 Milestone 一个 worktree 对不熟悉 git worktree 的用户来说是心智负担，worktree 冲突处理不够自动化
