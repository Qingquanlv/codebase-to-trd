🌱 Spec Kit 深度解读

GitHub 出品的规格驱动开发工具包 — "让规格成为真正的源代码"
Spec Kit 是 GitHub 官方开源的 AI 辅助开发工作流框架，87.7k Star，核心理念是 规格驱动开发（SDD）：把 AI 的输入（规格文档）提升为第一公民，让代码成为规格的派生物。 项目本体是一个 Python CLI（specify-cli），其真正的产品价值在于内置的 Markdown 提示词模板和遍布 25+ 主流 AI Agent 的集成适配器。 读完本文，你将理解：它的核心工程决策（提示词即产品）、集成架构（抽象基类+注册表）、 可扩展生态（Extension Hook 系统），以及它巧妙规避的和尚未解决的挑战。

核心结论

  • Spec Kit 的本质是"提示词包管理器"——Python CLI 只是安装器，真正的产品是 Markdown 命令文件。
  • 集成系统用单一抽象基类 IntegrationBase + 注册表支持 25+ AI Agent，新增 Agent 只需一个子类文件。
  • Extension Hook 系统让社区无需 Fork 核心就能在 before/after 每个开发阶段插入自定义逻辑，已孵化 40+ 扩展。
  • v0.6 将所有模板打包进 wheel（air-gapped 可用），并通过原子写入 + 保守深度合并保护用户已有配置。
  • 最大局限：命令提示词本身无法被版本约束强制检查，社区扩展质量参差不齐，大型团队协作规格漂移检测仍依赖外部扩展。

1 摘要

1.1 项目背景

AI 编程工具浪潮中出现了一个新的反模式：Vibe Coding——开发者把模糊的想法直接扔给 AI，让它"随便生成点什么"。 这种方式产生的代码缺乏规格约束，难以维护，更无法在团队间传递意图。与此同时，传统项目中"规格写完就扔"的问题从未解决—— PRD、设计文档与最终代码之间的漂移是行业痼疾。

GitHub 工程团队将这些痛点提炼成规格驱动开发（SDD）方法论，并于 2025 年以 Spec Kit 开源： 一套让 AI Agent 能够结构化地从"想法"走到"代码"的工具链。项目发布后迅速积累 87.7k Star、7.5k Fork， 社区贡献了 40+ 扩展和 8 个预设，成为 AI 辅助开发领域最活跃的工作流框架之一。

1.2 项目预期

1.2.1 预期一：用规格文档取代随意的 Vibe Coding

通过 /speckit.specify 命令强制 AI Agent 在写代码前先生成结构化 PRD（含用户场景、功能需求、验收标准）， 通过 /speckit.plan 生成技术实施方案，通过 /speckit.tasks 拆分为可追踪的任务列表， 最后才由 /speckit.implement 驱动代码生成。每个阶段输出标准 Markdown 文件，版本控制友好。

1.2.2 预期二：让同一套工作流覆盖所有主流 AI Agent

开发者使用的 AI Agent 各异（Claude Code、Copilot、Cursor、Gemini CLI……），Spec Kit 通过 集成适配系统将同一套命令模板部署到各 Agent 的专属命令目录，实现"一次学习，随处使用"。 目前支持 25+ Agent，未来新增只需一个子类文件。

1.2.3 预期三：通过生态扩展满足差异化需求

内置的 Extension Hook 系统和 Preset 机制允许社区在不修改核心代码的情况下： 插入自定义工作流步骤（如 CI Gate、Jira 同步、自动 Review）、替换核心命令为更精简或更复杂的变体（如 Lean 预设）。 40+ 社区扩展覆盖了安全审计、多 Agent 并行、Spec 漂移检测等高级场景。

2 项目设计理念

2.1 规格即源代码

Spec Kit 的核心哲学引用自 spec-driven.md："Code becomes its expression in a particular language and framework. Maintaining software means evolving specifications."——规格是一等公民，代码是派生物。 这在工程上体现为：所有核心命令（/speckit.specify、/speckit.plan 等）都会在项目根目录创建 版本控制友好的 Markdown 文件（specs/001-feature/spec.md），这些文件和代码一起提交、一起 Review、一起演进。

2.2 提示词即产品

Python CLI（src/specify_cli/）只有约 9,000 行代码，职责极为单一：把打包在 wheel 里的 Markdown 提示词文件复制到正确的位置。真正的产品价值在那些 Markdown 文件本身——它们精心设计了 AI 的推理路径、 引导结构、质量校验逻辑。这个决策的好处是：更新提示词不需要改 Python，也不需要重装工具； 代价是：提示词逻辑无法被类型系统或 linter 保障正确性。

2.3 开放生态优先

Spec Kit 刻意将自身定位为"工具套件"（Kit）而非"平台"。 Extension 系统允许任何人发布独立仓库并通过 PR 提交到社区目录； Preset 系统允许完全替换核心命令（如 Lean 预设用 50 行替换 300 行的完整版命令）； --ai generic --ai-commands-dir 选项允许任何未被官方支持的 Agent 接入。 这种设计让社区在 6 个月内贡献了 40+ 扩展，远超核心团队能维护的范围。

2.4 Agent 无关性（Integration Neutrality）

支持 25+ Agent 的核心设计是：每个 Agent 只是对同一套命令模板的不同"部署方式"—— Copilot 部署到 .github/copilot/，Claude Code 部署到 .claude/commands/， Gemini CLI 用 TOML 格式而非 Markdown。 抽象基类 IntegrationBase 定义了统一接口，具体 Agent 的差异被封装在各自的子类里， 主流程代码完全不感知 Agent 差异。

3.2 关键模块实现细节

3.2.1 specify init —— 项目初始化流程

要解决的问题：一条命令完成工具检测、Agent 选择、模板部署、Git 初始化， 且需在 25+ Agent 下行为一致，同时支持全新项目和存量项目（--here 模式）两种场景。

核心决策：使用 StepTracker（Rich Tree 实时渲染）展示安装进度； 所有模板资产打包进 wheel（pyproject.toml 的 force-include），安装后无需网络； VSCode settings.json 使用"礼貌合并"——只添加新 key，不覆盖用户已有值。

def merge_json_files(existing_path: Path, new_content: Any, verbose: bool = False):
    """礼貌深度合并：只添加新 key，保留用户已有的所有值"""

    # 现有文件不存在时直接返回新内容
    if not exists:
        return new_content

    def deep_merge_polite(base: dict, update: dict) -> dict:
        result = base.copy()
        for key, value in update.items():
            if key not in result:
                # 只添加缺失的 key
                result[key] = value
            elif isinstance(result[key], dict) and isinstance(value, dict):
                # 嵌套 dict 递归合并
                result[key] = deep_merge_polite(result[key], value)
            else:
                # key 已存在且不是 dict → 保留用户值，忽略模板值
                pass
        return result

    merged = deep_merge_polite(existing_content, new_content)
    # 内容无变化时返回 None，调用方跳过写入（保护用户注释/格式）
    if merged == existing_content:
        return None
    return merged

这段代码体现了"礼貌性"的工程哲学：Spec Kit 是访客，用户的 settings.json 是主人。 合并时只贡献缺失的配置，绝不覆盖用户已有设置，且当内容无变化时直接跳过写入（保留用户的注释和格式化）。 原子写入（先写临时文件再 os.replace）防止写入中断导致文件损坏。

3.2.2 集成适配系统 —— 支持 25+ AI Agent 的架构

要解决的问题：不同 AI Agent 的命令文件格式、目标目录、命名规则各不相同： Claude Code 用 .claude/commands/speckit.specify.md， Gemini CLI 用 TOML 格式写入 .gemini/commands/， Codex 用 .agents/skills/speckit-specify/SKILL.md。 如何在不写大量 if-else 的情况下支持全部？

方案：三层继承 + 注册表。IntegrationBase 定义接口， MarkdownIntegration / TomlIntegration / SkillsIntegration 分别实现三种格式变体，每个 Agent 的子类只需设置 3 个类属性（key、config、registrar_config）。 INTEGRATION_REGISTRY 在 integrations/__init__.py 中通过扫描所有子类自动构建。

# 文件: src/specify_cli/integrations/base.py (line 56-165)
class IntegrationBase(ABC):
    """所有 Agent 集成的抽象基类"""

    # 每个子类必须设置这 3 个类属性
    key: str = ""               # Agent 唯一 ID（e.g. "claude"）
    config: dict | None = None  # 目录配置（folder、commands_subdir 等）
    registrar_config: dict | None = None  # 命令注册配置

    def commands_dest(self, project_root: Path) -> Path:
        """从 config 推算命令文件部署路径，子类无需覆写"""
        folder = self.config.get("folder")
        subdir = self.config.get("commands_subdir", "commands")
        return project_root / folder / subdir

    def command_filename(self, template_name: str) -> str:
        """默认文件名格式，TOML/Skills 子类会覆写"""
        return f"speckit.{template_name}.md"

    # 具体部署逻辑由 MarkdownIntegration/TomlIntegration 等实现

关键设计：commands_dest 和 command_filename 提供合理默认值， 子类只在需要偏离默认行为时才覆写（TomlIntegration 覆写 command_filename 返回 .toml 后缀；SkillsIntegration 覆写 commands_dest 返回 skill 子目录路径）。新增 Agent 通常只需创建一个 5~20 行的子类文件。

3.2.3 扩展管理器 —— Extension Hook 系统

要解决的问题：核心 SDD 流程（specify → plan → tasks → implement）对于 90% 的项目已足够， 但剩下 10% 需要：Jira 同步、自动 commit、多 Agent 并行、CI Gate…… 如果把所有这些都加入核心，核心将失控膨胀。

方案：Hook 系统。扩展通过 extension.yml 声明自己的命令， 安装后会在 .specify/extensions.yml 注册钩子（before_specify、after_plan 等）。 核心命令（如 /speckit.specify）在执行前后读取这个文件， 按 optional/mandatory 标志决定是提示用户还是自动执行对应扩展命令。

3.2.4 /speckit.specify —— SDD 工作流的心脏

要解决的问题：如何让 AI Agent 从一段随意的功能描述生成高质量、可测试、与技术无关的规格文档？ 如何防止 AI 在规格里偷偷写下"用 React 实现"这类实现细节？如何在 AI 不确定时提问但又不让提问过多打断用户？

方案：命令文件本身是一套精密的 AI 引导脚本，包含： ① 前置 Hook 检查（执行前寻找 before_specify 钩子）； ② 自动生成特性短名（2-4 词，用于目录命名）； ③ 顺序编号策略（sequential 或 timestamp）； ④ 分层验证清单（Success Criteria 必须技术无关、需求必须可测）； ⑤ 最多 3 个 [NEEDS CLARIFICATION] 标记的上限约束； ⑥ 后置 Hook 检查。

## 验收标准设计原则

**好的例子**（技术无关、用户视角）：
- "用户可以在 3 分钟内完成结账"
- "系统支持 1 万并发用户"

**坏的例子**（泄露实现细节）：
- "API 响应时间低于 200ms"    ← 太技术化，改为"用户立即看到结果"
- "React 组件高效渲染"        ← 框架细节不属于规格
- "Redis 命中率高于 80%"      ← 技术指标不属于规格

## 澄清限制
- 最多 3 个 [NEEDS CLARIFICATION] 标记
- 优先级：功能范围 > 安全/隐私 > 用户体验 > 技术细节
- 常见可以使用合理默认的情况（不要提问）：
  - 数据保留：使用行业标准
  - 认证方式：Web 应用使用 session 或 OAuth2

这段提示词的精妙之处在于它用"反例教学"训练 AI 识别规格文档中的不良模式—— 这比说"不要写技术细节"更有效，因为 AI 能从具体案例中归纳出判断规则。 "最多 3 个澄清"的硬限制防止 AI 陷入无休止的问答，强迫它在信息不完整时做出合理假设。

4 方案评价

4.1 优势分析

架构简洁，扩展成本极低

  • 新增 Agent 集成：一个 5-20 行子类，无需修改主流程
  • 新增扩展：独立仓库 + 一个 extension.yml，无需 PR 核心代码
  • 新增预设：只需覆写命令 MD 文件，无 Python 代码
  • 抽象层次清晰：CLI 层 / 集成层 / 模板资产层 / 运行时层 职责不重叠

工程细节周到

  • 原子写入防止配置文件损坏
  • 礼貌深度合并保护用户已有设置
  • wheel 打包 air-gapped 可用（v0.6+）
  • 语义化版本约束（speckit_version: ">=0.6.0"）
  • Claude Code 非标准安装路径专项处理

社区飞轮效应明显

  • 6 个月内 40+ 社区扩展，覆盖 Jira/Azure/Linear 等主流集成
  • 8 个社区预设，包括"小说写作模式"等创意应用
  • 87.7k Star 带来持续的外部贡献
  • Extension 目录网站、VSCode 可视化插件等生态已自发形成

方法论与工具一体化

  • spec-driven.md 提供完整的 SDD 理论框架
  • 提示词文件是方法论的可执行版本，两者始终保持一致
  • Newsletter 月刊维持社区方向感

4.2 局限分析

提示词质量无法被工具链保障

  • 命令文件是 Markdown 纯文本，无类型检查、无 lint
  • 提示词语义正确性只能靠人工测试，CI 无法自动回归
  • 不同 AI 模型对同一提示词的理解差异可能导致输出不一致
  • 提示词迭代需要大量人工 LLM 测试，成本高

规格漂移问题未内置解决

  • 代码实现后 spec.md 与代码的一致性无内置检测机制
  • Spec Sync 扩展依赖 AI 理解代码变更，可靠性有限
  • 大型团队多分支并行时，规格冲突的合并策略缺失

扩展生态质量参差不齐

  • 官方不审核扩展代码本身，安全责任由用户承担
  • 40+ 扩展中许多缺乏测试和维护承诺
  • 扩展间依赖关系（requires）复杂时的解析逻辑尚弱

运行时行为不可预测

  • Hook 执行由 AI Agent 解释，不是 Python 代码，无法单测
  • 不同 Agent 对 EXECUTE_COMMAND 指令的响应方式不同
  • 命令执行失败时没有统一的回滚机制
  • feature.json 状态文件在并行 Agent 场景下可能产生竞争
