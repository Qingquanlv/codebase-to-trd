oh-my-openagent 深度解读

最强 AI 编程 Harness — 批量并行 Agent + 哈希锚定编辑 + 全模型无锁定
oh-my-openagent（原名 oh-my-opencode，简称 OmO）是一个 OpenCode 插件，以 Sisyphus 多智能体编排框架为核心，将 Claude、GPT-5.4、Kimi K2.5 等多个主流模型集成为"专业分工、并行协作"的 AI 开发团队。区别于普通 AI 编程助手，它在「工具层」做了决定性改造：Hashline 内容哈希编辑工具将代码修改成功率从 6.7% 提升至 68.3%，Todo 续行强制器保证长任务不中途放弃，Context Window 自动恢复机制消除模型上下文耗尽的中断。类比来说，如果 OpenCode 是 Arch Linux，OmO 就是 Ubuntu——一键安装，开箱即用。

核心结论

  • OmO 的核心创新不是 prompt 工程，而是编辑工具层：Hashline 系统用内容哈希替代行号，彻底解决了 AI 编辑"对不上行"的问题。
  • Sisyphus 调度器通过类别路由（quick / deep / ultrabrain / visual-engineering）自动为不同任务匹配最优模型，用户无需手动指定模型。
  • 完整的失败兜底层（Todo 续行强制器 + Context 自动恢复 + Session 错误恢复）使得 AI 对复杂长任务的交付能力接近人类水准。

1 摘要

1.1 项目背景

AI 代码工具已经从"补全单行"演进到"执行跨文件任务"，但"Agent 干到一半停下来"的问题始终困扰开发者。模型自身能力提升飞快，真正的瓶颈却藏在 harness（线束）层：

  • 编辑工具层：模型读取文件后，记住的行号随着文件修改迅速失效，导致 stale-line 错误；
  • 上下文层：长任务会把 context window 填满，Agent 直接崩溃而非优雅继续；
  • 任务追踪层：模型在完成 70% 工作后可能自认为"done"并停止，剩余 todo 被遗忘；
  • 模型绑定层：Claude Code 等工具将用户锁定在单一 Provider，无法随时切换更便宜或更快的模型。

作者 YeonGyu-Kim 亲身花费 $24,000 美元测试各种工具后，将所有痛点的解法集成进这个插件。项目在 GitHub 积累了 51.1k Star，在真实生产环境中被 Google、Microsoft 工程师使用。

1.2 项目预期

1.2.1 预期一：让 AI 真正干完活

通过 Todo 续行强制器（TodoContinuationEnforcer）监听 session.idle 事件，一旦 Agent 暂停而 todo 列表还有未完成项，系统自动注入续行 prompt，将任务推进到 100% 完成。配合 Ralph Loop（/ulw-loop），Agent 进入自我循环直至所有任务关闭。

1.2.2 预期二：模型不绑定，按任务类型自动选最优

Sisyphus 调度器通过类别路由将工作委派给不同专家 Agent：Hephaestus（GPT-5.4，深度自主执行）、Oracle（架构/调试）、Librarian（文档搜索）、Explore（快速 grep）。每个类别映射到最合适的模型，用户只需描述目标，Harness 自动分配。

1.2.3 预期三：编辑工具不再是 Agent 成功率的瓶颈

Hashline 系统在文件读取时为每一行附加内容哈希标签（11#VK|function hello() {），Agent 编辑时引用该标签而非行号。文件若在读取后被修改，哈希不匹配则编辑被直接拒绝，防止腐败应用。这一改变在 Grok Code Fast 基准上将成功率从 6.7% 提升到 68.3%。

2 项目设计理念

2.1 Ultrawork 哲学：一个词驱动全团队

OmO 的核心用户体验目标是极简：安装完成后输入 ultrawork（别名 ulw），全部 Agent 激活，直到任务完成自动停止。这背后是"烘焙而非配置"的设计思路——所有最佳实践、工具选择、模型路由都已固化在插件内，不要求用户懂 LLM 编排。用户只需告诉系统"做什么"，不需要指定"怎么做"。

2.2 插件而非分叉：在 OpenCode 之上增厚

OmO 以 @opencode-ai/plugin SDK 接入 OpenCode，而非 fork OpenCode 的代码库。这个设计决策意味着：OpenCode 升级时 OmO 可以无缝跟进；用户原有的 hooks、commands、skills、MCPs 全部保持兼容；OmO 的功能通过 工具注入（tools）、Hook 劫持（hooks）、Agent 注册（agents）三条路叠加在宿主上，而不是替换宿主。

2.3 竞品横向对比

OmO 在多智能体、可扩展性和工具层上做了差异化，适合需要无人值守完成大型任务的开发者：

工具定位优势劣势
oh-my-openagent OpenCode 超级增强插件 多 Agent 并行、Hashline 编辑、全模型兼容、完整失败恢复 依赖 OpenCode 宿主、配置较多、社区尚在成长
Claude Code Anthropic 官方 CLI Agent 官方支持、与 Claude 深度集成、稳定 锁定 Anthropic、单 Agent、无并行执行
Cursor Agent IDE 内嵌 AI 助手 GUI 体验好、代码上下文丰富 封闭生态、长任务能力弱、无多 Agent
OpenCode 开源 CLI AI 编程框架 开放插件系统、多模型支持 开箱即用性差、需要手动配置

OmO 在无人值守的大型任务执行上具有明显领先优势；对于快速 10 分钟改动，Claude Code 或 Cursor 的即时响应体验更佳。

3 项目使用方式与执行链路

3.1 当前适合解决什么问题

✅ 适合的任务类型

  • 大型特性开发（跨越 10+ 文件）
  • 长任务无人值守执行（ultrawork 模式）
  • 代码库批量重构（如清除 8000 条 eslint 警告）
  • 需要并行探索多条方案的技术决策
  • 需要 LSP 精确重命名/引用的工程级变更
  • 将已有技术栈迁移到新框架

⚠️ 不太适合的任务类型

  • 10 分钟内可完成的小改动（overhead 过大）
  • 纯创意/开放式探索（边界不清晰会让 Agent 无限漫游）
  • 需要 GUI 交互的前端调试（需配合 Tmux/Playwright）
  • 没有 OpenCode 环境的生产服务器

3.2 最小可运行路径

安装完成后只需一个词即可让全套 Agent 激活执行任务。安装本身也可以由另一个 Agent 完成（直接粘贴 URL 给 Claude Code / Cursor）。

# 方式一：让 Agent 帮你安装（推荐）
paste into agent: "Install oh-my-opencode: https://raw.githubusercontent.com/code-yeongyu/oh-my-openagent/refs/heads/dev/docs/guide/installation.md"

# 方式二：手动安装 npm 包
npm install -g oh-my-opencode

# 进入项目目录，启动 OpenCode
cd your-project && opencode

# 在 OpenCode 对话中输入一个词，全套 Agent 启动
ultrawork

# 深度初始化（自动生成分层 AGENTS.md）
/init-deep

3.3 核心使用模式

方式一：ultrawork 无人值守模式
何时用 有明确目标但任务较大，想让 AI 全程跑完
输入 自然语言描述任务目标（如"重构 auth 模块，添加 JWT 支持"）
输出 完整代码变更 + Git commit，期间可多次 Agent 并行执行
方式二：/start-work Prometheus 规划模式
何时用 任务边界不清、有架构决策需要提前厘清
输入 输入 /start-work，然后回答 Prometheus 的澄清问题
输出 验证过的分步计划，可直接交 Sisyphus 执行
方式三：Background Agent 并行探索
何时用 需要同时探索多条技术路线或并行执行独立子任务
输入 Sisyphus 自动分解任务后调用 background_task 工具
输出 各子任务结果异步汇聚，主 Agent 整合后产出最终方案
方式四：/init-deep 项目初始化
何时用 首次在一个代码库中工作，希望 Agent 有完整上下文
输入 在项目根目录运行 /init-deep
输出 每个子目录生成 AGENTS.md，Agent 按目录自动加载相关上下文

3.4 从需求到交付的一次完整链路

以一个具体需求为例：为现有 Node.js 应用新增用户登录失败次数限制功能

① 用户输入目标
在 OpenCode 对话中输入 "ultrawork：为登录接口加上连续失败 5 次锁定 15 分钟的限制"，Sisyphus 收到请求。
② IntentGate 意图分析
Sisyphus 在执行前分析真实意图，识别为"Mid-sized Task"（边界明确的中等规模任务），跳过深度访谈，直接拆解步骤并创建 TaskCreate 待办。
③ 并行委派子 Agent
Sisyphus 通过 background_task 同时启动：Explore Agent（grep 现有 auth 代码结构）、Librarian Agent（查找 Redis / rate-limiter 文档）。两者并行执行，主线程等待。
④ Hephaestus 深度实现
子任务结果汇聚后，Sisyphus 委派 Hephaestus（GPT-5.4，类别=deep）进行实际代码实现。Hephaestus 读取文件时，Hashline 钩子为每行注入内容哈希；编辑时通过哈希验证精确定位，零 stale-line 错误。
⑤ TodoContinuationEnforcer 监控
若 Hephaestus 在完成 80% 后停止响应（session.idle），Todo 续行强制器检测到未完成 todo，自动注入续行提示，推动任务进入下一步：测试覆盖 → comment-checker → Git commit。
⑥ 产出与收尾
任务完成，所有 todo 关闭，生成 atomic commit，Context Window 若临近上限则触发 anthropic-context-window-limit-recovery 进行摘要压缩以备下次会话继续。

4.1.2 目录与核心文件说明

【插件入口与初始化层】 ★ src/index.ts · src/create-tools.ts · src/create-hooks.ts

这层是 OmO 接入 OpenCode 的桥梁。它在 OpenCode 启动时被调用一次，负责按配置装配所有 Agent、注册工具、安装 Hook，然后将控制权交还给 OpenCode。没有这层，其他所有模块都只是不被加载的代码。

src/create-hooks.ts安装所有 Hook，包括 todo-continuation-enforcer、hashline-read-enhancer、context-window-recovery 等

【Agent 编排层】 ★ src/agents/builtin-agents.ts · src/agents/agent-builder.ts · src/agents/sisyphus/

这层定义了"AI 开发团队"里的每个角色，以及如何根据模型能力动态生成 prompt。每个 Agent 目录内有多个模型变体（default / gemini / gpt-5-4），确保 prompt 风格与模型特性匹配，而不是用同一套 prompt 对付所有模型。

这层是 OmO 超越竞品的核心战场。Hashline Edit 解决"AI 对不上行"问题，LSP 提供 IDE 级重命名/诊断，ASTGrep 让代码搜索理解语法而非字符串。这三个工具共同保证了 Agent 的编辑动作"精准落地"。

4.2.1 关键模块一：Hashline 编辑系统

📁 src/tools/hashline-edit/ + src/hooks/hashline-read-enhancer/ 🏷️ 内容哈希验证 + 双阶段 Hook

Hashline 系统分为两个阶段协同工作。读取阶段：HashlineReadEnhancer Hook 拦截 OpenCode 的 Read 工具输出，将标准行格式 11: function hello() { 转换为 11#VK|function hello() {——行号 + 短哈希 + 内容。哈希由 (行号, 内容字符串) 计算得出。编辑阶段：Agent 提交编辑时引用 11#VK 标签，HashlineEditExecutor 重新计算当前文件该位置的哈希，若不匹配（文件被并发修改）则拒绝本次编辑。

这一机制将 Grok Code Fast 基准上的成功率从 6.7% 提升至 68.3%，因为根本原因不是模型能力，而是编辑工具对"行内容稳定性"的假设在多次修改后失效。

// 文件: src/hooks/hashline-read-enhancer/hook.ts
// 将标准行格式 "11: content" 转换为 "11#HASH|content"
function transformLine(line: string): string {
  const parsed = parseReadLine(line)
  if (!parsed) return line
  // 若行内容被截断，跳过哈希注入（截断行不稳定）
  if (parsed.content.endsWith(OPENCODE_LINE_TRUNCATION_SUFFIX)) return line
  const hash = computeLineHash(parsed.lineNumber, parsed.content)
  return `${parsed.lineNumber}#${hash}|${parsed.content}`
}

4.2.2 关键模块二：多智能体编排与动态 Prompt

📁 src/agents/builtin-agents/ + src/agents/agent-builder.ts 🏷️ 类别路由 + 模型专属 prompt 变体

export function buildAgent(
  source: AgentSource,
  model: string,
  categories?: CategoriesConfig,
): AgentConfig {
  const base = isFactory(source) ? source(model) : { ...source }
  const categoryConfigs = mergeCategories(categories)

  // 按 category 覆盖 model 和 temperature
  if (agentWithCategory.category) {
    const cfg = categoryConfigs[agentWithCategory.category]
    if (cfg) {
      if (!base.model) base.model = cfg.model
      if (base.temperature === undefined) base.temperature = cfg.temperature
    }
  }

  // 将激活的 Skill 内容前置到 prompt
  if (agentWithCategory.skills?.length) {
    const { resolved } = resolveMultipleSkills(agentWithCategory.skills, { ... })
    const skillContent = Array.from(resolved.values()).join("\n\n")
    base.prompt = skillContent + (base.prompt ? "\n\n" + base.prompt : "")
  }
  return base
}

4.2.3 关键模块三：Todo 续行强制器

📁 src/hooks/todo-continuation-enforcer/ 🏷️ 事件驱动状态机 + 倒计时注入

TodoContinuationEnforcer 是 OmO 实现"不停工"承诺的技术核心。它以状态机运行：监听 session.idle（Agent 停止响应）、session.error（包括 Abort 和 Token 上限）、tool.execute.after（工具调用完成）等事件，维护每个 Session 的 stagnation count、abort 标记、lastInjectedAt 时间戳。当检测到"Agent 停了但 todo 未清"时，触发续行注入，将未完成 todo 列表格式化后作为新的用户消息推送，重新激活 Agent。

// session.idle 事件处理核心逻辑
if (event.type === "session.idle") {
  // 如果用户主动取消，不再续行
  if (state.wasCancelled) return
  // 如果是 Token 上限触发的停止，等待 Context 恢复完成
  if (state.tokenLimitDetected) return
  // 检查是否有未完成 todo
  const incompleteTodos = await fetchIncompleteTodos(sessionID)
  if (incompleteTodos.length > 0) {
    // 注入续行提示，包含未完成 todo 列表
    await injectContinuationMessage(sessionID, incompleteTodos)
    state.lastInjectedAt = Date.now()
    state.stagnationCount = 0
  }
}

4.2.4 关键模块四：Context Window 自动恢复

📁 src/hooks/anthropic-context-window-limit-recovery/ 🏷️ 多策略分级压缩

当 Anthropic API 返回 context window 超限错误时，OmO 不会让 Agent 崩溃，而是按优先级尝试多种压缩策略：① 工具输出截断（pruning-tool-output-truncation：对大型工具输出按比例截断）、② 历史去重（pruning-deduplication：移除重复的工具调用对）、③ 摘要压缩（summarize-retry-strategy：将历史消息总结成简短摘要）、④ 激进截断（aggressive-truncation-strategy：极端情况下强制丢弃最旧消息）。每个策略尝试后重新估算 token 数，若仍超限则升级到下一策略。 

// 文件: src/hooks/anthropic-context-window-limit-recovery/recovery-strategy.ts
// 按策略优先级依次尝试压缩
const STRATEGIES = [
  pruneToolOutputTruncation,   // 策略1：截断大型工具输出
  pruneDeduplication,          // 策略2：去除重复历史
  summarizeRetry,              // 策略3：摘要压缩
  aggressiveTruncation,        // 策略4：激进截断（最后手段）
]
for (const strategy of STRATEGIES) {
  const result = await strategy(messages, tokenBudget)
  if (result.estimatedTokens <= tokenBudget) return result
}

5 方案评价

5.1 优势分析

  • 工具层创新：Hashline 编辑系统从根本上解决了 AI 编辑的 stale-line 问题，成功率提升 10 倍，远超 prompt 层优化的收益
  • 真正的无人值守：Todo 续行强制器 + Context 恢复 + Session 错误恢复三层保护，长任务能自愈而非中途放弃
  • 模型无锁定：Category 路由架构让整个系统与具体模型解耦，可随时替换为更便宜/更快的模型，已验证 Kimi K2.5 + GPT-5.4 组合可超越纯 Claude 效果
  • IDE 级精度：LSP rename/diagnostics 和 ASTGrep 让 Agent 的代码操作具备 IDE 精度，减少因文本匹配失败导致的错误
  • Skill 资源隔离：Skill Embedded MCP 让每个技能按需启停自己的 MCP 服务，context window 不被常驻 MCP 污染
  • 社区验证：51.1k Stars，真实案例包括"一夜将 45k 行 Tauri 应用迁移为 SaaS"，工程可信度高

5.2 局限分析

  • 依赖 OpenCode 生态：OmO 是插件而非独立工具，用户必须先安装和运行 OpenCode，增加了入门门槛
  • 配置复杂度：完整功能（类别路由、模型 fallback、disabled_hooks）需要理解 JSONC 配置，对非技术用户不友好
  • 模型成本不可预测：并行 Background Agent 在高频使用时可能快速消耗 API 配额，特别是使用按量付费模型
  • Hashline 不向后兼容：enabled 时改变了 Read 工具的输出格式，与不支持 Hashline 的第三方 Agent 可能存在兼容问题
  • 快速迭代的稳定性风险：4980 次 commits，162 个 releases，快速更新意味着 API 接口和行为可能频繁变动
  • 主要针对 Anthropic 优化：Context Window 恢复模块目前以 Anthropic 为主，其他 Provider 的 Token 上限处理机制较弱
