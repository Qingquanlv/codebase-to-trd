GitHub Spec Kit 深度解读

面向 AI 时代的规格驱动开发（SDD）脚手架与 Specify CLI
Spec Kit 是 GitHub 开源的规格驱动开发（Spec-Driven Development, SDD）工具包：通过 specify CLI 在项目中落地模板、脚本与多 Agent 集成，再配合 /speckit.* 类命令，把「规格 → 技术方案 → 任务列表 → 实现」串成可重复的工作流。与纯 Markdown 行为规范不同，Spec Kit 带有可安装的 Python CLI、按 Agent 分发的集成层，以及哈希跟踪的安装清单（Manifest），用于安全卸载与版本对齐。读完本文，你将理解 SDD 在本项目中的具体含义、specify init 落盘了哪些目录、集成系统如何扩展，以及扩展与 Preset 生态如何运作。

核心结论

  • 规格优先于代码 — SDD 将 PRD/规格与实现计划视为可演进的主资产，代码是其生成物；Spec Kit 用模板与命令把这一理念产品化
  • Specify CLI 是引导器 — uv tool install specify-cli --from git+… 安装后，specify init 写入 .specify/、命令模板，并按所选 AI 集成复制 slash command 或 skills
  • 集成即插件 — INTEGRATION_REGISTRY 注册 Claude、Copilot、Gemini 等二十余种集成；统一基类处理 Markdown / TOML / Skills 三种落盘格式
  • 可审计的安装 — IntegrationManifest 记录每个生成文件的 SHA-256，卸载时仅删除哈希仍匹配的文件，避免误删用户改动
  • 捆绑 core_pack 支持离线 — Wheel 内嵌 templates/、scripts/、部分扩展与 Preset，企业内网可不经 PyPI 完整初始化

1 摘要

1.1 项目背景

大模型写代码能力提升后，「先写代码再补文档」的模式暴露出规格与实现长期漂移的问题。Spec-Driven Development 主张规格与实现计划足够精确、可执行，使 AI 能在结构化上下文中生成实现与测试，并把团队意图保留在自然语言与设计资产中。

GitHub Spec Kit 将这一方法论封装为：Specify CLI（Python 3.11+，Typer + Rich）负责项目脚手架；模板与命令（templates/commands/*.md）在各 Agent 中以 /speckit.specify、/speckit.plan 等形式暴露；社区扩展与 Preset 通过目录清单与 CLI 子命令安装，形成可组合生态。

1.2 能解决的问题

1.2.1 「规格与实现悄悄漂移」——可执行的规格主资产

传统开发中，PRD 写完就束之高阁，代码演进后规格文档与实现长期脱节，AI 编程工具更难从陈旧文档中提取准确上下文。Spec Kit 把 spec.md、plan.md、tasks.md 定义为版本化的主资产，与代码同支同仓，/speckit.specify 等命令强制 Agent 先读规格再动手，规格不更新则 PR 无从对齐。

1.2.2 「多 AI 工具各自为政，接入摩擦高」——统一多集成脚手架

Claude Code、Cursor Agent、Copilot、Gemini CLI 各有独立的命令格式（Markdown / TOML / Skills）与目录约定，团队切换工具时需要手动重写所有指令。specify init --ai <tool> 通过 INTEGRATION_REGISTRY 一次性生成对应格式的 slash command 文件，新成员换工具只需重跑 init，无需手动维护各套配置。

1.2.3 「脚手架生成的文件难以安全管理」——哈希跟踪的可审计 Manifest

脚手架写入项目的文件往往与用户自定义内容混在一起，升级或卸载时容易误删改动。IntegrationManifest 为每个生成文件记录 SHA-256，specify uninstall 只删除哈希仍匹配的文件，已被用户修改的文件自动跳过，让「脚手架内容」与「团队定制」清晰分离。

1.2.4 「SDD 工作流无法在 AI 会话中稳定落地」——命令驱动的阶段化流水线

即便团队认可 SDD 理念，若没有统一入口，AI 每次对话都可能绕过规格直接生成代码。Spec Kit 提供 /speckit.constitution → specify → plan → tasks → implement 的有序命令链，配合 core_pack 离线捆绑，在内网或离线环境下同样可以完整初始化，把 SDD 流程变成可重复的操作手册。

2 项目设计理念

2.1 核心理念

  1. 规格即主资产：功能规格（spec.md）、实现计划（plan.md）、任务分解（tasks.md）在分支上版本化，Agent 按阶段读写这些文件。
  2. 命令驱动工作流：/speckit.constitution 建立项目宪法；specify → plan → tasks → implement 形成递进流水线。
  3. 多 Harness 一等公民：通过 IntegrationBase 子类为 Claude Code、Cursor Agent、Copilot、Gemini CLI 等分别定义目录结构与命令格式（Markdown / TOML / Skills）。

2.2 与同类方案横向对比

工具定位优势劣势
GitHub Spec KitSDD 脚手架 + 多 Agent 集成 + 扩展市场官方维护、CLI 安装简单、集成覆盖广、Manifest 安全卸载、捆绑资源支持离线依赖团队遵守 slash 命令流程；Agent 行为仍受模型能力约束
纯 Cursor Rules / AGENTS.md单编辑器行为约束零额外工具、上手快无统一规格目录结构；无哈希跟踪的多 Agent 命令生成
Backstage / 内部开发者门户服务目录与模板企业级治理、插件丰富部署与运维成本高；与 AI 会话内 SDD 流程非同一抽象层
Superpowers 等纯技能库Markdown 行为规范零运行时、铁律叙事强无 CLI 脚手架；不自带规格模板与分支目录约定

赢面：把 SDD 的目录约定、命令模板与 Agent 集成一次性写进项目，并可复现升级。输面：不强制执行人类评审；规格质量仍依赖团队纪律与模型遵循度。

2.3 Spec Kit 是什么（一句话）

一套以 Specify CLI 为中心、用模板与集成清单把「宪法 → 规格 → 计划 → 任务 → 实现」接到多种 AI 编程 Agent 上的开源 SDD 工具包。

2.4 当前适合解决什么问题

适合

  • 希望用统一目录与命令管理功能规格与实现计划的团队
  • 同时使用多种 AI Agent，需要一致的工作流入口（slash 命令命名）
  • 需要可重复初始化与可升级脚手架（uv tool install 固定版本）
  • 企业内网或代理环境，需要捆绑模板减少对外网络依赖

不太适合

  • 仅需代码风格检查、不需要规格产物的项目
  • 无法采用分支化规格目录（如 specs/NNN-feature/）的遗留流程
  • 期望运行时强制规格合规（本工具以文件与命令为主，非编译器插件）

3.4 常见误解

误解实际情况
「Spec Kit 会在 CI 里自动拦截不合规格代码」核心仓库提供的是脚手架与命令；是否接入 CI 取决于团队是否编写检查步骤或使用社区 CI Guard 等扩展。
「specify init 每次都会覆盖我的 constitution」ensure_constitution_from_template 在已有 constitution.md 时会跳过复制，保留现有文件。
「卸载集成会删掉我改过的命令文件」IntegrationManifest.uninstall 默认仅在文件哈希与记录一致时删除；被用户修改的文件会列入 skipped。
「SDD 等于不写代码」仓库明确描述代码仍是实现载体；SDD 改变的是驱动关系与资产优先级，而非取消实现。

4 项目使用方式

4.1 从需求到交付的一次完整过程

以「用 Spec Kit 为内部工具新增导出 CSV 功能」为例，走一遍标准 SDD 链路（含 CLI 与 Agent 命令）。

① 安装 CLI
执行 uv tool install specify-cli --from git+https://github.com/github/spec-kit.git@vX.Y.Z，将 Specify CLI 装入全局环境。
② 初始化项目
在仓库根运行 specify init --here --ai claude，CLI 写入 .specify/、安装 core_pack 脚本与模板，并为 Claude 注册 skills 格式命令。
③ 建立宪法　/speckit.constitution
生成或更新 .specify/memory/constitution.md，约束测试策略、UX 原则与团队规范，所有后续命令都以此为行为准则。
④ 编写规格　/speckit.specify
根据需求描述创建特性分支与 specs/…/spec.md，填充用户故事与验收场景（「导出 CSV，支持日期范围过滤」）。
⑤ 澄清需求（可选）　/speckit.clarify
对规格中模糊或欠定义的区域进行问答式澄清，建议在 /speckit.plan 之前运行，避免技术方案建立在错误假设上。
⑥ 技术计划　/speckit.plan
读取 spec.md 与 constitution.md，生成 plan.md，包含技术选型、模块拆分与依赖说明。
⑦ 生成任务　/speckit.tasks
将 plan.md 拆解为可勾选的 tasks.md；可选接 /speckit.checklist 生成「需求完整性核查清单」（类似用英语写单元测试），在任务执行前做最后质量把关。
⑧ 一致性分析（可选）　/speckit.analyze
跨工件分析 spec / plan / tasks 的覆盖度与一致性，找出遗漏场景；建议在 /speckit.tasks 之后、/speckit.implement 之前运行。
⑨ 转为 GitHub Issues（可选）　/speckit.taskstoissues
将 tasks.md 中的任务条目批量创建为 GitHub Issues，便于团队认领、追踪进度与关联 PR。
⑩ 实现与收尾　/speckit.implement
Agent 按 tasks.md 逐条实现代码改动，任务完成后勾选对应条目，最后走正常 PR 评审与合并。

5.3 关键模块实现细节

5.3.1 IntegrationManifest（可审计卸载）

📄 src/specify_cli/integrations/manifest.py 🏷 存储层 · 集成安装 bookkeeping

要解决的问题：集成会向用户仓库写入多个文件；若卸载时盲目删除，可能删掉用户手工编辑过的命令或并行业务文件。

解决方案：安装时记录每个相对路径的 SHA-256；卸载时比对当前文件哈希，仅删除仍与记录一致的文件，否则跳过并返回列表。

6 方案评价

6.1 优势分析

官方背书与活跃生态

GitHub 维护，社区扩展目录与测试矩阵覆盖大量 Agent。

可复现安装

uv tool install …@tag 固定版本；模板与 CLI 版本通过 core_pack 对齐。

安全卸载语义

Manifest 哈希比对减少误删用户改动。

扩展与 Preset

子命令管理扩展目录与优先级，便于团队定制。

6.2 局限分析

行为仍依赖 Agent

slash 命令是否被模型严格遵守无法由 CLI 强制。

学习曲线

SDD 流程与目录约定需要团队培训，非「安装即自动变好」。

单体 CLI 文件

主要逻辑集中在超大 __init__.py，贡献者上手成本高。
