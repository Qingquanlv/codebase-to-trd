claude-task-master（Taskmaster）深度解读

npm task-master-ai · MCP + CLI · tasks.json · Monorepo
claude-task-master 是 Eyal Toledano 等维护的开源仓库，对外品牌为 Taskmaster（npm 包 task-master-ai）。它提供面向 AI 编程会话的任务系统：以 tasks.json 为中枢，支持依赖、子任务、优先级、复杂度分析与扩展；通过 MCP（FastMCP 实现的 dist/mcp-server.js）嵌入 Cursor / Claude Code，亦提供 task-master CLI（打包自 scripts/dev.js）。核心逻辑分布在 mcp-server/（MCP 与 task-master-core 直连封装）、packages/tm-core/（TasksDomain 等领域服务）与 apps/cli、apps/mcp 等 workspace 包。官方文档站：docs.task-master.dev。

核心结论

  • 双通道同一套任务模型 — 编辑器经 MCP 调工具；终端经 task-master / dev.js 调同一套 direct 函数与 @tm/core
  • MCP 工具面可裁剪 — 环境变量 TASK_MASTER_TOOLS 控制注册子集（core / standard / all / 列表），降低工具爆炸与策略风险
  • 混合实现栈 — tool-registry.js 同时挂载传统 JS 工具与来自 @tm/mcp 的 TS 工具（含 Autopilot）
  • 领域层在 tm-core — TasksDomain 聚合服务，负责与存储、标签、生成 Markdown 任务文件等交互
  • 强依赖模型与密钥 — PRD 解析、研究、扩展等路径需配置多厂商 API Key 或 Claude Code / Codex CLI OAuth（见 README）

1 摘要

1.1 项目背景

AI Agent 在长会话中容易丢失「下一步做什么」与「哪些依赖未完成」。Taskmaster 把工程拆成可机器读的任务图，并给 IDE 暴露标准 MCP 工具，使模型能列出任务、取下一个、更新状态、展开子任务，而不是只在聊天里口头列清单。

技术栈：Node ≥20、TypeScript（多包）、tsdown 构建、Jest 测试、Vercel AI SDK 家族依赖调用各模型；许可证为 MIT + Commons Clause（限制转售/SaaS 化，详见 LICENSE）。

1.2 项目预期

1.2.1 理解数据平面

能说明 tasks.json 路径、字段与 generate 产出的单文件任务关系。

1.2.2 能配置 MCP 与模型

能在 mcp.json / .env 中配置 npx task-master-ai 与至少一个提供商密钥。

1.2.3 能跟踪请求路径

从 MCP 工具调用到 task-master-core 再到 TasksDomain 的分层有基本概念。

1.2.4 能评估 Commons Clause

能在选型会议上说明与纯 MIT _fork 合规差异。

2 项目设计理念

2.1 核心理念

  1. 任务优先于闲聊：把「下一步」结构化为可查询、可更新的数据，而不是依赖模型上下文记忆。
  2. 与编辑器共生：MCP 作为首选集成，降低切换成本。
  3. 可渐进自动化：Autopilot、复杂度报告、批量 expand 等能力按团队成熟度选用。

2.2 与同类方案横向对比

工具定位优势劣势
TaskmasterMCP + CLI + tasks.json 任务图 + AI 解析/扩展Cursor 一键装 MCP；多模型；依赖与子任务一等公民；Autopilot需 API 成本；Commons Clause；仓库体量大
纯 Todo MCP轻量 checklist极简无 PRD→任务图、无复杂度/标签体系
Issue TrackerJira / Linear团队协作与权限成熟会话内模型原生操作成本高；与代码侧上下文割裂
OpenSpec / 规格工具规格与变更目录审计与并行变更强与 Taskmaster 侧重点不同，可并存

2.3 定位一句话

把「AI 驱动的开发待办」做成带依赖图的项目内数据库，并用 MCP/CLI 统一读写接口。

2.4 当前适合解决什么问题

适合

  • Cursor / Claude Code 深度用户，希望会话内可调用任务 CRUD 与 next
  • 有 PRD / 需求文档，希望自动拆任务并保留 parse-prd 后续演进
  • 需要依赖顺序与子任务指导多步实现
  • 希望复杂度分析 + expand控制粒度

不太适合

  • 无法使用外部 LLM API 且不使用 Claude Code/Codex 托管路径
  • 团队已有强制 Issue 系统且禁止平行 .taskmaster 状态源
  • 无法接受 Commons Clause 对商业再分发的限制

3.4 常见误解

误解实际情况
「只支持 Claude」README 列举多提供商与 OpenRouter；仓库名历史遗留，能力已泛化。
「安装 MCP 后无需任何 API Key」多数 AI 功能仍需密钥；仅部分托管 CLI 路径例外。
「tasks.json 与 Jira 自动双向同步」Taskmaster 以本地/仓库文件为主，外部系统需自建集成。
「apps/mcp 即运行时全部」发布包入口为打包后的 mcp-server，其中 tool-registry 再 import @tm/mcp 部分工具。

4 项目使用方式

4.1 从 PRD 到可执行子任务的一条链路

1
初始化 — MCP initialize_project 或 CLI task-master init，生成 .taskmaster 骨架。
2
准备 PRD — 将需求写入约定路径的 Markdown/HTML（参见官方教程）。
3
parse_prd — 调用模型解析为 tasks.json 中的任务列表。
4
（可选）analyze + expand — 评估复杂度并拆子任务。
5
next_task — 取依赖已满足的下一项。
6
实现与 set_task_status — 开发完成后更新状态；必要时 Autopilot 辅助提交。

5.3 关键模块实现细节

index.js + scripts/dev.js · CLI 启动层

要解决的问题

为 npm 包提供可编程导出的 initProject，同时保留轻量 task-master init/list/next 命令；重逻辑放在打包后的 dev 管线。

解决方案

index.js 使用 Commander 注册 init，其余命令通过 spawn('node', [devScriptPath, …]) 委托；scripts/dev.js 先加载 .env、Sentry、Auth 检查，再动态 import ./modules/commands.js 的 runCLI。

mcp-server/src/index.js · MCP 服务层

要解决的问题

在单进程内托管数十个 MCP 工具，并按部署环境裁剪；兼顾可观测性与失败隔离。

解决方案

TaskMasterMCPServer 构造 FastMCP，在 init 中调用 getToolsConfiguration() 读取环境变量，再 registerTaskMasterTools；对注册失败工具记录 failedTools 而不使整个进程崩溃。

6 方案评价

6.1 优势分析

IDE 一体化

MCP 深度集成 Cursor 等，工具调用路径短。

任务模型丰富

依赖、子任务、标签、复杂度与 expand 流水线成熟。

工程化程度高

Monorepo、分层测试、tsdown 与多应用边界清晰。

工具可裁剪

TASK_MASTER_TOOLS 便于企业最小权限暴露。

6.2 局限分析

API 成本

PRD 解析与研究路径持续消耗 token 与费用。

许可证 Commons Clause

商业分发形态需法务审阅。

仓库复杂

新人需区分 mcp-server、apps、mcp 与 tm-core 三条线。
