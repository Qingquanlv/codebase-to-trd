OpenSpec 深度解读

AI 原生规格层：openspec/specs 与 changes，OPSX 工件图 + npm CLI
OpenSpec（@fission-ai/openspec）是 Fission-AI 维护的开源工具：在目标仓库中建立 openspec/specs/ 作为行为真相，用 openspec/changes/<change>/ 管理提议增量（proposal、design、tasks、delta specs），通过 Node.js CLI（Commander）与 25+ AI 工具的 slash / skills 集成协同工作。OPSX 工作流用 schema.yaml 描述ArtifactGraph（工件依赖 DAG），把硬编码流程拆成可编辑模板。读完本文，你将理解其与 GitHub Spec Kit 的定位差异、目录模型、以及 src/core/artifact-graph 与 Zod 校验在系统中的位置。

核心结论

  • 双区目录模型 — specs/ 描述「系统现在如何工作」；changes/ 并行容纳多个未合并变更，归档时再合并回 specs
  • OPSX = 可定制工件图 — YAML 定义 artifacts 与 requires 边，graph.ts 用 Kahn 算法算构建顺序与就绪工件，模板 Markdown 替代「全写死在 TS 里」
  • 变更即数据 — ChangeSchema（Zod）约束 why、whatChanges、deltas[]，CLI validate 可交互修复
  • 轻量brownfield优先 — 哲学强调 fluid / iterative / brownfield；与「重阶段门」的规格框架形成对比
  • 遥测可选 — Commander preAction 调用 PostHog 风格遥测（首启提示），可用环境变量关闭

1 摘要

1.1 项目背景

AI 编程助手的能力强，但需求若只存在于聊天记录，结果难以预测、也难以审计。OpenSpec 在仓库内引入一层轻量规格文件，让人与模型在写代码前对齐「要改什么、验收是什么」，并把当前真相与进行中变更分目录管理，降低并行功能分支的认知冲突。

技术栈：TypeScript 源码、Commander CLI、Zod 校验、Vitest 测试；发布为 npm 包，要求 Node.js ≥ 20.19.0。

1.2 项目预期

1.2.1 理解目录与真相来源

能解释 openspec/specs/ 与 openspec/changes/ 的职责边界，以及 archive 后 specs 如何更新。

1.2.2 掌握 OPSX 与 legacy 工作流

能说明 schema.yaml、工件模板与 slash 命令 /opsx:* 的关系，以及 openspec config profile 扩展命令集的作用。

1.2.3 会用 CLI 与校验闭环

能使用 openspec init、update、validate、change 等子命令维护项目内 OpenSpec 状态。

1.2.4 能对比选型

能向团队说明 OpenSpec 与 Spec Kit、Kiro、或无规格裸用 AI 的取舍。

2 项目设计理念

2.1 核心理念

  1. Fluid not rigid：不强制线性阶段门，可按需创建或修订 proposal / specs / design / tasks。
  2. Iterative not waterfall：实现中发现设计问题可回到规格与 delta，而非「阶段已过」。
  3. Brownfield-first：以增量（Delta）描述对现有 spec 的修改，适合遗留系统演进。

2.2 与同类方案横向对比

工具定位优势劣势
OpenSpecnpm CLI + 仓库内 specs/changes + OPSX 可编辑工作流轻量、多 Agent、工件图可定制、Zod 强校验、棕地叙事依赖团队持续维护 specs；遥测与外部服务需合规评估
GitHub Spec KitPython Specify CLI + 多 Harness 集成官方背书、Manifest 卸载安全、捆绑离线资源相对更重；阶段与模板由发行版驱动，自定义成本不同
KiroAWS IDE 内深度集成产品化体验工具链锁定；OpenSpec 强调「已有工具」
无规格裸用 AI对话即需求零 setup不可预测、难审计、难并行协作

README 中自我定位：相对 Spec Kit「更轻、更少仪式」；相对 Kiro「不绑定单一 IDE」。

2.3 OpenSpec 是什么（一句话）

在仓库里用 openspec/ 目录承载规格真相与变更增量，并用 CLI + AI slash 命令驱动 propose → apply → archive 循环的 AI-native 规格系统。

2.4 当前适合解决什么问题

适合

  • 多人在多分支上并行演进功能，需要可审查的规格增量
  • 棕地项目，需要以 delta 描述对现有域 spec 的修改
  • 已使用 Claude / Cursor / Copilot 等，希望统一 slash 工作流
  • 希望自定义 OPSX 模板与依赖图而不等上游发版

不太适合

  • 团队拒绝在仓库中维护任何 Markdown 规格
  • 强依赖「单一集中式需求系统」且无法镜像到 Git 文件
  • 无法使用 Node 20+ 或禁止 npm 全局安装

3.3 状态存储载体

存储层载体职责真相来源
主规格openspec/specs/<domain>/spec.md描述系统当前行为与需求场景合并后真相
变更沙箱openspec/changes/<id>/（proposal、design、tasks、spec delta）未归档前的并行工作副本提议真相（归档前）
项目配置openspec/config.yaml（可选）默认 profile、项目上下文注入团队约定
全局配置用户目录下全局 openspec 配置（若启用）默认工具链、遥测首启标记等用户环境
Agent 挂载.claude/skills/ 等（因工具而异）让模型发现 slash / skill 命令由 init/update 再生成，非业务真相

3.4 常见误解

误解实际情况
「OpenSpec 会替我写代码」它管理规格与任务工件；实现仍由开发者 / Agent 完成，CLI 提供校验与脚手架。
「没有阶段门等于没有流程」OPSX 用依赖图约束工件创建顺序，只是不强制「永远先写完计划再写一行代码」的瀑布节奏。
「specs 与代码自动双向同步」合并与对齐需要 human / Agent 执行 archive 与 PR；非编译器级自动同步。
「遥测无法关闭」项目提供首启提示与配置；企业环境应阅读 telemetry 文档并按政策禁用或屏蔽。

4 项目使用方式

4.1 从需求到交付的一次完整过程

以「为仪表盘增加暗色模式」为例（README 演示流）。

① 全局安装 CLI
npm install -g @fission-ai/openspec@latest，保证 Node ≥ 20.19。
② 项目内初始化
在项目根执行 openspec init，生成 openspec/ 骨架并安装 Agent skills。
③ OPSX 提议
在 Agent 中执行 /opsx:propose add-dark-mode，创建 openspec/changes/add-dark-mode/ 下 proposal、specs、design、tasks。
④ 应用实现
/opsx:apply 按 tasks 勾选推进实现与测试。
⑤ 归档合并
/opsx:archive 将变更移入 changes/archive/… 并更新主 specs/。
⑥ 升级维护
上游发版后 npm install -g …@latest 并运行 openspec update 刷新 Agent 指令。

4.3 核心使用模式

OPSX 默认

propose → apply → archive；适合大多数功能迭代。

扩展 profile

openspec config profile + openspec update 打开 new / continue / verify / sync 等命令。

CLI 校验模式

在 CI 或本地用 openspec validate 做结构 gate，与 Agent 解耦。

5.3 关键模块实现细节

5.3.1 ArtifactGraph（拓扑与就绪集）

📄 src/core/artifact-graph/graph.ts 🏷 OPSX · DAG

要解决的问题：多个工件（proposal、specs、design…）存在依赖，需要稳定、可重复的构建顺序与「当前可写」集合，供 AI 与人类决策。

解决方案：ArtifactGraph.fromYaml 解析 schema；getBuildOrder 使用 Kahn 算法对 requires 边拓扑排序；getNextArtifacts 结合已完成集合返回就绪节点。

1工作流加载器graph.ts磁盘 schemaread YAML路径由 profile 决定
2ArtifactGraphgraph.ts（内部）parseSchema校验 artifacts 结构

5.3.2 ChangeSchema 与 Zod 校验

📄 src/core/schemas/change.schema.ts 🏷 数据模型

要解决的问题：变更 JSON 若自由文本，AI 易生成缺字段或非法 operation，导致 archive/merge 失败。

解决方案：用 Zod 精确定义 Change / Delta 形状，并在 validator 层输出可操作的错误列表，支撑 CLI 与交互式修复。

📄 src/cli/index.ts 🏷 入口 · 遥测

6 方案评价

6.1 优势分析

轻量与双区模型

specs/changes 分离清晰，支持并行变更与审计。

OPSX 可演进

模板 + DAG 让社区可试验工作流而不必 fork 整个 CLI 内核。

强类型校验

Zod + Vitest 保证核心数据结构稳定。

多工具生态

skills 生成覆盖主流 Agent，降低采纳摩擦。

6.2 局限分析

纪律依赖

规格质量与 archive 节奏仍依赖团队与模型配合。

Node 版本门槛

要求较新 Node，部分企业镜像需升级基线。

遥测合规

使用 PostHog 类服务需在组织内评估隐私与退出机制。
