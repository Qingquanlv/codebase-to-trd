Ralph 深度解读

不到 100 行 Bash 实现的自主编程 Agent 循环 — 让 Amp 或 Claude Code 按 PRD 反复执行直到完成所有需求
snarktank/ralph 它是一个基于 Geoffrey Huntley 的 Ralph 模式构建的极简自主 Agent 框架：先借助 LLM 将需求拆解为若干用户故事，并写入 prd.json；再通过循环脚本反复调用 Amp 或 Claude Code，逐个推进这些故事，直到全部完成。整套实现非常巧妙且轻量，核心目标是用“外部化状态 + 新鲜上下文 + 持续校验”的方式，缓解长上下文腐烂、经验难以沉淀，以及缺少反馈约束时 agent 容易跑偏的问题。

核心结论

  • Shell 即编排层 — ralph.sh 不到 100 行 Bash，是整个系统的唯一调度者；没有任何框架依赖，只要有 jq 和目标 AI 工具即可运行。
  • 文件即内存 — prd.json（任务清单）+ progress.txt（学习日志）+ git history（代码快照）三件套完全替代对话历史，彻底消除长上下文导致的质量衰减。
  • 每次实例全新 — 每个 Story 在独立的 AI 进程中完成，上下文污染从架构层面被切断，而非依赖 prompt 技巧去"压缩"历史。
  • 信号即停止条件 — Agent 主动检查所有 Story 是否完成，通过输出 <promise>COMPLETE</promise> 优雅退出，无需外部监控或人工介入。
  • 双工具无缝切换 — 同一套循环通过 --tool amp|claude 参数同时支持 Amp 和 Claude Code，AI 工具的替换成本为零。

1 摘要

1.1 项目背景

自主编程 Agent 在实践中面临两个根本性障碍：第一是上下文污染——当一次对话推进到第 20 轮时，LLM 的注意力被稀释，代码质量和指令遵循度都会下滑；第二是工具锁定——大多数 Agent 框架深度绑定特定模型或 SDK，切换成本高昂。

Ralph 的解法是回到最简单的工程原语：Bash 循环 + 文件 + git。每次循环都起一个全新的 AI 进程，AI 通过读文件获取上下文、写文件持久化状态，从而让任意支持 CLI 调用的 AI 工具都能被纳入同一个自主循环。

技术栈：Bash（主循环）、jq（JSON 操作）、Amp CLI 或 Claude Code CLI（AI 执行层）、Git（代码状态持久化）。无需 Node.js、Python 或任何框架。目标用户：需要无人值守推进较大功能需求的独立开发者或小团队，有清晰的 PRD 文档和可运行的质量检查命令。

1.2 能解决的问题

1.2.1 「AI 写到一半上下文撑不住」——全新会话隔离

对话式 AI 工具在执行多步骤任务时，历史轮次越长质量越差。Ralph 将任务拆解为足够小的 User Story，每个 Story 在独立进程中完成，从不累积历史对话。跨 Story 的上下文通过 progress.txt 的结构化文件传递，而不是对话 token。

1.2.2 「任务做到一半崩了要从头来」——磁盘持久状态

prd.json 中每个 Story 有独立的 passes 字段记录完成状态，git 历史记录每次提交的代码变更。任何时候重启 Ralph，它都能从上次完成的位置继续，而不是重头执行整个任务。

1.2.3 「不知道 AI 做了什么、在哪里」——可观测的进度追踪

progress.txt 是追加写入的人类可读日志，每次迭代都记录实施内容、修改文件和 Learnings。通过 cat prd.json | jq '.userStories[] | {id, title, passes}' 随时查看哪些 Story 完成，git log 看完整的提交历史。

1.2.4 「换一个 AI 工具得重写一套 Agent 逻辑」——工具无关的循环

ralph.sh 将 AI 工具视为黑盒：Amp 用 cat prompt.md | amp，Claude Code 用 claude --print < CLAUDE.md，输出只需包含完成信号即可被识别。未来换任何支持 CLI 的工具，只需修改一行 spawn 命令。

2 项目设计理念

2.1 全新实例隔离上下文（Fresh Context Per Iteration）

Ralph 最核心的设计决策是：永远不复用 AI 会话。每次迭代，ralph.sh 都重新 spawn 一个全新的 AI 进程，前一次迭代的对话历史对当前实例完全不可见。这不是工程偷懒，而是刻意的架构选择：LLM 的"注意力"在长对话中会发散，而新鲜的 context window 对每个 Story 都能给出全力专注的响应。

代价是：跨迭代的信息必须显式序列化到文件。这正是 progress.txt 中 Codebase Patterns 章节的意义——把隐性的上下文记忆提炼为显式的结构化知识，让每个新实例都能快速"热身"。

2.2 文件即状态机（File-Driven State）

Ralph 没有数据库、没有内存中的状态对象，所有状态都存储在三个文件中：prd.json（任务状态）、progress.txt（学习积累）、git history（代码快照）。这个选择带来的好处远超存储本身：

  • 可恢复：进程崩溃后重启，从 prd.json 的 passes 字段断点续跑
  • 可观测：任何时刻 cat prd.json 即可了解全局进度
  • 可干预：用户可以手动编辑 prd.json 跳过某个 Story 或修改优先级
  • 可归档：不同特性的运行记录自动存入 archive/ 目录，互不干扰

2.3 竞品横向对比

Ralph 的定位是"最小可用的自主迭代循环"，与市场上其他 Agent 方案在架构复杂度与适用范围上有明显差异：

工具核心定位优势劣势
Ralph 极简 Bash 循环驱动的自主迭代 Agent 零框架依赖、双工具支持、可恢复、PRD 驱动 无复杂度感知、无并行执行、无 UI、依赖预先写好的 PRD
GSD 2 磁盘状态机驱动的完整 Agent 编排框架 复杂度分级、模型动态路由、崩溃恢复、Git worktree 隔离 TypeScript 重型框架、初始学习成本高、依赖闭源 Pi SDK
Aider Git-native 的交互式 AI 代码编辑 与 Git 天然结合、多模型支持、轻量 无自主循环、无任务拆解、每步需要人工驱动
Claude Code Anthropic 官方交互式 AI 编程助手 工具调用能力强、与 Claude 深度整合、即时反馈 无持久状态、长任务上下文污染、无自主调度

Ralph 的核心赢面：需要无人值守推进一个已明确拆解好的需求列表时，Ralph 是成本最低的可用选项——不需要学框架，只需要一个写好的 prd.json。输面：任务拆解（PRD 准备）是用户责任，Ralph 不帮你思考做什么，只帮你做完你已经想清楚的事。

2.4 适合解决什么问题

✅ 适合的任务类型

  • 已有清晰 PRD（User Stories + 验收标准）的功能开发
  • 需要无人值守迭代推进的多步骤需求（超过 3 个 Story）
  • 希望保持干净 git 历史（每个 Story 一次 commit）
  • 需要在 CI/CD 中自动化执行的重复性开发任务
  • 需要跨会话断点续跑（可能被中断的长期任务）

⚠️ 不太适合的任务类型

  • 10 分钟内可以手动完成的小改动（启动 Ralph 成本更高）
  • 探索性研究或需要频繁人工交互的任务
  • PRD 尚未明确、需要边讨论边做的模糊需求
  • 需要实时 UI 反馈或迭代设计决策的前端主导任务
  • 项目完全没有质量检查命令（typecheck/test）的情况

prd.md 示例（喂给 /ralph 技能后生成 prd.json）

# PRD: User 相关结构补全 school 字段

## Introduction

用户（User）已通过 `school_class` 外键间接关联到学校（User → SchoolClass → School），
服务层 `_enrich_user_school_class` 也已正确解析出 `school_id / school_name`，
但以下层面仍缺少 school 字段的完整支持：

1. `BaseUser` schema 缺少 `school_id / school_name / class_id / class_name`
2. 用户列表 API 缺少 `school_id` 过滤参数
3. `get_user_list` service 方法缺少 `school_id` 参数
4. `_build_user_search_filters` 不支持按学校过滤

本 PRD 的目标是在不改变数据模型（不新增 FK）的前提下，把 school 字段贯通到
Schema → Service → API 三层。

---

## Goals

- `BaseUser` schema 返回 `class_id / class_name / school_id / school_name`
- 用户列表接口支持按 `school_id` 过滤
- 全链路类型安全，ruff check 和 mypy 无新增错误
- 不破坏现有的创建/更新/详情接口行为

---

## User Stories

### US-001: BaseUser schema 补全 school/class 字段

**Acceptance Criteria:**
- [ ] `BaseUser`（`src/schemas/users.py`）新增 `class_id / class_name / school_id / school_name` 四个可选字段
- [ ] 现有 `UserCreate` / `UserUpdate` 不受影响
- [ ] `ruff check src/` 无新增错误；`uv run pytest` 通过

### US-002: 用户列表 API 支持 school_id 过滤

**Acceptance Criteria:**
- [ ] `GET /api/v1/users/list` 新增可选查询参数 `school_id: int | None`
- [ ] 传入 `school_id` 时，仅返回该学校下所有班级的用户
- [ ] 与现有 `username / email` 过滤可组合使用

### US-003: UserService.get_user_list 支持 school_id 参数

**Acceptance Criteria:**
- [ ] `get_user_list` 方法签名新增 `school_id: int | None = None`
- [ ] `_build_user_search_filters` 通过 `Q(school_class__school_id=school_id)` 跨表过滤

---

## Non-Goals

- 不在 User 模型上新增直接 `school` 外键
- 不修改创建/更新用户时的 `school_id` 输入逻辑
- 不增加按 `class_id` 过滤的 API（超出本次范围）

3.4 常见误解

误解实际情况
「Ralph 会自动拆解任务」Ralph 只负责执行已拆解好的任务。任务拆解（写 PRD + 转 prd.json）是用户的职责，可借助内置的 /prd 和 /ralph Skill，但 Ralph 本身不生成 prd.json。
「多个 Story 可以同时执行」Ralph 每次迭代只处理一个 Story（priority 最小且 passes=false），严格串行执行。这是刻意设计——并发会带来代码冲突和 prd.json 写入竞争问题。
「progress.txt 越长 AI 越聪明」AI 每次迭代优先读取 Codebase Patterns 章节，而不是全文扫描。随着 progress.txt 增长，应定期将通用规律提炼到顶部章节，避免关键信息被"淹没"在迭代记录中。
「不需要质量检查命令也能用」没有质量门控的 Ralph 会积累破损代码——错误的 Story 会被 commit 后标记为 passes=true，后续 Story 在错误基础上继续叠加。typecheck/test 是 Ralph 工作质量的最基本保障。
「完成信号等于任务正确完成」<promise>COMPLETE</promise> 只表示所有 Story 的 passes=true，不保证代码正确性。质量由 Quality Gate（typecheck/test）保障，信号只是退出条件。

4 项目使用方式

4.1 从需求到交付的完整过程

以一个具体需求为例：为现有任务管理系统新增任务优先级功能（高/中/低三级，带筛选）

① 调用 PRD Skill 生成需求文档
在 Amp 或 Claude Code 中运行 Load the prd skill and create a PRD for 任务优先级功能，Skill 会提出 3-5 个澄清问题（如范围、用户群、成功指标），根据回答生成 Markdown PRD 并保存到 tasks/prd-task-priority.md。
② 调用 Ralph Skill 转换为 prd.json
运行 Load the ralph skill and convert tasks/prd-task-priority.md to prd.json，Skill 将 PRD 中的 User Stories 拆解为每个可在一次 context window 完成的原子 Story，生成 prd.json（含 branchName、优先级、验收标准）。
③ 启动 Ralph 循环
执行 ./scripts/ralph/ralph.sh --tool claude 15，Ralph 检查归档条件，启动最多 15 次迭代的循环。第一次迭代 spawn Claude Code，Claude 读取 prd.json + progress.txt，检出 ralph/task-priority 分支。
④ AI 逐 Story 自主实现
Claude 选取 priority=1 的 Story（添加 DB 字段），实现并运行 typecheck，通过后提交 feat: US-001 - Add priority field to database，将 prd.json 中 US-001 的 passes 改为 true，追加学习记录到 progress.txt。ralph.sh 未检测到完成信号，继续下一迭代。
⑤ 迭代直到所有 Story 完成
后续迭代依次完成 US-002（UI 组件）、US-003（编辑面板）、US-004（筛选功能）。每个 Story 独立验证、独立提交。UI Story 包含浏览器验证步骤。每次 ralph.sh 解析输出中是否含 COMPLETE 信号。
⑥ 完成信号触发退出
最后一个 Story 完成后，Claude 检查 prd.json 中所有 passes=true，在输出中回复 <promise>COMPLETE</promise>。ralph.sh grep 检测到信号，打印 "Ralph completed all tasks! Completed at iteration X of 15"，exit 0。

5.3 关键模块实现细节

5.3.1 关键模块一：主循环状态机（ralph.sh）

📁 ralph.sh 🏷️ 编排层：Bash for 循环 + 信号检测 ⚡ 触发时机：用户直接执行 ./ralph.sh

要解决的问题：如何以最低依赖让 AI 工具持续执行多个独立任务，并在所有任务完成后自动退出？对话式 AI 工具本身没有"循环"概念——用户每次手工触发、手工判断是否继续，不适合无人值守的多 Story 场景。

解决方案：用 Bash for 循环驱动 AI 进程，每次迭代完全重新 spawn，避免会话状态积累。完成判断不依赖外部监控，而是由 AI 自己在输出中嵌入信号串；ralph.sh 用 grep -q 解析每次迭代的 stdout，检测到信号立即 exit 0。这种"协议即信号"的模式是 Ralph 最核心的设计决策。

sequenceDiagram participant SH as ralph.sh（主循环） participant FS as 文件系统 participant AI as AI 进程 SH->>FS: 读取 prd.json.branchName SH->>FS: 读取 .last-branch SH->>SH: 比较 branchName，决定是否归档 loop for i in seq 1 MAX_ITERATIONS SH->>AI: spawn 全新 AI 进程（注入指令文件） AI-->>SH: OUTPUT（实时通过 tee 输出到 stderr） SH->>SH: echo $OUTPUT | grep -q COMPLETE alt 检测到完成信号 SH->>SH: exit 0 else 未检测到 SH->>SH: sleep 2，继续 end end SH->>SH: exit 1（超出最大迭代次数）

5.3.2 关键模块二：智能体指令协议（prompt.md / CLAUDE.md）

📁 prompt.md · CLAUDE.md 🏷️ 执行层：每次迭代的完整行为规范 ⚡ 触发时机：每次 AI 进程 spawn 时注入

要解决的问题：每次迭代的 AI 实例没有记忆——它不知道前一次做了什么、当前项目有什么规范、任务从哪里来。如何在不依赖会话历史的前提下，让每个新实例都能准确完成当前任务？

解决方案：prompt.md / CLAUDE.md 是一份完整的"行为契约文档"，明确规定了每个实例的完整工作流：先读哪些文件、如何选 Story、质量检查的要求、如何写 progress、何时输出完成信号。通过文件内容注入代替会话历史，实现"无记忆也能准确工作"。两份文件内容几乎相同，差异仅在于 AGENTS.md vs CLAUDE.md 的更新目标和 thread URL 记录（Amp 特有）。

5.3.3 关键模块三：Archive 归档机制

📁 ralph.sh（Archive 段） 🏷️ 编排层：多特性运行隔离 ⚡ 触发时机：ralph.sh 启动时检测到 branchName 变更

要解决的问题：当用户开始一个新特性（更换 prd.json）时，上一次运行的 prd.json 和 progress.txt 会被新内容覆盖，历史进度和学习记录丢失。同时 progress.txt 如果不清空，新特性的 AI 实例会读到大量与当前任务无关的旧记录。

解决方案：在 ralph.sh 启动时比较当前 prd.json 的 branchName 与 .last-branch 文件的记录。若两者不同，说明用户切换了特性，自动将旧 prd.json 和 progress.txt 拷贝到 archive/YYYY-MM-DD-feature-name/，然后重置 progress.txt 为仅含头部的全新文件。这是一个纯 Bash 实现的轻量"状态隔离"机制，对用户完全透明。

sequenceDiagram participant SH as ralph.sh participant FS as 文件系统 SH->>FS: jq .branchName prd.json → CURRENT_BRANCH SH->>FS: cat .last-branch → LAST_BRANCH SH->>SH: 判断 CURRENT_BRANCH != LAST_BRANCH alt 分支名不同（切换特性） SH->>FS: mkdir -p archive/YYYY-MM-DD-feature-name/ SH->>FS: cp prd.json → archive/.../prd.json SH->>FS: cp progress.txt → archive/.../progress.txt SH->>FS: 重置 progress.txt（仅保留 "# Ralph Progress Log" 头部） SH->>FS: echo CURRENT_BRANCH > .last-branch else 分支名相同（同一特性继续） SH->>SH: 跳过归档，直接进入主循环 end SH->>FS: echo CURRENT_BRANCH > .last-branch（无论是否归档）

6 方案评价

6.1 优势分析

极简依赖，工具无关

整个编排层是不到 100 行 Bash，唯一外部依赖是 jq。通过 --tool 参数轻松切换 Amp 和 Claude Code，不需要修改任何业务逻辑。未来接入新 AI 工具只需修改一行 spawn 命令。

架构级上下文隔离

不是通过 prompt 压缩历史，而是通过进程重启彻底切断上下文积累。每个 Story 得到完整的、干净的 context window，对话质量不随任务数增长而衰减。

完全可观测可干预

所有状态（任务进度、学习记录、代码变更）都是人类可读的文件。任何时候可以查看进度、手动修改 prd.json 跳过或修改 Story、用 git log 审查提交历史、或直接编辑 progress.txt 修正错误记录。

断点续跑与自动归档

进程崩溃后重启，自动从 prd.json 中最后一个 passes=false 的 Story 继续。切换特性时自动归档历史数据，不同特性的运行记录互不干扰。

6.2 局限分析

用户需要提前写好 PRD

Ralph 不帮你思考"做什么"，只帮你做完"已经明确的事"。需求模糊、Story 拆解粗糙或验收标准不可验证，会直接导致 AI 实现偏差甚至死循环。PRD 准备质量是 Ralph 成败的关键前置条件。

严格串行，无并发优化

Story 按优先级串行执行，即使多个独立 Story 可以并行，Ralph 也不支持。大型 PRD（20+ Stories）需要数十次迭代才能完成，无法通过并发加速。

无复杂度感知，无成本控制

无论 Story 大小，每次迭代都用同一个 AI 工具和同等的 context budget。没有 GSD 2 那样的复杂度分级和模型动态路由，大量简单 Story 会产生不必要的成本。

错误静默吞没

ralph.sh 用 || true 吞掉了 AI 进程的非零退出码，无法区分"AI 正常结束但未完成"和"AI 进程崩溃"两种情况。调试失败场景时需要手动阅读 stderr 输出，缺乏结构化错误处理。

progress.txt 膨胀导致长上下文风险

progress.txt 采用 append-only 设计，迭代越多文件越长；每轮 AI 又被要求先读顶部的 Codebase Patterns，重复占用 token。若把大段 diff、详细日志写入，膨胀更快，最终可能触及模型上下文窗口上限。设计层面已有部分缓解手段：把可复用规律收入顶部 ## Codebase Patterns 短列表；把目录约定写到 AGENTS.md / CLAUDE.md 减轻 progress 负担；换 branchName 时 ralph.sh 将旧文件归档到 archive/ 并重置，等于阶段性清空记忆。自行维护时建议定期轮转或只保留最近 N 次迭代记录。
