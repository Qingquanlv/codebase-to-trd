Specmatic 深度解读

把Spec编译成统一场景模型，再分发给契约测试、Stub 与兼容检查 — Kotlin / JVM，多协议（OpenAPI、AsyncAPI、GraphQL、gRPC、WSDL 等）
specmatic 是面向契约驱动开发（CDD）的工具集：同一份 API 规格既可驱动对真实服务的测试，也可起“预设响应型”的 mock 服务（Stub） 替代下游，并支持向后兼容检查、代理录例、JUnit5、MCP 等周边能力。 Specmatic 说的不是“由契约直接生成业务代码”，而是 Contract-Driven Development：先把 API 规格当成可执行契约，围绕它来驱动协作、联调、验证、替身和版本演进，类似原TDD，BDD的思路。

核心结论

  • Spec = 可执行编译器 — Specmatic 会在运行时把 OpenAPI 这类规格翻译成内部统一对象，再把这套对象同时拿去做测试和起 Stub。它不会存一份中间 YAML。
  • 规格为中心，而不是入口为中心 — 不管输入是 OpenAPI 还是别的规格，Specmatic 都会先把它收敛成统一的 Scenario；真正被执行、被匹配、被复用的，不是原始规格文本，而是这套统一场景模型。
  • Stub 与 Test 镜像 — 测试链和 Stub 链并不是两套平行实现，而是围绕同一批 Scenario 运转；正因为它们共用同一套契约语义，测试结果和替身行为才更容易对齐。
  • 架构有显式代价 — 通过把很多核心逻辑集中到 Feature 换来了 test / stub / examples 的统一复用；但代价是 Feature 这个对象会越来越重，代码阅读、维护和边界判断也会更复杂。（§6.2）。

1 摘要

1.1 项目背景

微服务互相依赖时，集成测试昂贵且脆弱；若团队各自解释 OpenAPI，很容易出现「文档写了、实现没跟上」或「消费者假设与提供者真实行为不一致」。Specmatic 把规格文件本身当作可执行契约：在 CI 中对运行中的 API 做符合性检查，或用 Stub 在本地挡住不稳定下游，从而支持更频繁的独立部署。官方叙事见仓库 README 与 specmatic.io。

技术栈：Gradle（Kotlin DSL）、Kotlin 2.3、JUnit 5 / Platform、Ktor、Jackson、Swagger Parser、Cucumber Gherkin、部分能力依赖 Netty / OkHttp 等。构建：根目录 ./gradlew；产物包含可执行 JAR（application 模块）、Maven 中央库上的 specmatic-core、junit5-support、specmatic-mcp 等。

规模（粗估）：core/src/main/kotlin 下约 509 个 Kotlin 源文件；application 约 35 个，负责 Picocli 子命令装配。目标用户：需要契约测试、API Mock、兼容门禁或代理录例的后端团队与平台工程。

1.2 能解决的问题

1.2.1 「文档与实现悄悄分叉」——规格符合性验证

OpenAPI 文档写好、但服务实现逐渐偏离，是微服务团队最常见的隐患。Specmatic 把规格文件编译成可执行的 Scenario 集合，在 CI 中对运行中的服务发起真实 HTTP 调用并断言，而不是只做静态 diff。任何字段缺失、状态码不符或类型错误都会立即让 CI 失败（见 §4.1、§5.2.1）。

1.2.2 「下游未就绪、联调被阻塞」——基于同源规格的 Stub

消费者团队需要在下游服务还没完工时照常开发和测试。specmatic stub 直接从同一份规格起一个与契约行为一致的 HTTP 替身，匹配逻辑与契约测试共用同一套 Feature / Scenario，不会出现「挡板行为与真实服务规则不同步」的漂移问题（见 §5.2.2、§5.3.3）。

1.2.3 「API 版本升级悄悄破坏下游」——向后兼容检查

提供者升级规格时，很难人工判断所有变更对已有消费者是否安全。Specmatic 内置 BackwardCompatibilityCheckCommandV2，对两个版本的规格做差异分析，把「删字段、改类型、去掉响应码」等破坏性变更在发布前拦截，作为 CI 门禁（§3.4）。

1.2.4 「真实流量与规格脱节」——代理录例回填

已有服务缺乏规格、或规格年久失修时，可以用 specmatic proxy 让真实流量经过代理，把实际的 Request / Response 对录制为示例文件并反向推导契约素材，形成「从线上学习、回填到规格」的闭环（§4.3；源码定位 §5.2.3）。

2 项目设计理念

2.1 规格编译为统一场景模型

OpenAPI / AsyncAPI 等输入经 Swagger 解析与内部转换，落成可遍历的场景：Scenario 携带 HttpRequestPattern / HttpResponsePattern、示例行、绑定、协议元数据等。后续无论是生成外向请求还是 Stub 内向匹配，都针对这一模型操作，而不是各子系统各自解析 YAML。

2.2 同一场景模型同时服务 Test 与 Stub

Feature 聚合场景列表并提供 lookupResponse（Stub 路径）与供 ScenarioAsTest 使用的同一套 Scenario（Test 路径）。设计取舍是一致性优先：匹配器、示例、状态机在两条链路上复用，减少「挡板与契约测试各写一套规则」的漂移。

2.3 以 JUnit Platform 为执行编排中心

specmatic test 不显式手写「for each scenario」循环，而是构造 LauncherDiscoveryRequest 仅选择 SpecmaticJUnitSupport，由 JUnit 完成发现、并行、listener 与报告钩子。这样 CLI 成为薄编排：把 CLI 参数写入线程本地配置后，把控制权交给与 IDE 相同的执行内核（详见 §5.2.1）。

3.4 常见误解

误解实际情况
「Specmatic 只做 Stub」Stub 与契约测试同等重要；specmatic test 是主路径之一（§4.3）。
「只校验文档、不测真实服务」默认外向路径对 testBaseURL 发真实 HTTP，用响应匹配契约期望（§5.2.1、§5.3.2）。
「和 Schemathesis 一样是随机属性测试主导」Specmatic 以规格符合与示例/模式驱动为主，不是 Hypothesis 式海量随机 fuzz；选型见 §6.2。
「等同于 Pact 消费者驱动契约」Specmatic 以规格文件为权威源组织协作，与消费者驱动的双向发布模型不同（可并存，但不等价）。

4 项目使用方式

4.1 从一份 OpenAPI 到一次契约测试执行过程

具象带读：假设 api.yaml 里有一条操作：POST /orders，契约写明「成功时应返回 201，且 JSON body 里必须出现 orderId（例如 string 类型）」。你把订单服务起在 http://localhost:8080，然后执行 specmatic test api.yaml --testBaseURL=http://localhost:8080。Specmatic 会先把这条操作（及响应 schema / 示例）编译进 Feature 下的一条或多条 Scenario；JUnit 发现 SpecmaticJUnitSupport 后，为其中一条场景构造 ScenarioAsTest，由它生成符合契约形状的 POST /orders 请求体、发到真实 baseURL，再把实际 HTTP 响应与契约中的响应模式做匹配——若你的实现返回 200 却漏了 orderId，或状态码不是 201，这条场景就会失败并计入退出码。

同一逻辑链的抽象视图如下（与 §5.2 各图对应；上面例子即图中「每个 Scenario」里的一次迭代）。

4.2 最小可运行路径

下列命令为形态示例：具体 flags 以 specmatic --help 与 官方文档为准；路径需换成本地契约与被测地址。

契约测试（外向）

# 对运行中的实现跑契约（示例）
specmatic test api.yaml --testBaseURL=http://localhost:8080

Stub（内向）

# 基于契约起本地替身（端口等按子命令帮助填写）
specmatic stub api.yaml

代理录例

# 流量经代理录制/转换，再回填契约或示例（详见文档）
specmatic proxy …

三条路径在 SpecmaticCommand 中均以独立子命令注册（TestCommand、StubCommand、ProxyCommand 等，见 §5.2），共享同一套 core 解析能力，但进程目的不同：test 偏「验证」、stub 偏「替代」、proxy 偏「观测与素材回流」。

4.3 核心使用模式

验证真实服务

CI 或本地启动被测服务后，用 specmatic test 将契约中每条 Scenario 转为 HTTP 调用并断言。适合作为部署门禁：规格变更必须同步通过契约套件。

Stub 替代下游

消费者团队在下游未就绪或环境不稳定时，用 specmatic stub 起与契约一致的 HTTP 替身，减少联调摩擦（实现落点 §5.2.2、§5.3.3）。

代理录例 → 回填契约

让流量经过代理，把真实交互沉淀为可复用示例或契约素材，再与 strict 模式、示例目录等策略结合，形成「从线上学习」的闭环（proxy 包与 ProxyCommand）。源码级定位见 §5.2.3（本文对 proxy 只作定位，不做与 test/stub 同深度的展开）。

5.2 三条执行调用链

呼应 §4.3 核心使用模式，按三条路径分别追踪从命令入口到核心动作的完整函数跳转链。所在模块列对应仓库文件路径。

5.2.1 test 链

specmatic test 是 Specmatic 的外向契约验证路径。进程启动后，CLI 把所有配置（testBaseURL、并行度、过滤器）写入线程本地，随后委托给 JUnit Platform：仅发现 SpecmaticJUnitSupport 这一个测试类，由它展开全部 Scenario 为 DynamicTest；每个 ScenarioAsTest 独立生成 HTTP 请求、发向被测服务、匹配响应，最终由 ContractExecutionListener 汇总退出码。

与 stub 链的核心区别：test 是主动出向——进程是客户端，目标是正在运行的真实服务；stub 是被动入向——进程是服务端，等待客户端连入。两条链共享同一套 Feature / Scenario 领域模型，差异仅在编排层（JUnit vs Ktor）与 I/O 方向。

5.2.2 stub 链

specmatic stub 是 Specmatic 的内向契身替代路径。进程启动后解析端口与契约路径，构造 HttpStub（同时加载 Feature 与示例目录），启动 Ktor Netty 监听；当客户端发来请求，Ktor 将其转交 Feature.lookupResponse，由场景匹配逻辑找到最优 Scenario 并合成响应，最终返回给客户端。进程保持运行直到 SIGINT 或超时。

与 test 链的核心区别：stub 进程是长驻服务端，每次请求在内存中完成匹配与响应合成，不发出任何出向 HTTP；test 进程是短促客户端，跑完全套 Scenario 后即退出。两条链共用 OpenApiSpecification → Feature → Scenario 规格编译路径。

5.2.3 proxy 链（定位）

specmatic proxy 是 Specmatic 的流量录例与回填路径，与 test / stub 并列为三大主路径之一。进程在本地或流水线中作为中间代理，把真实流量透明转发到上游服务，同时录制请求/响应对，经 ExampleTransformer 转换为可复用的示例文件或契约素材，回填至示例目录或规格。

本文展开范围：proxy 实现横跨 io.specmatic.proxy 录例/转换与 Stub 示例管道衔接处，属于「工具链支线」，涉及文件写出、流量整形与团队工作流，深拆会拉长篇幅。此处仅列调用链骨架；需要深挖时从 application 中的 ProxyCommand 起跳，沿 proxy 包内初始化与 handler 阅读即可。

5.3 关键模块实现细节

5.3.1 关键模块一：OpenAPI → Feature / Scenario（规格编译）

📁 core/src/main/kotlin/io/specmatic/conversions/OpenApiSpecification.kt · core/.../core/Feature.kt 🏷️ 编译器模式：规格文件 → 统一场景模型

要解决的问题：把多格式规格（含 $ref、overlay、示例目录、字典、AsyncAPI）变成内存中可遍历、可匹配、可生成的场景集合，同时对外屏蔽格式差异，让 test 与 stub 路径共享同一套场景模型。

解决方案：OpenApiSpecification.fromFile() 触发 Swagger Parser 展开 $ref，得到 io.swagger.v3.oas.models.OpenAPI 对象；再经 toFeature() 折叠为 Feature(scenarios: List<Scenario>)，同时合并 overlay、加载示例目录、应用 strict/lenient 配置。最终每条 path × method × status × 示例行产生一个独立的 Scenario。

5.3.2 关键模块二：ScenarioAsTest（外向测试执行）

5.3.3 关键模块三：HttpStub 场景匹配（内向 Stub）

📁 core/src/main/kotlin/io/specmatic/stub/HttpStub.kt · core/.../core/Feature.kt 🏷️ Chain of Responsibility 变体：顺序评估 Scenario 列表取最优匹配

要解决的问题：对任意入站 HTTP 请求，在已加载的场景集合中找到最能「解释」该请求的场景，并合成符合契约的响应——而不是简单的精确路径匹配（请求体内容、头、查询参数均参与匹配打分）。

解决方案：Ktor 路由将入站请求转为 HttpRequest 后委托给 Feature.lookupResponse；后者调 lookupScenario 对全部场景逐一匹配打分，取最优者调 generateHttpResponse(serverState) 合成响应；finally 块无条件清空 serverState（防止跨请求污染）。未命中场景时合成 400/500 级错误响应并附说明。

6 方案评价

评价条目与 §5、§2 显式挂钩，避免「放之四海皆准」的空泛优缺点。

6.1 优势分析

契约与替身同源

与 §2.2、§5.3.3 一致：测试与 Stub 共享 Feature/Scenario，降低规格漂移风险。

多协议与工程化

仓库内除 REST/OpenAPI 外尚有 WSDL/SOAP、GraphQL、gRPC、AsyncAPI 等子系统；大量测试与发布流水线体现可维护性。

JUnit 生态对齐

对应 §2.3：specmatic test 走 JUnit Platform，IDE 与 CI 接入成本低。

命令到类的映射清晰

工程上主链可落到具体类型：specmatic test → TestCommand → JUnit → SpecmaticJUnitSupport → ScenarioAsTest；specmatic stub → Ktor → HttpStub → Feature.lookupResponse（见 §5.2）。新人可用 §3.5 顺序在仓库内自导航。

6.2 架构代价与取舍

以 Feature 作为中心枢纽

把 test、stub、examples 都收在 Feature 这一个中心对象上，好处是复用更强、规则更统一；代价是这个对象会越来越重，文件更大、职责更多，一个小改动也更容易同时影响多条链路。

JUnit Platform 编排

把契约测试挂到 JUnit Platform 上，好处是能直接复用 Surefire、并行执行、listener 和报告生态；代价是排障时不能只看 Specmatic 自己，还得同时看 CLI 参数、线程本地配置、系统属性和 JUnit 环境。

同规格驱动 Test 与 Stub

让 test 和 stub 共用同一套规格与场景，好处是两边更容易保持一致；代价是 strict 模式、示例策略、headers 等边界规则会同时影响两条链，出了问题通常要回到同一套匹配逻辑里查。

模糊测试与边界发现

Specmatic 很擅长验证“当前实现是否符合已经写下来的契约”；但它不是以随机探索未知 bug 为主的工具。若团队的重点是 fuzz 和边界发现，通常还需要再配 Schemathesis 这类工具。
