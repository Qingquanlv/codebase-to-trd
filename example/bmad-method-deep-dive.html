BMAD-METHOD 深度解读

npm bmad-method · 敏捷 AI 驱动开发 · 安装器 + 技能模块生态
BMAD-METHOD（BMAD Method）是 BMad Code 维护的开源仓库：通过 bmad-cli（bmad / bmad-method 双 bin）将核心技能（src/core-skills/）与 BMM 等方法模块（src/bmm-skills/）安装到目标项目的 _bmad 目录，并借助 IdeManager 写入 Claude Code、Cursor 等 IDE 配置。卖点包括 bmad-help 引导、Party Mode 多角色协作、以及 Scale-Domain-Adaptive 叙事下的工作流深度自适应。技术栈：Node ≥20、Commander、@clack/prompts、yaml/js-yaml；文档站与教程见官方站点。

核心结论

  • 「发行版 = 技能源 + 安装器」 — 业务价值主要在 src/*-skills/ 的 Markdown/YAML 资产；tools/installer/ 负责交互/无头安装与升级
  • Installer 为中心编排器 — installer.js 协调模块清单、已安装检测、文件拷贝、ManifestGenerator 与 IDE 钩子
  • IDE 适配配置驱动 — platform-codes.yaml 描述各平台安装行为，IdeManager 懒加载 handler
  • 非交互安装可进 CI — install 支持 --directory --modules --tools --yes 等旗标
  • 质量闸门在仓库内 — validate:skills、validate:refs、markdownlint 等与 test/ 保障技能引用一致

1 摘要

1.1 项目背景

生成式 AI 已能写代码，但缺少可重复的敏捷仪式与角色分工时，输出易发散、难与团队治理对齐。BMAD Method把分析、规划、架构、实现等阶段沉淀为可安装、可版本化的技能与工作流文件，并通过一条 CLI 安装到每个仓库，使 Cursor / Claude Code 等环境中的 Agent 能按同一套「方法模块」协作。

仓库同时包含 Astro Starlight 文档站（website/）与大量校验脚本（tools/validate-*.js），形成「内容 + 工具链」单体仓库。

1.2 能解决的问题

1.2.1 「AI 输出发散，缺少可重复的敏捷仪式」——结构化工作流模块化安装

团队使用 AI 写代码时，常因没有统一的分析→规划→架构→实现节奏，导致 Agent 输出方向不一致、难与项目目标对齐。BMAD-METHOD 把敏捷各阶段的角色与工作流沉淀为 src/*-skills/ 下可版本化的 Markdown/YAML 资产，通过 bmad-cli 一条命令装入任意项目的 _bmad 目录，让每个仓库共享同一套方法论基线。

1.2.2 「AI 配置文件散落各处，多 IDE 同步是噩梦」——统一 IDE 挂载与清单维护

Claude Code、Cursor、Windsurf 等每个 IDE 都有自己的 rules/config 格式，手动维护多端一致性成本极高。IdeManager 读取 platform-codes.yaml 按需写入各 IDE 配置，ManifestGenerator 维护 _config 清单；支持 --tools claude-code,cursor 同时挂载多端，也可进 CI 以 --yes 无头运行，实现配置即代码。

1.2.3 「Agent 不知道能做什么，技能发现全靠记忆」——bmad-help 可发现技能入口

随着团队积累的技能文件增多，Agent 无法自动感知哪些工作流可用，每次都要靠开发者手动粘贴背景。bmad-help 作为统一入口技能，引导 Agent 和用户发现已安装的核心技能与方法模块（core / BMM / BMB / TEA），无需阅读完整文档即可按需激活正确的工作流。

1.2.4 「单一 AI 视角盲区多，决策质量不稳定」——Party Mode 多角色协作

重要决策仅依赖一个 AI 角色时，容易产生确认偏差和盲区。Party Mode 让多个专家角色（如 PM、架构师、QA）在同一上下文中以不同视角评审方案，形成类似结构化头脑风暴的协作模式，在规格评审、架构选型等关键节点提升决策质量，同时保留完整的讨论轨迹。

2 项目设计理念

2.1 核心理念

  1. AI 作协作者而非替代思考：README 强调 facilitated workflows 引导人产出更好判断。
  2. 方法与工具链绑定：通过安装器把同一方法论同步到各项目与 IDE。
  3. 模块化扩展：BMM、BMB、TEA 等官方模块按需叠加，避免单仓无限膨胀认知负载。

2.2 与同类方案横向对比

工具核心定位落地成本落地效果
BMAD-METHOD 敏捷全阶段方法论 + npm 安装器 + 多 IDE 挂载 中：npx bmad-method install 即可启动，但需学习 bmad-help / Party Mode 等概念，技能文件体量大 覆盖需求→规划→架构→实现全链路；Party Mode 多角色协作；跨 IDE 一次同步；适合追求完整敏捷仪式的中大型团队
GitHub Spec Kit 规格驱动开发（SDD）+ AI 命令行工具集 低：Cursor Agent Skill 形式，/speckit.specify 系列命令开箱即用，无需额外安装产物 强于「需求→任务→GitHub Issues」端到端规格落地，但不含 BMAD 式多角色仪式和 Party Mode；与 BMAD 互补而非替代
OpenSpec 规格变更管理 + ArtifactGraph 拓扑排序 + Zod 验证 中：需引入 CLI 和 ChangeSchema 概念，变更流程较严格 变更可审计、规格文件是真相来源；偏「规格一致性维护」，不含敏捷引导和多角色技能；可与 BMAD 配合（BMAD 做流程、OpenSpec 守规格）
Superpowers AI 协作行为规范 + Iron Law + Skill.md 元技能体系 极低：复制 SKILL.md 即可使用，无安装步骤，无额外依赖 解决「AI 自作主张」和「会话重置后忘记约定」的问题；不含敏捷阶段工作流或多 IDE 同步能力；BMAD 可以视为 Superpowers 理念在敏捷流程层面的重量级实现

总结：BMAD-METHOD 的核心赢面在于完整的敏捷工作流覆盖（需求到交付）+ Party Mode 多角色 + 跨 IDE 一键安装；Spec Kit 和 OpenSpec 更偏规格/任务维度，Superpowers 更偏行为约束维度，三者与 BMAD 是正交互补关系，可以叠加使用。

2.3 定位一句话

把可版本化的敏捷 AI 方法模块装进任意项目目录，并用安装器完成跨 IDE 挂载与清单维护的「方法操作系统」发行版。

2.4 当前适合解决什么问题

适合

  • 希望团队在多仓库间共享同一套 AI 敏捷工作流与角色
  • 已使用 Claude Code / Cursor 等，需要可发现技能（bmad-help）与结构化步骤
  • 需要 非交互 CI 安装 统一基线（--yes 等）
  • 想要 Party Mode 类多视角评审/头脑风暴

不太适合

  • 团队拒绝在仓库中落任何生成目录（_bmad）
  • 仅需极简单文件提示词，不接受 npm 与 Node 20 依赖
  • 强合规环境禁止从 npm 拉取未审计包（需自行镜像与签核）

4 项目使用方式

4.1 从需求到交付的一次完整过程

以「从零搭建一个新功能」为例，展示四个阶段的全部可用技能。每步均为按需选用，非必须全部执行。已有代码库（brownfield）时，可先用 bmad-generate-project-context 生成精简上下文再接入后续步骤。

① Analysis（分析）
按需从「轻」到「重」选，不是每步都必须。已有代码库时可先跑 [GPC] 生成项目上下文。
[CB]bmad-product-brief产品简报——概念已较清晰时快速结构化目标、背景与约束。
[WB]bmad-prfaqPRFAQ 逆向工作法——从「新闻稿 + FAQ」倒推产品定义，与 [CB] 二选一或互补。
[BP]bmad-brainstorming发散头脑风暴，破冰、探索方向，适合需求模糊初期。
[MR]bmad-market-research市场与竞品调研，输出结构化洞察。
[DR]bmad-domain-research领域术语与业务约束梳理，为 PRD 打底。
[TR]bmad-technical-research技术可行性与方案对比，帮助在架构阶段做有据可查的决策。
[GPC]bmad-generate-project-context已有代码库时生成精简项目上下文，供后续 LLM 高效消费（brownfield 常用）。
② Planning（规划）
需求与设计阶段。大多数项目从 [CP] 开始。
[CP]bmad-create-prd生成 PRD——多数项目的「正式起点」，输出完整产品需求文档。
[VP]bmad-validate-prdPRD 质量校验——在 PRD 产出后运行，检查完整性与一致性。
[CU]bmad-create-ux-designUX 设计——UI 比重大时强烈建议，在 PRD 之后执行。
[EP]bmad-edit-prd迭代修改 PRD，响应评审反馈或需求变更。
③ Solutioning（方案）
技术设计与任务拆分，依赖 PRD 完成后执行。
[CA]bmad-create-architecture生成架构文档与关键技术决策记录（ADR）。
[CE]bmad-create-epics-and-stories在架构之后拆分 Epic + User Story，需明确依赖架构输出。
[IR]bmad-check-implementation-readiness就绪检查——验证 PRD / UX / 架构 / Epic-Story 是否对齐，确认能否开干。
④ Implementation（实现）
迭代交付阶段。Story 小循环（CS → DS → CR）会多轮重复，[SS] 随时可查。
[SP]bmad-sprint-planning生成或更新 Sprint 计划，作为实现阶段的「路线图」。
[CS]bmad-create-story创建并校验单条 Story（含 Validate Story 等子动作）。↩ 循环起点
[DS]bmad-dev-story按 Story 实现、测试、修改。
[CR]bmad-code-review代码评审——通过则进下一条 Story；不通过回 [DS]。↩ 循环终点
[SS]bmad-sprint-status随时查看 Sprint 进度，决定下一步走哪个 workflow。
[QA]bmad-qa-generate-e2e-tests补充自动化 API / E2E 测试，在实现阶段与 [DS] 衔接运行。
[ER]bmad-retrospective回顾——一个大块 / Epic 结束时可选执行，沉淀经验。

4.2 最小可运行路径

三步即可开始：安装 → 唤起 bmad-help → 按向导推进。bmad-help 会自动检测已完成内容并告知下一步，无需记忆工作流顺序。

# Step 1: 安装（在项目根目录执行）
npx bmad-method install

# 非交互 / CI 示例
npx bmad-method install --directory /path/to/project --modules bmm --tools claude-code --yes

# Step 2: 打开 AI IDE，立即唤起智能向导
bmad-help

4.3 三条规划路径对照

根据项目规模和复杂度选择对应路径，story 数量仅作参考，实际应按规划需求而非数字选择。

6 方案评价

6.1 优势分析

端到端方法库

从头脑风暴到实现的技能与工作流覆盖面广，减少团队重复造轮子。

安装器成熟

交互/无头双模式、升级检测、多模块与多 IDE 组合，工程化程度高。

开源与文档站

MIT 许可；Starlight 站点降低学习曲线。

仓库内质量闸门

skills/refs 校验与测试目录体现可维护性意识。

6.2 局限分析

体量大

克隆与 IDE 索引成本高；新人易被目录淹没。

运行时依赖 Node

纯文档团队也需装 Node 才能跑官方安装路径。

与宿主产品耦合

各 IDE 插件策略变化时，platform-codes 需跟进。
