Oh My OpenAgent 深度解读

OpenCode 插件 · 多模型编排 · Hashline · LSP / AST-Grep / Tmux
Oh My OpenAgent 是面向 OpenCode 终端 AI Agent 的「全家桶插件」，把多模型编排、Hashline 锚点编辑、LSP / AST-Grep 工具链和 Claude Code 兼容层整合为一个 npm 包，一次安装即获得原本需要逐个手配的工程级 harness 能力。

核心结论

  • Prompt 即合同，Hook 是执行者 — OmO 的最大设计亮点在于把"文字约定"与"运行时强制"分离：Prompt 写规则，Hook 真正保证规则被执行。二者缺一不可，任何一端缺失都会造成协议空转。
  • 多 Agent 角色分工精确，委派链可审计 — 11 个 agent 按 primary / subagent 两层分工，Sisyphus 只编排不实现，Hephaestus 只实现不规划，Prometheus 只规划不动手；每层职责边界由 allowedTools 白名单在注册时强制，而不是依赖模型自觉。
  • 动态 Prompt 编译降低幻觉面积 — buildDynamic*Prompt() 系列根据实际可用的 agents / skills / categories / model 动态收缩，禁用某个 agent 后其引导段自动消失，模型不会看到指向不存在能力的指令，从根源减少幻觉式工具调用。
  • Hashline 把行号锚点问题从模型侧移到传输侧 — 通过读增强注入哈希、写校验哈希的闭环，行号偏移失败率系统性下降，无需依赖模型在长上下文中"记住"正确行号。
  • Claude Code 零成本迁移 — 兼容层加载现有 hooks / skills / MCP 配置，存量 Claude Code 用户无需重建资产即可切换到 OpenCode 生态，降低迁移决策成本。

1 摘要

1.1 项目背景

终端 AI 编程工具爆发式增长，但「单模型 + 弱编辑原语」常导致上下文浪费与失败编辑。OpenCode 提供开放宿主，而本仓库把作者在多模型、多工具上的试错结果产品化为一个默认插件：编排代理、内置 MCP、LSP、Tmux、Ralph Loop、Todo 强化等，目标是一句 ultrawork 即可进入高强度交付模式。

技术栈：TypeScript、Zod 配置模式、Bun 构建；可选平台二进制通过 optionalDependencies 分发；CLI 入口 bin/oh-my-opencode.js（如 doctor）。

1.2 能解决的问题

1.2.1 「OpenCode 开箱能力不足，工具链需大量手配」——全家桶插件一键激活 LSP / AST-Grep / MCP

原生 OpenCode 虽开放，但 LSP 代码导航、并行后台代理、内置 MCP 服务器等企业级工具需要用户自己拼接。Oh My OpenAgent 将所有常用工具（grep、glob、LSP、ast-grep、session、background 等）和多模型编排逻辑封装为单一插件，注册后即可通过 tool-registry 按需裁剪，无需逐个手配。

1.2.2 「陈旧行号导致 AI 补丁频繁失败」——Hashline 锚点闭环消灭行偏移错误

模型记住的文件片段与磁盘状态不同步时，用行号引用极易打出错位 patch。Hashline 在读文件时把每行转化为 LINE#HASH 锚点；写文件时 hashline_edit 先校验锚点哈希，磁盘漂移则拒绝应用，从协议层而非提示层解决「.harness 问题」。

1.2.3 「多代理编排靠手写提示，模型选择反复手调」——Sisyphus + Category routing 自动化编排

团队使用多个专用 AI 模型时，需要每次在提示里手写「用哪个模型做哪种任务」，稍有变化就要维护大量提示。Sisyphus 作为主编排代理，通过 Category routing 把任务类型（visual-engineering、ultrabrain、deep、quick 等）自动映射到对应模型，用户只需配置一次类别→模型对照表即可。

1.2.4 「从 Claude Code 迁移时 hooks/skills 资产全部废弃」——Claude Code 兼容层零成本迁移

团队已经在 Claude Code 上积累了 hooks、commands、skills、MCP 配置等资产，迁移到 OpenCode 生态意味着全部重建。Claude Code 兼容加载器读取 Claude Code 侧 agents/commands/skills/MCP 等现有配置，让存量资产在 OpenCode 下继续生效，大幅降低迁移摩擦。

2 项目设计理念

2.1 核心理念

  1. Harness 优先：失败往往来自编辑工具与上下文结构，而非仅模型智商；因此投入 Hashline、LSP、AST-Grep 等稳定原语。
  2. 多模型编排：由 Sisyphus 与类别路由把「谁写前端 / 谁啃硬逻辑」自动化，用户少手选模型。
  3. 兼容即迁移成本：Claude Code 兼容降低存量用户切换摩擦。

2.2 与同类方案横向对比

工具定位优势劣势
Oh My OpenAgentOpenCode 上的重型插件：多代理 + 工具全家桶开箱功能密度极高；Hashline；并行 background；官方文档推动 Agent 安装学习曲线陡；默认遥测需治理；强依赖 OpenCode 演进节奏
Vanilla OpenCode可脚本化的开源终端 Agent 宿主轻量、可深度定制多模型编排、LSP 集成、MCP 组合需自建
Claude CodeAnthropic 官方 CLI 工作流产品化体验、模型配套优化供应商锁定叙事；与本项目 README 宣称的开放多模型策略不同
IDE 内嵌 Agent（如 Cursor）编辑器一体化UI 集成交互好与本仓库「终端 + OpenCode」栈不同，可比性主要在「Agent 能力」而非部署形态

2.3 定位一句话

把 OpenCode 变成「多模型、多代理、多工具」的成品 harness 发行版——作者类比为 Debian 之于 Ubuntu：上游是 OpenCode，本仓库是加厚默认体验的一层。

2.4 当前适合解决什么问题

适合

  • 已在或愿意采用 OpenCode，希望少配插件即获得 LSP、并行子代理、内置 MCP
  • 受困于编辑工具陈旧行号导致的失败 patch，愿意启用 Hashline 流程
  • 从 Claude Code 迁移，希望保留 hooks/skills 习惯
  • 需要 ultrawork / Ralph Loop 等「推到完成为止」的交互风格

不太适合

  • 团队禁止第三方遥测且无法接受关闭后的支持成本
  • 不使用 OpenCode、只想要独立 CLI（本仓库是插件而非独立 IDE）
  • 极简场景：只需单文件补全，不需要编排与后台代理的复杂度税

5.2 关键模块实现细节

src/index.ts · src/plugin-dispose.ts · 启动层 · OpenCode 载入插件时

要解决的问题

OpenCode 在配置变更或热重载时会重新调用插件的入口函数 OhMyOpenCodePlugin(ctx)。如果不主动清理上一次的运行时资源，多次调用会叠加出：

  • 后台 agent 任务泄漏：上次 backgroundManager 中仍在运行的子 session 继续占用 token 和 context
  • LSP 进程孤立：lspManager 持有的 language server 进程不会自动退出，持续占用端口和内存
  • MCP 连接悬空：skillMcpManager 已建立的 MCP 连接在新实例初始化后变为僵尸连接
  • Hook 重复注册：disposeHooks 未调用时旧的 event listener 仍在队列里，导致同一事件被处理两次

解决方案

src/index.ts 在模块作用域维护一个 let activePluginDispose: PluginDispose | null = null 单例。每次插件函数被调用时，第一步就是 await activePluginDispose?.()——确保上一次的所有资源在新一轮装配开始前完全释放。装配完成后再把新的 dispose 函数写入 activePluginDispose。

plugin-dispose.ts 的 createPluginDispose() 接收四个资源句柄，返回一个幂等异步函数（内部用 disposePromise 防止并发重入）：

// src/plugin-dispose.ts（核心逻辑）
return async (): Promise<void> => {
  // 幂等保护：并发调用时等待同一个 Promise，不会二次执行
  if (disposePromise) { await disposePromise; return }

  disposePromise = (async () => {
    await backgroundManager.shutdown()      // ① 停止所有后台 agent 任务
    await skillMcpManager.disconnectAll()   // ② 断开全部 Skill MCP 连接
    await lspManager.stopAll()             // ③ 终止所有 LSP 进程
    disposeHooks()                           // ④ 注销全部 hook event listener
  })()

  await disposePromise
}

每一步都包裹在独立的 try/catch 里，某个资源释放失败只记录日志，不阻断后续步骤，保证其他资源一定得到清理。

代价与局限：activePluginDispose 是模块级变量，多插件实例场景下不适用；dispose 顺序（background → MCP → LSP → hooks）是硬编码的，若依赖关系变化需同步修改。

大任务往往需要多轮 LLM 调用才能完成，但 AI 会话天然是一次性的——模型不会在输出后自动"继续做完"。用户如果手动循环提示则效率极低；如果不循环则大量任务会在完成 70% 时停住。需要一种机制让 Agent 在任务未完成时自动驱动下一轮，直到收敛。

解决方案

Ralph Loop（也称 ulw-loop，由 ultrawork / ulw 关键字触发）是 OmO 的核心续写驱动机制：在每轮 LLM 回复结束后，由 continuation hook 检查任务是否真正完成；若未完成，自动向会话注入「继续」指令触发下一轮。ralph-loop hook 同时负责在 experimental.session.compacting 阶段注入续写意图锚点，确保 compaction 不截断未完成的任务链。

create-hooks.ts · chat.message hook · 每条消息进入 LLM 前

要解决的问题

用户输入往往是字面要求，而非真实意图。AI 被字面意思牵着走，容易产生「完成了需求但解决了错问题」的废话式响应——例如用户说「帮我删掉这个错误」实际意图是「修复这个 bug 并确保测试通过」。若 Agent 不先分析意图就直接执行，轻则浪费 token，重则引入破坏性变更。

解决方案

IntentGate（意图门）作为 chat.message hook 的前置卫士：在消息被发送给模型之前，先对其进行意图解析，提取用户真正想要的「目标状态」而非字面操作。解析结果注入到系统提示中，引导后续 LLM 调用以「达成目标」而非「完成指令」为导向。结合 first-message-setup 在首轮固化意图锚点，后续轮次的 context-injection 持续携带这份意图摘要，形成会话级意图一致性保障。

create-managers.ts · backgroundManager + tmuxManager · ultrawork / background 工具触发时

要解决的问题

复杂任务需要同时运行多个专家代理，但 AI 会话是单线程的——顺序执行多个子任务既慢又浪费。此外，某些任务需要真实的交互式终端（REPL、调试器、TUI 工具），而 AI 工具调用只能发 bash 命令，无法保持终端状态和会话上下文。

src/agents/sisyphus.ts · buildDynamicSisyphusPrompt · system prompt 动态编译器

要解决的问题

Sisyphus 作为 OmO 的编排层 Agent，要在每次会话里同时承担：意图分类、并行委派、子会话结果回收、todo 编排、Oracle 咨询、失败恢复、完成验证 7 类决策。如果用静态 system prompt，会出现三类故障：① 用户禁用某个 agent / skill 后，prompt 仍诱导模型去调一个"不存在"的能力，导致工具调用失败；② 不同模型（Claude / GPT-5.4 / Gemini）的弱点不同，一份 prompt 救不了所有模型的"lost-in-the-middle"或"忽略后置规则"问题；③ useTaskSystem 等运行时开关在切换时，prompt 里的"todo / task"用语必须同步切换，否则模型按错误协议输出。

解决方案

buildDynamicSisyphusPrompt(model, agents, tools, skills, categories, useTaskSystem) 是一份会在每次会话起始时动态编译的 system prompt 编译器。它把当前可用的 agents / tools / skills / categories / model / useTaskSystem 求值进多个 build*Section() 子构建器，再用模板字符串拼成最终 prompt。其工程价值不在「措辞优雅」，而在把整套 OmO 多 agent 协议固化成模型必须遵守、且能被 hook 配合验证的协作合同——动态 section 让 prompt 跟着配置自动收缩（禁用 agent 后引导消失），model-specific 分支让弱点被针对性补强（GPT-5.4 整段替换为 buildGpt54SisyphusPrompt，Gemini 在 3 处 replace 注入覆盖层）。

整体联动：System Prompt ↔ Hook ↔ 消息流

Sisyphus 的"自洽行为"并不是单靠 prompt 写规则得来的，而是 prompt（文字约定）+ hook（运行时执行者）+ 消息流（模型实际输入） 三层闭环：prompt 在 system 段约定"看到 X 标签就做 Y"，hook 在运行时把 X 标签真的注入到 user 消息流前面，模型按约定识别标签后做出 Y 行为。下图把三层联动画在一起。

## 关键词快速触发表（${keyTriggers}，按当前可用 agent 动态生成）
  "ultrawork" / "ulw"     → 切换至深度任务模式，启动 Sisyphus 全流程
  "/plan"                 → 路由至 Prometheus 规划阶段
  "/start-work"           → 路由至 Atlas 接管已有 plan
  截图 / 图片 / PDF        → 委派 multimodal-looker

## <intent_verbalization>
对每条用户消息，必须先口头分类一次（不是装样子，下游 Step 1.5 会用）：
  surface form : "<用户原话>"
  true intent  : "<去口语化后的真实意图>"
  routing      : "<路由到哪个 agent / phase>"

## Step 1.5 - Turn-Local Intent Reset
每个 turn 都从"当前消息"重新分类。
禁止把上一轮的 implementation mode 自动延续到本轮。

## Step 2.5 - Context-Completion Gate（进入实现的三个充要条件，必须同时满足）
  ① 当前消息含 implementation 动词（"实现 / 修改 / 重构 / 新增"）
  ② 范围具体可执行（明确文件 / 模块 / 接口边界）
  ③ 没有阻塞性 specialist 结果待返回（特别是 Oracle 还在跑）

# ─── Phase 1 — Codebase Assessment（仅 open-ended 任务）─────
判定仓库类型，决定后续严格度：
  Disciplined  → 严格遵守已有规范
  Transitional → 谨慎，新代码按新规范，老代码保持
  Legacy       → 最小侵入
  Greenfield   → 自由发挥但仍需测试

# ─── Phase 2A — Exploration & Research（最关键的一段）─────

## ${toolSelection}：当前可用工具/委派表
  explore           : 内部代码 grep（必须 run_in_background=true）
  librarian         : 外部 OSS / 文档检索（必须 run_in_background=true）
  oracle            : 架构 / 调试只读咨询

## <tool_usage_rules>（铁律）
  · 并行优先：一次撒 2-5 个，禁止串行
  · explore / librarian 必须 run_in_background=true
  · 同一个搜索不要撒 5 次（${antiDuplication} 提醒）

## 委派模板（prompt-as-spec，给模型抄作业用）
  task(subagent_type="explore", run_in_background=true, prompt="""
    [CONTEXT]   我正在分析 X 的实现
    [GOAL]      找到所有 Y 的调用点
    [DOWNSTREAM] 后续要交给 Hephaestus 做 Z 修改
    [REQUEST]   给我文件:行号 + 周围 5 行上下文
  """)

## Background Result Collection（阻塞协议，违反 = BLOCKING 反模式）
  1. 撒后台任务后立刻返回，不要轮询
  2. 等待 user 消息中出现 <system-reminder> 标签
  3. STOP. END YOUR RESPONSE. 系统会通知完成
  4. 收到 <system-reminder> → 调 background_output(task_id="...") 拿数据
  5. NEVER call background_output before <system-reminder>（这是 BLOCKING 反模式）
  6. Cleanup: 用 background_cancel(taskId="...") 单独清理

模型差异化处理

不同 model 的弱点不同，buildDynamicSisyphusPrompt 之外还有两层覆盖机制：

  • GPT-5.4 整段替换：在 createSisyphusAgent 里检测到 GPT-5.4 系列时，直接调用 buildGpt54SisyphusPrompt 生成完全不同的 prompt（GPT-5.4 对长 prompt 注意力衰减更明显，需要更紧凑的结构与更早的关键约束）。
  • Gemini 三处 replace 注入：Gemini 有 lost-in-the-middle 问题，所以把 delegation / verification override 插在 <Constraints> 之前（高注意力区），而不是 prompt 末尾：
    • 在 </intent_verbalization> 之后追加 buildGeminiIntentGateEnforcement() + buildGeminiToolMandate()
    • 在 </tool_usage_rules> 之后追加 buildGeminiToolGuide() + buildGeminiToolCallExamples()
    • 在 <Constraints> 之前插入 buildGeminiDelegationOverride() + buildGeminiVerificationOverride()

要解决的问题

Sisyphus 是编排者，但实际「执行 / 咨询 / 规划 / 评审」分别落在 6 个 specialist agent 上。每个 agent 都需要一份「职责 / 工具 / 边界 / 输出格式」明确的 system prompt 才能稳定输出——拿到错的 prompt 就会偏离任务.

解决方案

每个 agent 都有独立的 system prompt 编译路径，遵循三条共通设计原则：① 工具白名单——在 prompt 里和 agent 注册时双重声明 allowedTools，禁止越权；② 输出契约——用结构化标签（[Bottom line] / [Action plan] / [Effort]）让 Sisyphus 能正则解析；③ 心智隔离——subagent 拿到的 system prompt 完全不包含 Sisyphus 的 Phase 0~3 那一套，只有「你是谁、做什么、不做什么、按什么格式输出」。下面表 5-17 先做对比，随后给 6 个 agent 的 prompt 中文示例。

6 方案评价

6.1 优势分析

工具密度与 Agent 优先安装

官方文档推荐让 Agent 读 installation.md 自动完成安装，显著降低个人开发者首次集成成本；工具全家桶开箱即用。

Hashline 从协议层解决行偏移

Hashline + hashline-read-enhancer 闭环不依赖模型智慧，而是在传输层做哈希校验，系统性降低陈旧行号失败率。

多模型编排 + 类别路由

Category routing 把模型选择从提示层抽到配置层，Sisyphus 自动委派子代理；团队只需维护类别→模型对照表。

Claude Code 迁移零成本

加载器复用现有 hooks / skills / MCP 配置；存量 Claude Code 用户无需重建资产即可迁移到 OpenCode 生态。

6.2 局限分析

强绑定 OpenCode 插件 API

依赖 @opencode-ai/plugin 等上游包；OpenCode 的 breaking change 需要同步跟进发版，维护窗口随上游演进节奏绑定。

默认遥测 + 多二进制分发

PostHog 遥测默认开启；ast-grep 等可选二进制需平台匹配；企业环境需额外审计数据流和二进制来源合规。

心智模型复杂度高

需同时理解「宿主 opencode.json 配置 + 插件 JSONC + 多代理人设 + Hashline 闭环」四个层次，才算完全掌控；对入门用户门槛较高。

Hashline 需显式开启

hashline_edit 默认并非全局开启，读写语义不对齐时会导致锚点失效；需用户正确配置并理解读增强与写工具的依存关系。
