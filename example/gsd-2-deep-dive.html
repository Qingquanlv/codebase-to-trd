GSD 2 深度解读

基于磁盘状态机的自主编程 Agent — TypeScript + Pi SDK
GSD（Get Shit Done）2 是一个运行在终端里的自主编程 Agent，它的核心创新是把整个开发工作流的状态存在磁盘文件而非内存中，从而实现了崩溃恢复、多终端协同和无限长任务执行。读完本文，你将理解它与其他 AI 编程工具的本质差异、如何在项目中落地使用，以及那些让它"聪明"的关键工程决策。

核心结论

  • 磁盘即状态机 — 所有工作流进度存在 .gsd/ 目录，进程崩溃后从磁盘恢复，而不是重头开始
  • 每单元独立上下文 — 每个子任务（plan / execute / verify）都在全新的 200k token 窗口执行，彻底消除上下文污染
  • 声明式调度取代条件分支 — 用规则表（dispatch table）替换 130 行 if-else，"下一步做什么"变得可测试、可扩展
  • 亚毫秒复杂度分级驱动模型路由 — 纯启发式算法在调用 LLM 前判断任务难度，动态分配 Opus / Sonnet / Haiku，平衡质量与成本
  • Rust N-API 加速关键路径 — grep / glob / AST 解析 / git 操作通过 N-API 调用 Rust，绕过 Node.js I/O 瓶颈

1 摘要

1.1 项目背景

随着 LLM 能力提升，"让 AI 写代码"从聊天补全演进为自主 Agent。但现有工具面临两个根本性问题：一是上下文污染——长任务中历史对话越积越多，模型注意力被稀释，代码质量在几十轮后急剧下滑；二是状态易失——所有进度存在内存中，一次网络中断或进程崩溃就意味着一切重来。GSD 2 针对这两个痛点做了系统性设计。

1.2 项目预期

1.2.1 预期一：长任务持续推进能力

传统对话型 AI 工具的有效窗口通常在几十轮内。GSD 的目标是让 Agent 能稳定推进"新增功能模块"这类需要多天工作量的任务，关键实现是：把一个大需求拆解为 Milestone → Slice → Task 三层结构，每个最小单元（Unit）都在独立的全新 LLM 会话中执行，完成后结果写回磁盘，下个单元读磁盘而非历史对话获取上下文。

1.2.2 预期二：进程级可靠性

GSD 的 auto 模式可能持续运行数小时。通过锁文件（crash-recovery）+ JSONL 取证（session-forensics）+ OS 互斥锁（session-lock）三层保护，即使 Node.js 进程崩溃，下次启动时也能精确定位到崩溃位置并提示用户是否继续。

1.2.3 预期三：成本感知的自适应质量

项目内置了一套无需 LLM 调用的复杂度分级器：在调度每个 Unit 前，通过任务元数据启发式地判断 light / standard / heavy，并结合当前 budget 使用率和历史失败率动态调整分配的模型。目标是在预算耗尽前优先完成更多工作，而不是用最贵的模型处理所有任务。

2 项目设计理念

2.1 磁盘驱动状态机（Disk-Driven State Machine）

GSD 最根本的设计选择：所有工作流状态不存在内存里，只存在磁盘上。.gsd/ 目录下的 Markdown 文件（ROADMAP / SLICE / TASK / SUMMARY）加上 SQLite 数据库就是整个系统的唯一真相来源。

这个选择带来了一系列连锁收益：进程可以随时崩溃然后重启；用户可以同时在多个终端查看状态；Git worktree 隔离每个 Milestone 的开发分支；而且整个状态是人类可读的 Markdown，用户随时可以手动编辑干预。代价是磁盘 I/O 变成热路径，这也是引入 Rust 原生引擎的直接原因。

2.2 每单元新鲜会话（Fresh Session Per Unit）

对话型 AI 工具的通病是"历史越长，质量越差"。GSD 的解法是彻底的：每个 Unit（Task / Slice / Milestone 级别）都在一个全新的 LLM 会话（新的 context window）里执行。上一个 Unit 的输出不是通过对话历史传递，而是写成 SUMMARY 文件，下一个 Unit 的 Prompt 里按需 inline 这个文件的内容。

这种设计的精妙之处在于"按需"两字——auto-prompts.ts 里有一套分层上下文注入机制，根据 inline_level（full / references / compressed）和当前 context budget 决定把多少历史内容塞进 Prompt，而不是无脑全部 inline。

2.3 声明式架构理念（竞品横向对比）

GSD 在多个核心模块都选择了声明式（declarative）而非命令式（imperative）的设计：dispatch table 替代 if-else 链、规则化的 TTSR guardrails、schema 驱动的状态类型。这反映了一个设计判断：可观察性和可测试性比"快速写完"更重要。

与同类工具的横向对比：

2.4 当前适合解决什么问题

✅ 适合的任务类型

  • 中大型功能开发 — 如"新增用户权限系统"，需要设计、实现、测试多个步骤
  • 无人值守批量推进 — 可以挂在后台运行，完成一个 Slice 后自动继续
  • 需要干净 Git 历史的场景 — 每个 Milestone 一个独立分支，squash-merge 保持主分支整洁
  • 跨终端协同 — 磁盘状态可跨终端查看，多人项目可共享 .gsd/ 目录状态
  • 高可靠性要求的自动化场景 — CI/CD headless 模式、定时任务驱动的自动维护

⚠️ 不太适合的任务类型

  • 五分钟内的小修改 — 创建 Milestone 的开销大于任务本身
  • 完全开放式的探索性研究 — GSD 需要明确的目标和可验证的完成标准
  • 纯设计 / 头脑风暴 — 需要人机高频交互的创意过程不适合自主循环
  • 需要即时视觉反馈的前端调试 — 当前版本对 UI 像素级调整的理解有限

3 核心对象与术语

3.1 术语表

统一定义下列词（及表中扩展条目），后文各章在相同语义下使用，避免「同名不同义」。

术语定义
Milestone面向交付的顶层工作包：对应一条独立的产品目标（通常含 ROADMAP、独立 git worktree 与分支），完成后再合并回主线。
Slice在一个 Milestone 之下、按主题或时间切分的工作切片；把大目标拆成可顺序或并行推进的子计划，各自有 SLICE 级状态与 SUMMARY。
Task落在某个 Slice 内的具体实施项（文件/接口级的改动清单）；调度器按 Task 粒度决定下一步要跑哪种 Unit。
Unit最小可调度运行单元：一次 dispatch 决策后，在一个全新 LLM 会话里跑完的一步（如 discuss / plan / execute / verify）；不是「任意一句聊天」。
SUMMARY某个 Unit 结束时由 Agent 写出的压缩记忆（Markdown）；下一 Unit 通过 Prompt 注入引用它，而不是复用整段对话历史。
inline_level在 auto-prompts.ts 中控制历史文档注入详略的配置：full / references / compressed，与 context budget 一起决定 Prompt 多长。
stepModeAuto 会话布尔标记；为 true 时（/gsd step）每跑完一个 Unit 即 pauseAuto，等人再次触发，再进入下一轮 autoLoop。
dispatch声明式调度：由 DispatchRule[] 顺序求值，返回下一步 unitType 或 stop/skip；取代巨型 if-else。
worktreeGit 为单个 Milestone 提供的隔离工作目录 + 分支；Agent 在 worktree 内改代码，完成后 squash-merge 回主分支，减少文件冲突。
Pi SDKGSD 依赖的底层 LLM 与工具运行时（如 @gsd/pi-coding-agent 等闭源包）：负责会话、工具调用与模型路由，GSD 在其之上做工作流与磁盘状态机。

3.2 对象之间的关系

对象定义分属模块
Milestone 锁定「要交付什么」与「何时算完成」；承载 ROADMAP、预算与 worktree 边界，是权限与范围的最大容器。每个 Milestone 对应一个独立的 git worktree 与分支，完成后合并回主线。 state.ts
Slice 在 Milestone 目标之下，按功能主题、风险或并行度把工作切成多个 Slice；每个 Slice 有自己的计划文档（SLICE.md）与执行节奏，便于分阶段验收与回滚。 state.ts
Task 把 Slice 里的计划落成可执行的条目（文件级改动清单）；调度器根据当前磁盘状态选中 Task 后，映射为若干 Unit（不同类型的一次 LLM 会话），每步写回 TASK / SUMMARY 等产物。 state.ts
Unit 最小运行单元：新鲜会话与声明式调度都以 Unit 为边界——每 Unit 一次模型上下文、一次 dispatch 决策、一次可观测的落盘结果；比它更细会破坏「状态可恢复 / 可重试」的粒度。 auto/run-unit.ts · auto-dispatch.ts

Unit — 运行时调度概念（无独立 DB 表）

Unit 不是一行数据库记录，而是 dispatch 每轮决策后交给 run-unit.ts 执行的一次 LLM 会话。它的「形状」来自调度器返回的动作对象：

字段类型含义
unitTypestring本次会话的类型，决定用哪个 Prompt 模板。常见值：discuss-milestone / plan-milestone / plan-slice / execute-task / run-uat / complete-task / complete-slice / verify-milestone
milestoneIdstring本 Unit 操作的 Milestone 编号
sliceIdstring | null本 Unit 操作的 Slice 编号（plan-milestone 等顶层 Unit 无 Slice）
taskIdstring | null本 Unit 操作的 Task 编号（仅 execute-task / complete-task 有）
modelstring | null动态路由或 hook 覆盖后选定的 LLM model ID；null 时由 Pi SDK 取默认
promptstring由 auto-prompts.ts 按 inline_level 构建的完整 Prompt 字符串
matchedRulestring命中的调度规则名称；用于日志追踪与决策可解释性

3.3 状态存储载体

Markdown（.gsd/ 下 ROADMAP / SLICE / TASK 等）：人类可读、可 Git diff、可手工改写的主叙事与进度真相；适合作为跨 Unit 的契约与审查材料。

SQLite（gsd-db 等）：对同一状态做结构化索引与快速查询（关系、时间序、关联 ID）；引擎频繁读盘时走 DB，避免每次都全文扫 Markdown。

runtime 文件（如 .gsd/runtime/auto.lock、session 路径记录）：承载进程级瞬时信号——谁在跑、PID 是否存活、崩溃现场指向哪个 JSONL；不参与产品功能的长期叙事，但决定「能否安全继续跑」。

3.4 常见误解

  • GSD 不是普通聊天式 coding assistant：核心是磁盘状态机 + 调度循环，对话只是其中一个 I/O 形态；长任务靠落盘与 Unit，不靠堆消息。
  • GSD 不是单纯任务清单工具：清单（Task）之下还有自动执行、模型路由、验证门与崩溃恢复；会改代码、会跑工具、会消费预算。
  • SUMMARY 不是完整历史：它只是压缩记忆，用于跨 Unit 传递要点；细节仍在 TASK / SLICE / 代码与 Git 里，需要时应让 Agent 用工具读原文。

4 项目使用方式

4.1 从需求到交付的一次完整过程

以「为现有系统新增用户登录失败次数限制」为例，追踪一次完整的 GSD 执行链路：

① 用户提出目标
用户执行 /gsd discuss "为登录接口添加失败次数限制，5次后锁定15分钟"。GSD 启动 discuss-milestone 类 Unit，与用户澄清边界：计数放在 Redis 还是数据库？锁定是软锁还是硬锁？
② Plan Milestone
discuss 完成后自动进入 plan-milestone Unit。Agent 读取讨论结论，生成包含若干 Slice 的 ROADMAP.md（例如「数据模型变更」「接口逻辑实现」「联调测试」）。状态写入 .gsd/milestones/M01/ROADMAP.md 与 SQLite。
③ Plan Slice（逐个）
对每个 Slice，GSD 分配全新会话执行 plan-slice：Agent 读 ROADMAP、将其 inline 进 Prompt，输出 SLICE.md——文件级任务列表，例如「修改 User 表增加 login_attempt_count 字段」。
④ Execute Tasks（新鲜上下文）
每个 Task 在独立新会话中执行。auto-prompts.ts 按 inline_level 配置决定把多少 SLICE 内容放进 Prompt（full / references / compressed）。复杂度分级器决定本轮用 Sonnet 还是 Opus 执行。代码变更直接落在该 Milestone 专属的 git worktree 上。
⑤ Run UAT / 写入 SUMMARY
全部 Task 完成后，run-uat Unit 跑测试命令并写入结论（PASS / FAIL）。Slice 收尾时生成 SUMMARY.md，作为下一 Slice 的「压缩历史记忆」。
⑥ Worktree Merge + 继续
所有 Slice 通过后，将该 Milestone 的 git worktree squash-merge 回主线；调度器自动推进下一 Milestone，或在信息级别提示全部完成。用户可随时中断；下次 /gsd auto 会依据磁盘状态准确续跑。

同一条链路在 Headless 下逻辑不变，人机交互由 stdin 答案与 stdout JSON 事件替代；在 Step 下每个 Unit 末尾可插入人工审查点。最短命令路径见 4.2，三种运行形态差异见 4.3。

4.2 最小可运行路径

从零开始，用 GSD 完成第一个自主任务只需 5 步：

# 第一步：安装
npm install -g gsd-pi
# 第二步：进入你的项目目录，首次运行会触发 onboarding（配置 LLM API Key）
cd my-project && gsd
# 第三步：创建一个 Milestone（描述你的目标）
/gsd discuss "为用户登录接口添加失败次数限制功能"
# 第四步：启动自主执行
/gsd auto
# 第五步：查看进度（另一个终端或 Web UI）
gsd web # 打开浏览器 Dashboard

5.2.1 🪜 Step 模式（交互逐步）

/gsd step 是 GSD 的交互逐步执行模式。每次运行只推进一个 Unit（计划/执行/验证中的某一步），完成后立即暂停并将结果展示给用户，由用户确认后再手动触发下一步。适合首次接触新 Milestone、需要审查 AI 决策质量，或处于敏感代码区域不想放任自主执行的场景。

与其他模式的核心区别：Step 模式与 Auto 模式共享同一套 autoLoop 执行引擎，关键差异仅在启动时传入 options.step = true。这会使 bootstrapAutoSession 将 s.stepMode 置为 true，autoLoop 在每次完成一个 Unit 后调用 pauseAuto 将 s.active = false，循环自然退出挂起。用户审查输出后再次执行 /gsd step 才重新激活 —— 人始终在回路中，每步可审查、可回退，但代价是不能无人值守。

5.2.3 🔧 Headless 模式（CI/CD）

gsd --headless auto 是 GSD 的 CI/CD 专用模式。在无 TTY 环境（如 GitHub Actions、定时任务脚本、Docker 容器）中运行时，所有终端 UI 交互被完全禁用：输入来自 stdin JSON 预置答案，输出为结构化 JSON 事件流，进程以标准退出码结束（0=成功，1=失败）。整个过程无需人工介入，机器可直接解析执行结果。

与其他模式的核心区别：Headless 模式的入口完全不同 —— 通过 CLI 的 --headless 标志而非 /gsd 命令触发，在进入 autoLoop 之前需要完成两件准备工作：① loadHeadlessAnswers 从 stdin 读取 JSON 预置答案映射，用于拦截所有交互确认；② setupHeadlessEvents 将所有 TUI 事件注册为 JSON 序列化输出到 stdout。一旦进入 autoLoop，执行逻辑与 Auto 模式完全相同，区别仅在于 I/O 层：没有终端 UI，没有颜色输出，只有机器可读的 JSON 事件流和最终退出码。这使 CI/CD 系统可以无摩擦地消费 GSD 的执行结果。

5.3 关键模块实现细节

5.3.1 关键模块一：声明式调度管道（auto-dispatch.ts）

📁 src/resources/extensions/gsd/auto-dispatch.ts 🏷️ 规则表模式（Chain of Responsibility 变体）

要解决的问题：GSD 的工作流有 10+ 种 Unit 类型（discuss / plan / execute / uat / complete 等），决定"下一步该执行哪个"需要考虑当前 Milestone 状态、Slice 状态、是否有崩溃残留、预算是否超限……用命令式 if-else 实现会产生一个数百行的怪兽函数，无法单独测试每条规则，添加新规则会引入隐患。

解决方案：把所有调度逻辑提取为一张有序的 DispatchRule[] 数组，每条规则有 name 和 async match 函数。match 返回 DispatchAction（dispatch / stop / skip）或 null（不匹配，继续下一条）。调度引擎只需顺序评估规则表，第一条返回非 null 的规则获胜。

5.3.2 关键模块二：复杂度分级器（complexity-classifier.ts）

📁 src/resources/extensions/gsd/complexity-classifier.ts 🏷️ 启发式分级 + 自适应学习

要解决的问题：模型路由需要知道"这个 Task 有多难"，但用 LLM 来判断难度有明显的自我矛盾（贵的 LLM 才能判准难度，用了就没省到）。需要一个零 LLM 调用的快速分级器。

解决方案：两阶段判断。第一阶段按 unit 类型给默认分级（plan-milestone = heavy, complete-slice = light）；第二阶段对 execute-task 类型做元数据分析：读取 TASK.md 的文件引用数、依赖数、代码块数量、复杂度关键词，并结合历史失败率做自适应上调；最后用当前 budget 使用率做一次强制降级（90%+ 时 heavy → light）。

5.3.3 关键模块三：崩溃恢复机制（crash-recovery.ts）

📁 src/resources/extensions/gsd/crash-recovery.ts + session-forensics.ts 🏷️ 心跳锁文件 + JSONL 取证

要解决的问题：auto 模式的单个 Unit 执行可能持续数分钟到数十分钟。网络中断、内存溢出、用户强制关闭都会导致进程非正常退出。如何在下次启动时让用户无缝感知到"上次崩溃了"并提供恢复选项？

解决方案：三层嵌套保护。第一层：每次 Unit 开始前在 .gsd/runtime/auto.lock 写入含 PID + sessionFile 路径的 JSON 锁文件，正常退出时删除；下次启动检测到锁文件且 PID 进程已不存在 → 判定崩溃。第二层：从崩溃时的 Pi session JSONL 文件读取已完成的工具调用，重建崩溃前状态摘要（session-forensics）。第三层：OS 级 flock 互斥锁防止并发执行。

5.3.4 关键模块四：分层上下文注入（auto-prompts.ts）

📁 src/resources/extensions/gsd/auto-prompts.ts 🏷️ 分层预算分配 + 优雅降级

要解决的问题：每个 Unit 在独立新会话执行，但它需要了解当前任务的上下文——所在 Milestone 的目标、所属 Slice 的计划、关联 Task 的详情。把所有历史文档全部 inline 会超出 context window；什么都不 inline 则 Agent 缺乏背景会走弯路。

解决方案：三档 inline_level 配置（full / references / compressed）+ 30k 字符 preamble 上限的自动裁剪。full 模式把完整 Markdown 文件 inline；references 模式只 inline 文件路径（告诉 Agent "去读这个文件"）；compressed 模式只 inline 文件的关键摘要段落。超出 30k 上限时自动按章节边界截断，而不是硬截字符。

6 方案评价

6.1 优势分析

  • 磁盘状态机是真正的差异化：目前市场上很少有工具能做到跨会话、跨终端、跨崩溃的持久状态，这是 GSD 在"长任务可靠执行"场景的根本护城河
  • 声明式调度的工程严谨性：把调度逻辑变成可测试的规则表，每条规则独立，可以 unit test，这在同类工具中罕见
  • 成本感知是一等公民：复杂度分级 + 预算压力降级的组合，让 GSD 在有限预算下可以完成更多工作，不是简单地"永远用最贵的"
  • Rust 原生引擎的务实选择：没有为了"纯 TS" 而坚持，在性能关键路径引入 Rust 是正确的工程判断
  • 扩展架构良好：24 个扩展相互独立，主引擎不依赖具体扩展，添加新能力无需修改核心代码

6.2 局限分析

  • 依赖闭源 Pi SDK：核心 Agent 运行时（会话管理、工具执行、LLM 路由）封装在 @gsd/pi-coding-agent 里，这是一个闭源包，限制了社区贡献和问题排查深度
  • 初始学习曲线陡峭：Milestone / Slice / Task 三层结构对于"只想快速执行一个命令"的用户来说过重；onboarding 文档对新用户不够友好
  • 启发式分级的天花板：复杂度分级器无法感知代码库的全局状态，"修改一个被 50 个地方引用的核心接口"和"新增一个独立工具函数"可能被分到同一级别
  • SUMMARY 质量的链式风险：每个 Slice 的 SUMMARY 由 Agent 自己生成并作为下个 Slice 的上下文，一次质量差的 SUMMARY 会影响后续所有 Slice 的执行质量
  • Git worktree 的操作复杂性：每个 Milestone 一个 worktree 对不熟悉 git worktree 的用户来说是心智负担，worktree 冲突处理不够自动化
