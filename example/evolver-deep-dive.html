🧬 Evolver 深度解读

EvoMap 出品 · GEP 协议驱动的 AI Agent 自进化引擎 · Node.js + Git + 可审计资产
Evolver 是一个基于 GEP（Genome Evolution Protocol） 的 AI Agent 自进化引擎，把每次 prompt 调整变成带信号匹配、变更估算、回滚兜底的可审计进化操作。它扫描运行日志提取信号，按信号匹配选择 Gene（进化规则），生成受协议约束的 GEP 提示词，通过 CLI（单次或 Daemon 循环）驱动 Agent 完成下一步进化。

核心结论

  • 「进化」是有结构的，不是自由发挥 — 每次改动由 Gene 定义约束（max_files / forbidden_paths / validation 命令），Selector 基于信号匹配选 Gene，Solidify 负责验证 + git commit + 记录 EvolutionEvent，整个过程可审计可回滚
  • 核心模块是混淆的黑盒 — src/evolve.js（711KB 单行混淆）是整个引擎的大脑，源码不可读；这是 EvoMap 保护知识产权的主动决策，也是理解这个项目的最大障碍
  • Proxy Mailbox 是网络隔离的关键设计 — Agent 不直接调用 EvoMap Hub API，而是通过本地 localhost HTTP Proxy 读写 JSONL 邮箱；Proxy 负责认证、重试、同步，Agent 只做本地 IPC
  • 自适应调度是 Daemon 模式的核心价值 — 指数退避 + OMLS 空闲窗口调度 + 饱和检测（evolution_saturation 信号），让 Daemon 在系统空闲时做蒸馏/探索，在饱和时降频，避免无意义循环
  • Distillation 是知识沉淀的机制 — 每 5 次 solidify 或失败模式积累后，触发 skillDistiller 将重复成功/失败模式蒸馏成新的 Gene，形成知识复利

1 摘要

1.1 项目背景

AI Agent 的 prompt 工程通常以「改一改、试一试」的方式进行，没有标准化的变更格式、没有回滚机制、没有可复用的模式库。同一类错误被反复手动修复，成功的修复经验无法沉淀传播。EvoMap 团队在 2026 年初发布 Evolver，将生物进化的隐喻引入 AI Agent 的持续改进流程中：用 Gene（基因）封装进化规则，用 Capsule（胶囊）记录成功案例，用 EvolutionEvent（进化事件）建立审计日志，形成「信号驱动 → 协议约束 → 可审计落地」的完整闭环。

技术栈：Node.js 18+，100% JavaScript，仅依赖 dotenv（生产依赖只有这一个），开发依赖 javascript-obfuscator（用于混淆核心模块）。强依赖 Git（用于回滚、变更范围计算和 solidify）。

1.2 能解决的问题

1.2.1 「prompt 调整是黑盒操作，无迹可查」——GEP 协议强制可审计进化

每次修改 prompt 都是一次随机尝试，没有记录"为什么改"、"改了什么"、"结果怎样"。Evolver 的 GEP 协议强制每次进化必须绑定一个 Gene（定义改动意图和约束），记录 EvolutionEvent（追踪信号、意图、结果、blast radius），形成 append-only 的 events.jsonl 审计日志。

1.2.2 「同一类错误反复手动修复，没有知识沉淀」——Gene 库 + Distillation 蒸馏

Agent 在相似错误上重复失败，每次都从零处理。Evolver 的 Selector 基于日志信号（error / exception / failed 等）匹配现有 Gene，优先复用成功模式；每 5 次 solidify 后，Distillation 机制自动分析历史事件，将成功/失败模式蒸馏成新的 Gene，形成可复用的知识资产。

1.2.3 「Agent 进化失控，改坏了难以回滚」——Blast Radius 估算 + Git 回滚

Agent 有时改动范围过大，触碰到不该动的文件，结果难以恢复。Evolver 的每次进化前计算 Blast Radius（预计影响文件数和行数），Gene 约束中有 max_files 和 forbidden_paths 字段；验证失败时通过 EVOLVER_ROLLBACK_MODE=hard/stash/none 控制 git 回滚策略，失败不留痕迹。

1.2.4 「Agent 节点孤立，进化经验无法共享」——EvoMap Hub + Proxy Mailbox 网络

每个 Agent 各自积累经验，团队或社区之间的成功 Gene 无法流通。Evolver 通过 Proxy Mailbox 架构与 EvoMap Hub 通信，支持资产发布（asset/submit）、技能下载（node index.js fetch --skill）、任务领取（Worker Pool），让进化资产在网络中流通复用。

2 项目设计理念

2.1 核心理念

  1. 协议约束 > 自由发挥：进化不是「让 LLM 随便改」，而是严格按照 Gene 定义的策略、约束和验证步骤执行。Gene 是进化行为的合同，Selector 是合同匹配器，不允许在没有 Gene 的情况下自由编辑。
  2. 信号驱动选择：每次进化的起点是从 memory/ 日志中提取的信号（error / perf_bottleneck / user_feature_request 等），信号决定选择哪个 Gene，Gene 决定采用哪种策略。主观直觉被替换为客观信号。
  3. 最小可逆变更：Gene 策略中反复强调「smallest reversible patch」和 blast radius 估算；solidify 在验证失败时自动回滚；这与「先行完成、后行验证」的 AI 常见模式形成对比。
  4. 隔离网络通信：Agent 通过 localhost Proxy 读写邮箱，绝不直接接触 Hub API 或外部认证；Proxy 处理重试、认证、序列化。这在安全性和可测试性上都有明显优势。

3.4 常见误解

  1. 误解：Evolver 会自动修改你的代码 — 实际上 Evolver 是一个「提示词生成器」，它输出一份 GEP 协议约束的提示词到 stdout，不会直接编辑任何源文件。只有在宿主运行时（如 OpenClaw）解释 stdout 的 sessions_spawn(...) 指令，或者人工使用 node index.js solidify 命令时，才会真正发生代码变更。
  2. 误解：MIT 许可证意味着所有代码可读 — README 声明 MIT，但 package.json 显示 GPL-3.0-or-later，同时 src/evolve.js（711KB 单行混淆）是不可读的黑盒。README 底部小字说明「Core evolution engine modules are distributed in obfuscated form to protect intellectual property」。这是一个商业利益与开源精神之间明显的张力点。
  3. 误解：--loop 模式会让 Agent 不停地改代码 — Daemon 循环有多重节流机制：pending solidify 时挂起等待、adaptive sleep 指数退避、OMLS 空闲调度降频、evolution_saturation 信号触发 4x sleep。在没有新信号或处于饱和状态时，Daemon 基本处于休眠状态。
  4. 误解：Proxy 是网络代理服务器 — Proxy 是 本地 localhost HTTP 服务，是 Agent 与 EvoMap Hub 之间的本地 IPC 层，不是透明的网络代理。它负责将 Agent 的邮箱操作（send/poll/ack）异步同步到 Hub，Agent 绝不直接访问 Hub API。
  5. 误解：Gene 的 validation 命令可以执行任意 Shell 命令 — Solidify 有严格的命令白名单（仅 node/npm/npx 前缀），禁止命令替换（`` ` `` 和 `$(…)`）、shell 操作符（;/&/|/>/<），且有 180 秒超时。外部 Gene 在 promote 之前还要过一遍同样的安全检查。

4 项目使用方式

4.1 从需求到交付的一次完整过程

以「Agent 持续触发 exec 工具错误，需要修复并沉淀为可复用 Gene」为例，走一遍完整的 Evolver 进化链路。

① 错误日志写入 memory/
宿主运行时（如 OpenClaw）将 Agent 运行日志写入 memory/ 目录。日志包含错误片段如 error: unknown command 'process' 和工具结果 {"status":"error"}，这些将成为后续信号提取的输入。
② evolve.run() 提取信号
执行 node index.js，src/evolve.js 扫描 memory/ 提取信号：log_error、errsig:TOOLRESULT error、windows_shell_incompatible 等。这些信号被传入 Selector。
③ Selector 匹配 Gene
Selector 将信号与 assets/gep/genes.json 中所有 Gene 的 signals_match 对比打分，选出最匹配的 gene_gep_repair_from_errors（error / exception / failed / unstable 信号），输出 JSON 选择决策。
④ 生成 GEP 提示词并输出到 stdout
将 Gene 定义、信号、选择决策、历史 Capsule 组装成 GEP 协议提示词，打印到 stdout。使用 --review 时暂停等待人工确认；宿主运行时解释 sessions_spawn(...) 触发 Agent 执行该提示词。
⑤ Agent 执行修复，执行 solidify 验证
Agent 按 Gene 的 strategy 完成修复后，执行 node index.js solidify：先运行 Gene 的 validation 命令（node scripts/validate-modules.js + node scripts/validate-suite.js），通过则 git commit，失败则 git rollback（hard 模式）。
⑥ 记录 EvolutionEvent + 生成 Capsule
Solidify 成功后，将本次进化结果追加到 events.jsonl，并写入新 Capsule 到 capsules.json（含环境指纹、blast_radius、confidence）。每 5 次 solidify 触发 Distillation，自动蒸馏新 Gene。

4.2 最小可运行路径

最短路径：clone → install → 准备 memory 日志 → 运行 → 查看 GEP 提示词。

graph LR A(["git clone + npm install"]) --> B["准备 memory/ 日志\n（运行日志 / 错误信号）"] B --> C["node index.js\n提取信号 + 生成 GEP 提示词"] C --> D["--review 审查\n或宿主执行"] D --> E["node index.js solidify\n验证 + git commit + 记录 Event"] E --> F["Distillation 蒸馏\n新 Gene 沉淀"]
最小路径：安装 → 准备信号 → 进化 → 固化 → 蒸馏
# 1. 安装
git clone https://github.com/EvoMap/evolver.git && cd evolver && npm install

# 2. 可选：配置 EvoMap Hub（离线模式下不需要）
echo "A2A_HUB_URL=https://evomap.ai\nA2A_NODE_ID=your_node_id" > .env

# 3. 单次进化（需要 memory/ 目录有日志文件）
node index.js

# 4. 审查模式（人工确认后再执行）
node index.js --review

# 5. 固化（验证 + git commit + 记录事件）
node index.js solidify --intent=repair --summary="Fix exec command error"

# 6. Daemon 模式（持续循环）
EVOLVE_STRATEGY=repair-only node index.js --loop

# 7. 生命周期管理
node src/ops/lifecycle.js start    # 后台启动
node src/ops/lifecycle.js status   # 查看状态

4.3 核心使用模式

🔍 单次进化模式

node index.js 单次扫描 memory → 选 Gene → 输出 GEP 提示词。适合手动触发、审查后再执行的工作流。

👁 审查模式

node index.js --review 生成 GEP 提示词后暂停，展示 git diff 等待人工确认，再决定是否 solidify。适合生产环境谨慎操作。

⚙️ Daemon 循环模式

node index.js --loop 持续运行，自适应调度：空闲时做蒸馏/探索，有信号时进化，饱和时降频。配合 pm2 或 systemd 保活。

🌐 Worker Pool 模式

WORKER_ENABLED=1 参与 EvoMap 网络领取进化任务，需同时在 evomap.ai Dashboard 开启 Worker 开关，本地 + Hub 双端均打开才能收到任务。

5.3 关键模块实现细节

5.3.1 Daemon 主循环

📄 index.js 🏷 控制层 ⚡ node index.js --loop 触发

要解决的问题：持续运行的进化守护进程需要防止多实例并发（两个 Daemon 同时修改同一代码库会造成 git 冲突）、防止无意义的高频空转（没有信号时反复调用浪费资源）、以及在内存泄漏时优雅自重启（长期运行 Node.js 进程的常见问题）。

解决方案：三层防护：① PID Singleton Guard（evolver.pid 文件 + process.kill(pid, 0) 探活）防止多实例；② 自适应睡眠（指数退避 + 随机抖动 + OMLS 空闲乘数 + 饱和乘数 4x）防止空转；③ Suicide 机制（EVOLVER_MAX_CYCLES_PER_PROCESS=100 或 RSS > 500MB 时 spawn 子进程后退出）防止内存累积。

sequenceDiagram participant I as index.js(loop) participant PID as evolver.pid participant E as evolve.run() participant OMLS as idleScheduler participant Self as 自重启 I->>PID: acquireLock() 写 PID 文件 PID-->>I: 成功（或陈旧锁接管） loop 每个周期 I->>I: 检查 isPendingSolidify() alt 有 pending solidify I->>I: sleep(pendingSleepMs) else 无 pending I->>E: evolve.run() E-->>I: ok / error I->>OMLS: getScheduleRecommendation() OMLS-->>I: sleep_multiplier + should_distill I->>I: sleep(adaptive + jitter + saturation × omls) end I->>I: 检查 cycleCount / RSS alt 超过阈值 I->>Self: spawn 新进程 + process.exit(0) end end

5.3.2 GEP Selector：信号匹配选基因

📄 src/gep/selector.js 🏷 进化核心 ⚡ evolve.run() 内部调用

要解决的问题：从多个 Gene 中选出最适合当前问题的一个，既要考虑信号匹配度，又要考虑历史成功记录（Capsule），还要受 EVOLVE_STRATEGY 策略偏好的影响（如 repair-only 模式下不应选 innovate Gene）。

解决方案：Selector 对每个 Gene 计算综合分数：基础分 = 信号重叠比例；加权分 = 匹配 Capsule 的历史成功率（confidence × success_streak）；策略惩罚 = repair-only 策略对 innovate Gene 扣分。选分最高的 Gene 并输出 JSON 选择决策（含 gene_id + rationale），这份决策被嵌入最终的 GEP 提示词中，供 LLM 参考。

5.3.3 Solidify 固化机制

📄 src/gep/solidify.js 🏷 固化层 ⚡ node index.js solidify 或 review 后触发

要解决的问题：AI Agent 修改代码后需要一个安全的"提交"机制：验证改动是否合法（运行单测）、防止不安全的 Shell 命令趁机执行、失败时能完整回滚到修改前的状态，并且整个过程留下可审计的记录。

解决方案：Solidify 实现了一个三阶段原子化流程：① 执行 Gene.validation 数组中的验证命令（通过 isValidationCommandAllowed 白名单过滤后才执行）；② 验证全部通过则 git commit；③ 写入 EvolutionEvent + Capsule。失败时根据 EVOLVER_ROLLBACK_MODE 执行 git reset --hard HEAD（hard）或 git stash（stash）或不回滚（none）。

5.3.4 Proxy Mailbox 代理网关

📄 src/gep/mailboxTransport.js 🏷 网络通信层 ⚡ EVOMAP_PROXY=1 且 --loop 时启动

要解决的问题：AI Agent 直接调用外部 Hub API 存在认证密钥管理复杂、网络抖动导致进化中断、请求失败难以重试等问题。同时 Hub 需要向 Agent 推送任务和事件，直接轮询 HTTP 成本高。

解决方案：在 Agent 和 Hub 之间插入一个本地 HTTP Proxy（localhost:19820），Agent 只通过邮箱 API（POST /mailbox/send、POST /mailbox/poll、POST /mailbox/ack）与 Proxy 通信；Proxy 在后台异步同步邮箱到 Hub，处理认证、重试、序列化。地址通过 ~/.evolver/settings.json 发现，兼容 Proxy 进程重启后地址变化。

6 方案评价

6.1 优势分析

进化过程完全可审计

events.jsonl append-only 审计日志 + Capsule 成功记录 + Git 历史构成三层可追溯性。每次改动有信号驱动、Gene 约束、blast radius 记录，远超随手改 prompt 的可审计性。

Git 原生回滚安全网

验证失败时通过 git rollback 彻底还原，不会留下部分修改的中间状态。这是大多数 AI 改代码工具不具备的强安全保障，特别适合生产代码仓库。

极轻依赖，部署简单

生产依赖只有 dotenv 一个包，npm install + node index.js 即可运行。对比需要 Python 虚拟环境、Docker 容器的同类工具，部署摩擦极低，离线模式完整可用。

知识蒸馏形成复利

Distillation 机制（每 5 次 solidify 自动触发）将重复成功/失败模式提炼为新 Gene，知识不断积累。EvoMap 网络使蒸馏出的 Gene 可在社区流通，形成超越单节点的进化复利。

6.2 局限分析

核心模块混淆是根本性透明度障碍

src/evolve.js（711KB 单行混淆）是整个引擎的大脑，信号提取、GEP 提示词生成的具体逻辑对开发者完全不可见。调试困难、无法定制、与「开源」精神形成明显矛盾。

冷启动需要大量日志积累

Selector 的信号匹配和 Capsule 历史加权在 memory/ 为空时几乎无意义；初始 Gene 库只有 3 个，覆盖场景有限。项目价值需要一段时间的日志积累才能充分体现。

许可证定义不一致

README 说 MIT，package.json 说 GPL-3.0-or-later，核心模块混淆分发。这个三重矛盾对需要在企业项目中使用的用户造成法律不确定性，GPL 具有强传染性。

验证命令白名单限制场景

只允许 node/npm/npx 前缀的验证命令，Python 项目（pytest）、Go 项目（go test）、需要 Make 的项目都无法使用 Gene 内置验证，严重限制了在异构技术栈中的适用性。
