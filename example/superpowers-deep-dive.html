Superpowers 深度解读

让 AI Coding Agent 真正懂得软件工程纪律的行为规范框架
Superpowers 是一套装载在 AI Coding Agent 上的强制行为规范框架，通过精心设计的"技能（Skill）"把混乱的 AI 编程行为规范化为有纪律的软件工程流程。它的核心创新在于：技能即行为程序——用结构化 Markdown 精确控制 Agent 在每个关键节点的决策，强制暂停、强制思考、强制规范操作，而不是让 Agent 盲目生成代码。目前在 GitHub 拥有 150k+ Star。读完本文，你将理解它与其他 AI 规范工具的本质差异、如何在项目中落地，以及那些让它真正"有纪律"的关键设计。

核心结论

  • Skills即强制行为程序 — SKILL.md 不是提示词，而是带 Iron Law（铁律）的行为规范；铁律违反则必须重来，不接受例外
  • 1% 规则强制调用 — 只要有 1% 可能性适用某技能，Agent 就必须调用；消除"这个太简单了"的自我欺骗
  • Human Partner 不是 User — 全部技能统一用"your human partner"，强调协作而非命令执行；Agent 是有纪律的协作者
  • 零依赖纯 Markdown — 没有任何运行时依赖，技能就是 Markdown 文件，原生跨平台（Claude / Cursor / Codex / Gemini CLI）
  • 双阶段代码评审 — Spec Compliance（规格合规）→ Code Quality（代码质量），两层质量门缺一不可

1 摘要

1.1 项目背景

随着 AI Coding Agent 能力提升，一个根本性问题浮出水面：AI 代码质量不稳定——它可以很快生成代码，但往往绕过设计思考、跳过测试、在没有充分理解需求时直接动手，最终产出一堆"能跑但难维护"的代码。

Superpowers 的作者 Adam Wride 观察到：AI Agent 最强大的形态，是像资深工程师一样遵循严格、协作的工程流程——先澄清需求，再设计，再分解，再用测试保护每一步实现，最后规范地处理分支。技能（Skill）是实现这个目标的机制：用结构化文档向 Agent 注入"下一步应该做什么、不该做什么、违反了就重来"的行为规范。

1.2 能解决的问题

1.2.1 「AI 自作主张、跳过关键步骤」——铁律强制审批点

AI 助手在没有约束时会直接生成代码、跳过需求澄清和测试，等到问题暴露已是返工阶段。Superpowers 的 Iron Law 机制把「头脑风暴」「测试先行」「人工审批」等节点变成不可绕过的硬约束——不是建议，是执行路径上的物理屏障，确保每个关键决策点都有人类参与。

1.2.2 「每次会话都要重新告诉 AI 怎么工作」——可复用的结构化工作流

团队规范、代码质量标准、发布流程往往只活在 Wiki 或口耳相传中，AI 每次上下文都是空白。Superpowers 将这些流程封装为技能文件（SKILL.md），一次编写、随处 invoke，AI 在任意会话中都能加载同一套完整工作流，消除「靠记忆传递规范」的摩擦。

1.2.3 「AI 完成实现就宣告完工，验证环节缺失」——内置验证前置门

没有强制验证机制时，AI 常在实现后直接提交，测试、lint、类型检查被跳过。Superpowers 的 verification-before-completion 技能要求在任何「完成声明」之前必须运行验证命令并展示真实输出，把「证据先于断言」作为流程级要求而非个人习惯。

1.2.4 「团队无法统一 AI 协作模式，各自为政」——零成本落地与共享扩展

引入新工具往往需要大量配置与培训，阻力大。Superpowers 的接入成本是复制一个配置文件，通过 writing-skills 元技能可按标准格式创建团队专属技能并共享，让组织级 AI 协作规范像代码一样版本化、可审查、可演进。

2 项目定位与理念

2.1 设计哲学

Superpowers 的设计哲学可以归纳为三条原则：

  1. 纪律高于速度：宁可多花五分钟澄清需求，也不要在错误方向上快速前进。Brainstorming 技能的铁律就是：未获设计审批前不得写任何实现代码。
  2. 强制优于建议：技能中的 Iron Law 不是"建议遵守"，而是"违反就重来"。这是 Superpowers 与普通 Cursor Rules 或 system prompt 的根本区别——后者容易被 Agent 在"效率"名义下悄悄绕过。
  3. 协作而非执行：全部技能统一使用"your human partner"（你的人类伙伴）而非"user"（用户），这不是语言游戏，而是明确的设计意图——Agent 的角色是有专业判断力的协作者，而非被动接受指令的执行工具。

2.2 与同类方案横向对比

方案核心定位落地成本落地效果适合谁
Superpowers 跨平台强制行为规范框架 极低：复制一个配置文件（CLAUDE.md / AGENTS.md），零工具安装，零依赖 覆盖完整开发生命周期（头脑风暴 → TDD → 实现 → 验证 → 发布），跨 Claude / Cursor / Gemini 行为一致；效果上限受模型遵从度影响 追求工程纪律的开发者；需要跨平台统一 AI 行为的团队
GitHub Spec Kit SDD 脚手架 + 多 Agent 集成 + 扩展市场 低：uv tool install specify-cli + specify init，一次性生成配置文件；团队需学习 slash 命令体系 规格→计划→任务→实现链路完整，多工具集成开箱即用，Manifest 管理可安全升级；效果聚焦功能开发阶段，不覆盖调试、发布等环节 需在多种 AI 工具间落地 SDD 流程的团队
OpenSpec 工件依赖图驱动的规格迭代系统 中：安装 npm 包 + 学习 OPSX schema 格式 + 定义工件依赖图；初期配置成本较高 工件依赖自动拓扑排序，变更与规格分目录可追踪；工作流可定制但需主动维护 schema；生态尚早期，社区资源有限 有明确工件依赖关系、需可重复迭代规格的项目
GSD 2 磁盘状态机驱动的自主 Agent 高：依赖 Pi SDK、需配置状态机与任务格式，初始学习曲线陡峭 真正的无人值守执行、崩溃可恢复、支持无限长任务；人类介入节点少，适合确定性强的长周期任务，不适合需要频繁决策的创意工作 需长期自主推进复杂功能、可接受低干预的独立开发者
Cursor Rules (.cursorrules) 单一 IDE 的 AI 行为配置文件 极低：一个文本文件，无需安装任何工具，语法无学习成本 约束范围限于代码风格与简单行为偏好，无阶段化工作流、无组合复用；换工具时规则无法迁移 Cursor 重度用户，只需基础代码风格约束

Superpowers 的核心赢面：落地成本与 Cursor Rules 相当（复制一个文件），但覆盖深度远超其他方案——从创意到发布的全链路行为规范，且跨平台可复用。输面：不提供任务调度或状态持久化，效果依赖 Agent 模型对技能内容的理解与遵从。

2.3 Superpowers 是什么（一句话）

一套由 15 个结构化 Markdown 技能文件组成的行为规范库，通过平台配置文件（CLAUDE.md / AGENTS.md）注入 AI Coding Agent，强制 Agent 在创意工作、实现、调试、评审等关键节点遵循有纪律的软件工程流程。

2.4 当前适合解决什么问题

✅ 适合的场景

  • 中型功能开发 — 需要设计→实现→测试全流程，防止 AI 跳步骤
  • 多人/跨平台协作 — 团队统一 AI 行为规范，换平台行为不变
  • 测试纪律要求高的项目 — TDD 铁律确保每行生产代码前有失败测试
  • 需要设计评审的新功能 — Brainstorming 强制先出设计文档
  • 代码质量要求高 — 双阶段评审过滤低质量实现
  • 需要 Git 工作区隔离的功能开发 — using-git-worktrees 提供干净隔离

⚠️ 不太适合的场景

  • 五分钟内的小修改 — 触发完整设计流程的开销大于任务本身
  • 纯探索性代码试验 — 规范框架与随意探索存在摩擦
  • 无法理解长指令的弱模型 — SKILL.md 内容较长，弱模型可能遗漏关键铁律
  • 需要持久化状态跨会话恢复的任务 — Superpowers 无运行时状态

3 核心对象与术语

3.1 术语表

统一定义下列词汇；后文各章在相同语义下使用。

术语定义所在文件
Skill 一个 SKILL.md 文件，包含触发条件、铁律（Iron Law）和行为程序。每个技能负责开发生命周期中一个特定阶段的行为规范。 skills/**/SKILL.md
Iron Law 技能中不可违反的规则。与普通行为建议不同，Iron Law 是绝对约束——违反则必须重来，不接受任何"只这一次"的例外。 各 SKILL.md
Human Partner 项目统一用语，指与 Agent 合作的人类。刻意避免使用"User"——强调双方是协作伙伴，Agent 是有判断力的协作者而非执行工具。 using-superpowers/SKILL.md
1% Rule 如果某个技能有 1% 的可能适用于当前情况，Agent 就必须调用它。这是元技能（using-superpowers）的核心原则，消除"这个太简单了"的自我欺骗。 using-superpowers/SKILL.md
Design Doc Brainstorming 技能的输出物，是一份结构化设计文档，保存到 docs/superpowers/specs/YYYY-MM-DD-{name}-design.md。必须经 Human Partner 审批后才能开始实现。 docs/superpowers/specs/
Plan writing-plans 技能的输出物，是一份详细的实施计划，保存到 docs/superpowers/plans/YYYY-MM-DD-{name}.md。每步估计 2-5 分钟，含复选框和期望命令输出。 docs/superpowers/plans/
Subagent 在 subagent-driven-development 模式中，每个任务被派发给一个独立 context 的执行单元。每个 Subagent 完成自己的任务后经过双阶段评审，避免上下文污染。 subagent-driven-development/SKILL.md
Two-Stage Review 双阶段评审：第一阶段 Spec Compliance（检查实现是否符合设计文档）→ 第二阶段 Code Quality（检查代码风格、测试覆盖、边界条件）。两关均通过才算完成。 subagent-driven-development/SKILL.md
Red-Green-Refactor TDD 技能的核心循环：Red（写失败测试）→ Green（最少代码让测试通过）→ Refactor（在绿灯下优化）。这个循环是 Iron Law——跳过任何一步都违反铁律。 test-driven-development/SKILL.md
Harness Superpowers 中称 Claude Code、Cursor、Codex、OpenCode、GitHub Copilot CLI、Gemini CLI 等 AI Agent 运行平台为"Harness"（执行框架）。技能文件对所有 Harness 均适用。 CLAUDE.md / AGENTS.md
YAGNI "You Aren't Gonna Need It"。Brainstorming 技能的重要原则：只设计和实现需求中明确提出的功能，不要预测性添加"以后可能需要"的功能，避免过度工程化。 brainstorming/SKILL.md

3.2 对象之间的关系

Superpowers 中的核心对象可以分为两类：规范文件（Skill、Iron Law——定义 Agent 行为）和工件文件（Design Doc、Plan——开发过程的产出）。它们之间存在明确的生产-消费依赖链。

对象定义分属模块
Skill 单个行为规范单元，每个 SKILL.md 文件对应一个技能，包含触发描述、Iron Law 和步骤说明 skills/**/SKILL.md
Bootstrap Config 平台配置文件（CLAUDE.md / AGENTS.md），在 Agent 启动时把 using-superpowers 注入上下文，是整个框架的引导层 CLAUDE.md / AGENTS.md
Design Doc Brainstorming 输出的设计文档，是工件层的起点；writing-plans 消费它来生成实施计划 docs/superpowers/specs/
Plan writing-plans 输出的实施计划，是工件层的第二层；subagent-driven-development 消费它来分发任务给各 Subagent docs/superpowers/plans/
Subagent 执行层中独立 context 的任务单元。每个 Subagent 拥有 implementer / spec-reviewer / code-quality-reviewer 三个角色 subagent-driven-development/SKILL.md
Worktree Git 为功能分支提供的隔离工作目录。using-git-worktrees 技能管理 worktree 的创建与回收，防止主分支污染 skills/using-git-worktrees/

Design Doc — Brainstorming 输出的设计文档格式

节类型内容
stated_problemMarkdown 段落问题的完整陈述，来自与 Human Partner 的澄清对话
requirements有序列表明确的功能与非功能需求，YAGNI 原则剔除范围外内容
alternatives_considered对比表格至少两个架构方案及各自的权衡分析
chosen_approachMarkdown 段落选定方案及其选择理由
key_components文件列表受影响的文件与组件，含新建/修改说明
open_questions列表仍需 Human Partner 确认的遗留问题

Plan — writing-plans 输出的实施计划格式

节类型内容
goal单句本次实施的核心目标（一句话描述）
architecture2-3 句选定的技术方案概述，与 Design Doc chosen_approach 对应
tech_stack列表涉及的技术、库及版本约束
tasks[n].files文件列表该任务要创建/修改/测试的具体文件
tasks[n].steps复选框列表每步 2-5 分钟粒度，含期望命令与输出
tasks[n].success验收标准可验证的完成条件，用于 subagent 评审

3.3 技能状态存储

Superpowers 与 GSD 2 等工具的一个根本区别在于：它没有运行时数据库或持久化状态。整个框架的状态分布在三个层次：

存储层载体负责什么持久性
代码状态 Git（commits, branches, worktrees） 所有代码改动的源头；worktree 隔离功能分支；commit 是工作完成的证据 永久
设计/计划状态 Markdown 文件（docs/superpowers/specs/，docs/superpowers/plans/） Design Doc 记录为什么这样设计；Plan 记录具体执行步骤与完成状态（复选框） 永久（与代码同步 commit）
会话内任务状态 TodoWrite 工具（Agent 内存） 当前对话中的任务进度跟踪；仅在活跃会话中有效，关闭会话即消失 临时（会话级）

这种设计的含义是：Superpowers 本身不管理状态，它只管理行为。状态由 Git 和 Markdown 文件承载，恢复工作只需回到正确的 worktree 和读取上次的 Plan 文件，而不需要任何专有的状态恢复命令。

3.4 常见误解

误解实际情况
"技能只是建议性的系统提示词，不用就不用" 错。Iron Law 是绝对约束，违反则必须重来。Brainstorming 的铁律"未获 Human Partner 审批前不得写实现代码"与普通建议有本质区别——设计为让 Agent 主动报告违规，而非悄悄绕过。
"1% 规则是在描述精确概率" 错。1% 是最低调用门槛，语义是"只要有可能适用就必须调用"，强调无需充分理由就应触发——消除"这个太简单了，不需要走设计流程"的合理化借口。
"用了 TDD 就更慢，不适合快速迭代" 错。TDD 铁律（先写失败测试再写实现）短期加了写测试的时间，但规避了后期大面积调试和回归修复的更大时间成本。Superpowers 的 TDD 技能正是为此设计。
"Subagent 模式比单会话更慢" 通常相反。每个 Subagent 独立 context，避免了长对话中注意力稀释导致的错误和返工。双阶段评审提前拦截问题，总体比反复 Debug 快。
"Human Partner 和 User 只是用词不同" 错。这是精心的语言设计。"User"暗示被动消费者，"Human Partner"明确要求 Agent 以协作者姿态主动参与决策（如澄清问题、提出替代方案），而不仅仅是执行指令。
"Superpowers 会帮我管理任务队列和跨会话进度" 不会。Superpowers 是行为规范层，无运行时状态管理。跨会话进度恢复依赖 Plan 文件中的复选框状态（需人工或 Agent 读取）。如需自动状态机，应结合 GSD 2 等工具。

4 项目使用方式

4.1 从需求到交付的一次完整过程

以"为现有 Web 应用新增邮件验证功能"为例，走一遍 Superpowers 标准开发链路。

① 用户提出需求
用户描述："add email verification to the sign-up flow"。using-superpowers/SKILL.md 中的 1% 规则触发：Agent 识别这是一项创意/功能实现任务，必须先走 Brainstorming 技能，不得直接开始写代码。
② Brainstorming 澄清设计
Agent 加载 brainstorming/SKILL.md，主动向 Human Partner 提出 5 个澄清问题：Token 有效期多长？是否需要重发按钮？过期后是锁定账号还是允许重发？验证邮件的发送服务使用什么？……用户确认后，Agent 生成设计文档并保存到 docs/superpowers/specs/2026-04-14-email-verification-design.md。Iron Law：未经 Human Partner 书面审批设计文档，不得写任何实现代码。
③ 创建隔离工作区
Agent 加载 using-git-worktrees/SKILL.md，执行 git worktree add ../myapp-email-verification feature/email-verification，在隔离目录中创建新分支。主分支 main 在整个功能开发期间保持干净，所有改动在 worktree 中进行。
④ 拆解实施计划
Agent 加载 writing-plans/SKILL.md，基于审批后的设计文档，将工作拆解为 7 个独立任务（每个 2-5 分钟粒度），保存到 docs/superpowers/plans/2026-04-14-email-verification.md。每个任务含文件清单、步骤复选框和验收标准。
⑤ Subagent 逐任务执行（含 TDD）
Agent 加载 subagent-driven-development/SKILL.md，为 Plan 中每个任务派发独立 Subagent。每个 Subagent 遵循 Red-Green-Refactor 循环：先写失败测试（Red），再写最少实现（Green），再重构（Refactor）。完成后经 spec-reviewer 和 code-quality-reviewer 双阶段评审，均通过才标记为完成。Iron Law：所有测试必须通过，才能报告任务完成。
⑥ 收尾处置
Agent 加载 finishing-a-development-branch/SKILL.md，验证全部测试通过后，向 Human Partner 呈现四个选项：① Merge to main（合并到主分支）；② Open PR（开 Pull Request 等待评审）；③ Stash for later（暂存继续迭代）；④ Abandon（放弃，清理 worktree）。由 Human Partner 做最终决策。

4.3 核心使用模式

根据任务规模，Superpowers 有三种典型使用模式——框架本身不区分模式，差异在于触发的技能子集：

🔵 完整工程模式

适用：新功能开发、需要设计评审的重要改动

链路：using-superpowers → brainstorming → using-git-worktrees → writing-plans → subagent-driven-development（含 TDD）→ verification-before-completion → finishing-a-development-branch

特征：Iron Law 全部激活；Design Doc 须审批后才能写代码；每任务双阶段评审

🟢 快速修复模式

适用：Bug 修复、小范围代码改动（1% 规则判断无需完整设计流程）

链路：using-superpowers → systematic-debugging（如有 Bug）→ test-driven-development → verification-before-completion

特征：跳过 brainstorming 和 writing-plans；TDD 铁律仍强制执行；完成前验证不可省略

🟣 评审协作模式

适用：接手他人代码、处理 PR 评审意见

链路：using-superpowers → requesting-code-review / receiving-code-review

特征：评审请求前测试必须全绿；接收评审时要求技术性分析而非盲目接受意见

技能触发时机铁律核心

5.3 核心技能模块

5.3.1 关键模块一：using-superpowers（元技能）

📄 skills/using-superpowers/SKILL.md 🏷 元技能 · 每次对话必须调用

要解决的问题：AI Agent 在处理"看起来很简单"的任务时，会主动绕过设计评审、测试规范等流程——因为这些步骤让任务"更慢完成"。普通 system prompt 无法解决这个问题，因为 Agent 可以在任何时候以"效率"为由忽略建议性内容。需要一个让"跳步骤本身变成违规行为"的机制。

解决方案：元技能文件作为框架的"总调度器"，强制在每次对话开始时执行技能扫描。核心是 1% 规则（Iron Law）：只要有 1% 的可能性某个技能适用，就必须调用——最低门槛是可能性而非确定性。这彻底封闭了 Agent 的"合理化借口"逃生口。

5.3.2 关键模块二：brainstorming + writing-plans（设计层）

📄 skills/brainstorming/SKILL.md + skills/writing-plans/SKILL.md 🏷 设计阶段 · 所有功能开发前必须执行

要解决的问题：AI 的默认行为是"立即写代码"——接到需求就开始实现，跳过需求澄清和设计权衡。这导致两个常见失败模式：①构建了技术上正确但解决了错误问题的代码（理解偏差）；②添加了需求里没有提到的"便利功能"（YAGNI 违反）。这两种失败都要到代码完成后才被发现，返工成本极高。

解决方案：两阶段强制规范。brainstorming 技能通过铁律把"先得到设计审批才能写代码"变成不可绕过的约束；writing-plans 技能把通过审批的 Design Doc 拆解为 2-5 分钟粒度的可执行步骤，消除"任务太模糊导致 Subagent 乱猜"的问题。

5.3.3 关键模块三：subagent-driven-development（执行层）

📄 skills/subagent-driven-development/SKILL.md 🏷 执行层核心 · 独立上下文 + 双阶段质量门

要解决的问题：在单一长会话中执行多个任务存在两个问题：①注意力稀释——对话历史越长，模型对早期指令（包括 Design Doc 规格）的注意力越弱，后期实现容易偏离原始设计；②评审盲点——实现者对自己代码的审查存在认知偏差，倾向于忽视微妙的设计合规问题。

解决方案：两层隔离机制。第一层：每个任务派发给一个全新 context 的 Subagent（Implementer），只看当前任务的 Plan 和相关文件，不受长会话历史干扰。第二层：双阶段评审由两个独立 Subagent（Spec Reviewer 和 Code Quality Reviewer）执行，与 Implementer 完全隔离，消除自我审查盲点。

代价与局限：每个任务需要 3 次独立 LLM 调用（Implementer + 两个 Reviewer），时间和 Token 成本约是单次执行的 3 倍。评审质量本身也依赖模型能力——弱模型的 Reviewer 可能遗漏微妙的规格偏差。此外，subagent 模式目前主要面向串行任务；真正的并行执行（dispatching-parallel-agents 技能）需要任务间无依赖，适用场景相对有限。

6 工程设计评价

6.1 优势

✅ 零依赖跨平台

整个框架由纯 Markdown 文件组成，无任何运行时依赖、无需安装、无版本冲突。在 Claude Code、Cursor、Codex、Gemini CLI 上行为一致，迁移成本为零。

✅ 铁律机制不可绕过

Iron Law 不是建议，而是绝对约束。设计文档未审批不得写代码、测试未通过不得声称完成——这些规则有效防止了 AI Agent 为追求速度而走捷径的系统性风险。

✅ 覆盖完整生命周期

从需求澄清（brainstorming）到计划拆解（writing-plans），到执行（subagent-driven-development），到调试（systematic-debugging），到评审（requesting-code-review），到收尾（finishing-a-development-branch），链条完整无断点。

✅ 设计优先防止返工

Brainstorming 铁律强制先出设计文档再动手，从根源减少"构建了正确代码但解决了错误问题"的返工成本，这是 AI 编程中最常见的浪费之一。

✅ 独立 Context 减少幻觉

Subagent 模式为每个任务开辟独立上下文，避免长会话中历史噪音对后续任务的干扰，是对抗 AI "注意力稀释"问题的结构性解决方案。

✅ 社区扩展生态

writing-skills 元技能允许任何人按标准格式创建自定义技能。技能是可版本化的 Markdown 文件，可以通过 Git 共享，形成社区驱动的行为规范生态。

6.2 局限

⚠️ 无运行时强制执行

铁律的遵守依赖 AI 模型的理解和合规意愿——没有代码层面的强制机制。弱模型可能遗漏关键铁律，或在长上下文中忘记之前读过的规范内容。框架无法验证 Agent 是否真的遵守了规则。

⚠️ 无跨会话状态持久化

Superpowers 是纯行为层，没有运行时数据库。TodoWrite 的任务状态在会话结束后消失，大型任务的进度恢复依赖 Plan 文件中的复选框（需手动或 Agent 读取）。无法像 GSD 2 那样在崩溃后自动恢复到精确位置。

⚠️ 学习曲线与记忆成本

15 个技能需要开发者熟悉各自的触发时机和铁律内容。在实际工作中，"这个情况该用哪个技能"的判断需要一定的实践积累，新用户初期容易触发或漏触发。

⚠️ 全流程增加了摩擦

对于简单的 5 分钟改动，触发 brainstorming → writing-plans 的完整流程会让轻量任务感觉"过于正式"。1% 规则的设计初衷是防止漏触发，但对有经验的用户可能造成不必要的流程摩擦。
