Hermes Agent 深度解读

NousResearch 出品 · 自我进化 AI Agent · Python + SQLite + 16 平台网关
Hermes Agent 是 NousResearch 开源的自我进化 AI Agent：通过技能学习循环在每次复杂任务后自动创建和改进技能文件，配合 SQLite FTS5 跨会话记忆，让 Agent 越用越"懂你"。支持 16 个消息平台网关（Telegram / Discord / WeChat 等）接入，任意 OpenAI 兼容模型均可使用，提供 CLI、消息平台和编辑器插件三种入口。

核心结论

  • 自我进化是一等公民 — Skills 系统不是附加功能，而是 Agent 的核心差异点：每次任务完成后，Agent 被主动推送（nudge）去创建或改进技能文件，形成可复用的程序化记忆
  • 单一外部记忆提供者约束 — MemoryManager 强制"内置 + 最多一个外部提供者"，防止工具 schema 膨胀和记忆后端冲突，这是工程稳健性的主动权衡
  • 并发工具执行有精确安全分析 — 不是简单并发所有工具，而是维护三张集合（永不并发 / 读安全 / 路径作用域），路径重叠时自动降级为顺序执行
  • run_agent.py 是真正的单体核心 — 10,800 行，包含对话循环、错误恢复、流式输出、代理模式、IterationBudget，模块化提取在 agent/ 子包中持续进行
  • 网关用 contextvars 隔离会话状态 — 多平台消息同时到达时，每个会话独立的 contextvars 防止状态污染，是从 os.environ 到 contextvars 的一次关键重构

1 摘要

1.1 项目背景

大语言模型 Agent 普遍面临「会话即遗忘」的困境：每次对话重新开始，没有跨会话记忆，需要用户反复解释背景；同时 Agent 无法从成功的任务中学习可复用的工作方式。Nous Research 在 OpenClaw 项目基础上构建了 Hermes Agent，核心命题是：让 Agent 像人类一样积累经验、掌握技能、记住你的偏好。

技术栈：Python 3.11+、OpenAI 兼容 API（支持 OpenRouter / Anthropic / xAI / Qwen / MiMo / GLM 等 200+ 模型）、prompt_toolkit（TUI）、SQLite WAL + FTS5（状态与搜索）、Pydantic v2（数据校验）、16 个消息平台 SDK（Telegram Bot / discord.py / slack-bolt 等）。

1.2 能解决的问题

1.2.1 「Agent 无法积累经验，每次任务从零开始」——技能自创建循环

传统 AI 助手没有跨会话的程序化记忆：你教会它一个工作流，下次对话又得重新解释。Hermes 在每次完成复杂任务后通过 nudge 机制主动推送 Agent 创建技能文件（SKILL.md），并在后续使用中持续改进；任何人都可以把自己的技能发布到 Skills Hub，形成可组合的共享生态。

1.2.2 「Agent 只能在一台设备的一个 App 上使用」——16 平台统一网关

大多数 AI Agent 绑定特定界面（VSCode 插件、桌面 App），无法在 Telegram 发个消息让它在服务器上跑任务。Hermes 的 Gateway 进程同时监听 Telegram、Discord、WhatsApp、WeChat、Slack、Signal、iMessage、钉钉、飞书等 16 个平台，Agent 在 $5 VPS 上持续运行，你在手机上随时交互。

1.2.3 「AI 对话超过上下文窗口后记忆清零」——FTS5 跨会话召回 + 上下文压缩

上下文窗口满了之后，早期的对话内容会被丢弃，重要信息丢失。Hermes 用 SQLite FTS5 为所有历史会话消息建立全文索引，通过 session_search 工具在任意时刻召回相关历史；同时 ContextCompressor 在压缩前从记忆提供者提取关键信息注入摘要，避免"重要的事情被压没"。

1.2.4 「复杂任务一个 Agent 做太慢，无法并行」——子 Agent 委派与并发工具执行

单 Agent 线性执行大型任务效率低。Hermes 支持通过 delegate_task 工具派生隔离子 Agent 并行处理子任务，每个子 Agent 拥有独立的 IterationBudget；在单次工具调用批次内，Hermes 也会对读安全工具并发执行（最多 8 个线程），路径不重叠的文件操作同样可以并发。

2 项目设计理念

2.1 核心理念

  1. 学习即架构：技能系统不是插件，是 Agent 的核心记忆层。技能在使用中被改进（self-improve during use），而不只是静态文档。
  2. 平台无关，工具统一：所有 16 个消息平台共享同一个 _HERMES_CORE_TOOLS 工具集，send_message 通过 check_fn 门控（网关运行时才可用），平台差异被封装在适配器层，核心 Agent 无感知。
  3. 安全边界显式化：并发工具执行有三张显式集合（永不并发 / 读安全 / 路径作用域）；MemoryManager 限制最多一个外部提供者；危险命令通过 _DESTRUCTIVE_PATTERNS 正则启动审批流程。安全不靠约定，靠约束。
  4. RL 数据生产内置：trajectory_compressor.py 和 batch_runner.py 把日常 Agent 使用产生的轨迹转换为强化学习训练数据，Tinker-Atropos 子模块直接跑 RL 实验，模型自我改进的完整回路在同一个仓库里。

2.2 与同类方案横向对比

工具核心定位落地成本落地效果适合谁
Hermes Agent 自我进化多平台 AI Agent 低：curl | bash 安装，hermes setup 向导配置 跨会话记忆 + 技能积累 + 16 平台网关 + 子 Agent 并发；一次安装，随时随地用 需要持久化运行、多平台接入、长期积累经验的个人/小团队
Claude Code / Cursor Agent 编辑器内联 AI 编码助手 极低：编辑器插件，开箱即用 编码体验顶级，深度 IDE 集成；但绑定单一编辑器，无跨会话记忆，无多平台接入 专注编码场景的开发者
GSD 2 磁盘状态机驱动的自主 Agent 高：需配置 Pi SDK、状态机格式 崩溃可恢复、无限长任务、无人值守；无跨会话技能学习，人类干预节点少 需要长期无人值守执行的独立开发者
AutoGPT / CrewAI 多 Agent 编排框架 高：需要写 Python 代码定义 Agent 角色和任务流 自动化程度高，可编排复杂多 Agent 流程；上手成本高，无内置技能学习 需要复杂多 Agent 流水线的工程团队

Hermes 的核心赢面：自我进化（技能积累）+ 多平台无缝接入 + 强化学习数据生产内置，三者兼具。输面：run_agent.py 单文件 10,800 行带来较高的代码理解门槛；技能质量依赖模型遵从度。

2.3 Hermes 是什么（一句话）

一个以技能自创建 + SQLite 跨会话记忆为核心差异、支持 16 个消息平台网关、任意 OpenAI 兼容模型的自我进化 AI Agent，由 Nous Research 开源并持续演进。

2.4 当前适合解决什么问题

✓ 适合

  • 需要跨会话记忆的个人助理（记住偏好、积累知识）
  • 在多个消息平台切换但想用同一个 Agent 的场景
  • 需要在云服务器持续运行、手机随时触发的自动化任务
  • 希望 Agent 从工作中学习并改进工作流程
  • 需要并发执行多个子任务（研究 + 写代码 + 搜索同时跑）
  • RL 训练数据生产与模型微调研究

✗ 不适合

  • 仅需要单次问答、无需持久化的场景（用 ChatGPT 即可）
  • 高安全隔离要求的生产环境（终端工具默认有审批但门槛较低）
  • 对响应延迟要求极低的实时交互场景
  • 需要复杂多 Agent 拓扑的企业级编排（用 CrewAI 更合适）
  • Windows 原生环境（仅支持 WSL2）

3.3 状态存储载体

存储层持久性职责写入者
SQLite WAL (state.db)持久 ★ 真相来源会话元数据、完整消息历史、FTS5 搜索索引；WAL 模式支持并发读AIAgent / Gateway

3.4 常见误解

  1. 误解：技能（Skills）只是提示词模板 — 实际上 Skills 是 Agent 在任务中自主创建的程序化记忆，包含工作流程、踩坑记录、项目特定约定，并在后续任务中被自动加载和改进，是跨会话的「经验资产」而非静态配置。
  2. 误解：可以同时接入多个外部记忆提供者 — MemoryManager 强制限制最多一个外部提供者。第二个注册请求会被静默拒绝并打印警告。这是为了防止工具 schema 膨胀（多个 memory 工具让模型困惑）和后端状态冲突，是明确的设计决策。
  3. 误解：并发工具执行就是把所有工具批量并发 — 实际上有精确的安全分析：clarify 这类交互工具永不并发；web_search、read_file 等只读工具可并发；write_file、patch 等文件工具在目标路径不重叠时才并发，重叠时自动回退到顺序执行。
  4. 误解：Gateway 运行时 Agent 的状态通过全局变量共享 — Gateway v0.9 重构后使用 Python contextvars 为每个会话维护独立的运行时上下文，彻底替代了之前用 os.environ 传递会话状态的方式，防止多平台并发时的状态污染。
  5. 误解：Hermes 只能用 OpenAI 模型 — AIAgent 使用 OpenAI 兼容 SDK，支持 OpenRouter（200+ 模型）、Nous Portal、Anthropic、xAI、Qwen、MiMo、GLM、Mistral 等，通过 hermes model 随时切换，无需改代码。

4 项目使用方式

4.1 从需求到交付的一次完整过程

以「在 Telegram 上让 Hermes 每天早上 8 点发送新闻摘要」为例，走一遍从安装到自动化运行的完整链路。

① 安装并初始化
执行 curl -fsSL https://raw.githubusercontent.com/NousResearch/hermes-agent/main/scripts/install.sh | bash，安装后运行 hermes setup 向导，选择 LLM 提供者（如 OpenRouter）并配置 API Key。
② 配置 Telegram 网关
运行 hermes gateway setup 输入 Telegram Bot Token，配置完成后 hermes gateway start 启动网关进程（建议用 systemd 保活）。此后可在 Telegram 对话框与 Agent 交互。
③ 测试基础对话
在 Telegram 发消息 "你好，查一下今天 AI 领域的最新新闻"，验证 Agent 能正确调用 web_search 工具并返回结果。跨会话记忆在此会话中已开始积累。
④ 创建定时任务
在 Telegram 发消息 "帮我创建一个每天早上 8 点的定时任务：搜索今日 AI 新闻，总结成 3 条发给我"，Agent 调用 cronjob 工具创建 cron job，写入 cron 配置并通过 croniter 调度。
⑤ Agent 自动创建技能
完成任务后，Hermes 的 nudge 机制会提示 Agent 将「新闻摘要工作流」封装为 Skill 文件。下次要求类似任务时，Agent 会直接加载该技能执行，无需重新学习。
⑥ 查看和管理
运行 hermes 进入 CLI，输入 /insights --days 7 查看过去 7 天的用量统计；/skills 浏览积累的技能库；/compress 在上下文满之前主动压缩会话历史。

4.2 最小可运行路径

最短路径：安装 → 配置模型 → 开始对话。

graph LR A(["curl 安装脚本"]) --> B["hermes setup\n配置 LLM 提供者"] B --> C["hermes\nCLI 交互"] C --> D["hermes gateway start\n可选：多平台接入"] D --> E["cron / skills\n积累经验"]
最小路径：安装 → 配置 → 对话 → 扩展
# 1. 安装（Linux / macOS / WSL2 / Android Termux）
curl -fsSL https://raw.githubusercontent.com/NousResearch/hermes-agent/main/scripts/install.sh | bash
source ~/.bashrc

# 2. 运行设置向导（配置 LLM 提供者、API Key）
hermes setup

# 3. 开始对话
hermes

# 4. 可选：启动 Telegram 网关
hermes gateway setup   # 配置 Bot Token
hermes gateway start   # 启动网关

# 5. 诊断问题
hermes doctor

4.3 核心使用模式

💻 CLI 交互模式

运行 hermes 进入全功能 TUI，支持多行编辑、slash 命令自动补全、流式工具输出、Ctrl+C 中断重定向。适合开发调试和详细任务。

📱 消息平台模式

通过 hermes gateway start 让 Agent 在云服务器持续运行，从 Telegram / Discord / WeChat 等随时触发任务。Agent 在你睡觉时执行定时任务。

🔌 编辑器集成模式

通过 ACP 协议（Agent Client Protocol）接入 VS Code、Zed、JetBrains。使用 hermes-acp 工具集（不含 clarify / send_message 等交互工具），专注编码场景。

🔬 批量/RL 训练模式

用 batch_runner.py 批量生成 Agent 轨迹，trajectory_compressor.py 压缩为训练格式，配合 Tinker-Atropos 跑 RL 实验，产出更好的 tool-calling 模型。

5.3 关键模块实现细节

5.3.1 Agent 对话主循环

📄 run_agent.py 🏷 对话引擎 ⚡ 每次用户消息触发

要解决的问题：Agent 需要在一次用户请求中执行多轮工具调用（LLM → 工具 → LLM → 工具 → …），直到 LLM 返回最终文本响应。同时需要防止无限循环（消耗无限 token），并在上下文即将溢出时优雅压缩而非崩溃。

解决方案：AIAgent.run_conversation() 维护一个 while 循环，由 IterationBudget 线程安全地计数（父 Agent 默认 90 次）。每次迭代调用 LLM 并检查响应类型：若有 tool_calls 则执行工具并追加结果继续循环；若是纯文本则退出。每轮结束后检查 token 使用量，接近上限时触发 ContextCompressor。

sequenceDiagram participant L as run_conversation participant B as IterationBudget participant M as 模型 API participant T as 工具执行器 participant C as ContextCompressor L->>B: consume() 检查余量 B-->>L: True（允许继续） L->>M: chat.completions() M-->>L: 响应（tool_calls 或文本） alt 有 tool_calls L->>T: 并发执行工具 T-->>L: 工具结果 L->>L: 追加 tool 消息继续循环 else 纯文本响应 L->>L: 退出循环，返回结果 end L->>C: 检查 token 压力 C-->>L: 压缩后的消息列表（如需）

5.3.2 MemoryManager 记忆编排

📄 agent/memory_manager.py 🏷 记忆系统 ⚡ 每轮对话 prefetch / sync

要解决的问题：Agent 需要跨会话记忆，但不同团队有不同的记忆后端偏好（内置文件 / Honcho 对话管理 / Hindsight 插件）。若允许多个外部提供者并存，工具 schema 会因多个 memory 工具而膨胀，模型会困惑；同时多个后端并发写入同一记忆可能产生冲突。

解决方案：MemoryManager 强制「内置提供者 + 最多一个外部提供者」约束。内置提供者（MEMORY.md / USER.md）始终注册且不可移除；第二个外部提供者注册时被静默拒绝并打印警告。所有提供者共享统一的生命周期接口（prefetch / sync_turn / on_pre_compress / shutdown），任意提供者失败不阻塞其他提供者。

5.3.3 并发工具执行引擎

📄 run_agent.py 🏷 执行引擎 ⚡ LLM 返回 tool_calls 批次时触发

要解决的问题：LLM 一次可能返回多个工具调用（如同时搜索三个网站），串行执行浪费时间；但盲目并发会导致：交互类工具（clarify）同时弹出多个问题、文件写入工具覆盖彼此的结果、危险命令未经审批执行。

解决方案：维护三张工具集合精确管控并发。_NEVER_PARALLEL_TOOLS（clarify 等）任何时候都串行；_PARALLEL_SAFE_TOOLS（web_search / read_file / vision_analyze 等只读工具）直接并发；_PATH_SCOPED_TOOLS（read_file / write_file / patch）通过路径归一化检测重叠，重叠则串行，不重叠则并发。通过 ThreadPoolExecutor 最多 8 线程执行。

5.3.4 上下文压缩器

📄 agent/context_compressor.py 🏷 记忆管理 ⚡ token 使用量接近上限时触发

要解决的问题：长对话的 token 使用量会超过模型上下文窗口，但简单截断早期消息会丢失关键决策背景；同时每次都做完整摘要成本过高，需要根据上下文压力决定是否压缩以及压缩范围。

解决方案：ContextCompressor 维护一个压缩策略：读取当前 token 估算（来自 model_metadata），与模型上下文长度的配置阈值（默认 80%）对比；超过时调用 LLM 对早期消息生成摘要，同时调用 MemoryManager.on_pre_compress() 从记忆提供者提取关键信息注入摘要提示，防止重要信息在压缩中丢失；压缩后生成新的 parent_session_id 链，保留从该点开始的完整消息历史。

sequenceDiagram participant A as AIAgent participant CC as ContextCompressor participant MM as MemoryManager participant LLM as 压缩用 LLM participant DB as SQLite A->>CC: check_and_compress(messages, token_count) CC->>CC: 估算 token 压力（与阈值对比） alt 超过阈值 CC->>MM: on_pre_compress(messages) MM-->>CC: 关键记忆提示 CC->>LLM: 生成会话摘要（注入记忆提示） LLM-->>CC: 摘要文本 CC->>DB: 新建 Session（parent_session_id 链） CC-->>A: 压缩后的 messages（摘要 + 近期消息） else 压力未超阈值 CC-->>A: 原始 messages（无变化） end

6 方案评价

6.1 优势分析

自我进化学习闭环

技能从任务中自动创建和改进，配合 SQLite FTS5 跨会话记忆召回，Agent 的能力随使用时间持续提升；Skills Hub 社区共享让个人积累的工作流可以被他人复用，网络效应显著。

16 平台统一接入，无缝切换

同一个 Agent 实例通过 Gateway 进程覆盖 Telegram、Discord、WeChat、iMessage、DingTalk 等 16 个平台，contextvars 保证会话隔离；Agent 运行在你的服务器上，24/7 响应，不绑定任何设备。

并发工具执行的精确安全分析

三张工具集合（永不并发 / 读安全 / 路径作用域）精确控制并发边界，读密集型任务（多网站搜索 + 多文件读取）可显著提速，同时危险命令和交互工具有明确的串行保障。

RL 训练数据生产内置

trajectory_compressor.py + batch_runner.py + tinker-atropos 形成完整的 RL 数据生产管线，日常 Agent 使用产生的轨迹可直接用于微调更好的 tool-calling 模型，研究与实用两条线并进。

6.2 局限分析

run_agent.py 单文件 10,800 行

核心逻辑高度集中，理解和贡献门槛较高。虽然 agent/ 子包在持续提取模块，但对话循环的核心路径仍在大文件中，新贡献者需要相当时间才能定位具体功能。

技能质量依赖模型遵从度

技能自创建是 nudge 机制（推送给 Agent），而非强制执行；技能内容的质量（完整性、准确性）完全取决于 LLM 的遵从能力。弱模型可能创建无用甚至有害的技能文件。

安全审批机制尚需完善

危险命令检测基于正则启发式，复杂的 shell 命令链可能绕过；虽然 v0.9.0 做了大量安全加固（路径遍历、SSRF、Twilio webhook 验证等），但面向公开 Bot 部署时安全边界仍需仔细配置。

Windows 原生不支持，Android 尚早

Windows 必须用 WSL2；Android/Termux 路径在 v0.9.0 才正式落地，且部分依赖（faster-whisper、ctranslate2）在 Android 上不兼容，需要使用受限的 termux extra 而非完整的 all extra。
