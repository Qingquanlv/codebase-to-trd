OpenSpec 深度解读

AI 时代的规格驱动开发框架 — 让人类与 AI 在写代码之前先对齐
OpenSpec 是一个为 AI 编程助手设计的规格驱动开发（SDD）框架，通过在代码实现前建立轻量的 Artifact 层（proposal → specs → design → tasks），解决了"AI 凭着聊天记录写代码"的根本性不可预期问题。它在 39.5k Star 的高人气背后，有一套极为精妙的架构设计：DAG 工作流引擎 + YAML 可定制 Schema + 30 种工具适配器 + 文件系统即状态。
核心创新 将 AI 工作流建模为带依赖的有向无环图（DAG），通过文件存在性检测状态
最值得学习 3 层 Schema 解析机制 + 30+ 工具适配器的扩展性设计
关键取舍 轻量文件系统状态 vs. 无法支持并发/云端工作流
适用人群 使用 AI 编程助手开发中大型功能的个人开发者和团队

1 摘要

1.1 项目背景

2025 年 AI 编程助手全面爆发，Cursor、Claude Code、GitHub Copilot、Windsurf 等工具让开发者可以用自然语言驱动代码生成。但这带来了一个新问题：AI 太听话了。用户一句"帮我加个黑暗模式"，AI 就开始大量修改代码——但双方根本没有对齐"黑暗模式"的边界、影响范围和实现方式。

OpenSpec 诞生于 2025 年 8 月，由 Fission-AI 团队开发，核心观点是：AI 编码失控的根本原因不是模型不够聪明，而是没有规格约束。传统软件工程的 PRD/设计文档虽然解决了这个问题，但对 AI 时代的节奏来说太重了。OpenSpec 选择了一条中间路线：轻量 Markdown 规格 + 可执行的工作流自动化。

1.2 项目预期

OpenSpec 给用户带来的核心价值是：

  • 对齐优先：每个功能变更在写任何代码前，先产出 proposal.md（为什么要做）→ spec.md（做什么）→ design.md（怎么做）→ tasks.md（分几步做）
  • AI 指令结构化：通过 /opsx:propose、/opsx:apply、/opsx:archive 等斜线命令，让 AI 在结构化的工作流中行动而非随意发挥
  • 增量规格管理：变更不重写全量规格，通过 Delta Spec（ADDED/MODIFIED/REMOVED/RENAMED）增量叠加，归档时自动合并
  • 全工具通用：支持 25+ AI 编程工具，一次 openspec init 自动适配用户选择的工具

2 设计理念

2.1 Artifact 驱动工作流（核心创新）

OpenSpec 最关键的设计决策是：把一次功能变更建模为一组有依赖关系的 Artifact。每个 Artifact（proposal、specs、design、tasks）是一个 Markdown 文件，它们之间的依赖关系构成一个有向无环图（DAG）。

这个设计的价值在于：AI 永远知道自己在整个开发流程中的哪一步，接下来应该生成什么，需要先读哪些上下文。这把"AI 按需生成文档"变成了"AI 按图执行有序工作流"。

2.2 Schema-as-YAML（工作流可编程）

工作流本身被定义在 YAML 文件中（默认是 schemas/spec-driven/schema.yaml），而不是硬编码在程序里。这意味着用户可以创建自己的工作流 Schema：比如"只需要 proposal + tasks"的极简模式，或"加入 RFC 评审步骤"的企业模式。

Schema 支持三层解析：项目本地（最高优先级）→ 用户全局（~/.local/share/openspec/）→ 内置包（最低）。这套机制让 OpenSpec 既有开箱即用的默认行为，又具备完全可定制的扩展能力。

2.3 工具无关适配器（生态宽度优先）

OpenSpec 没有选择绑定单一 AI 工具，而是为每种工具实现了一个适配器（Adapter），将工具无关的斜线命令内容（CommandContent）转换为工具特定的文件格式。目前支持 30+ 适配器，覆盖 Cursor、Claude、Copilot、Windsurf、Cline、Codex 等几乎所有主流工具。

这套适配器架构的代价是：每次 AI 工具更新其命令格式，OpenSpec 就需要更新对应适配器。但换来的是极宽的用户覆盖面——这正是 39.5k Star 背后的核心原因之一。

2.4 文件系统即状态（零依赖追踪）

OpenSpec 有一个非常简洁的状态管理原则：一个 Artifact 是否完成，通过其对应文件是否存在来判断。不需要数据库，不需要锁文件，一切状态都在 openspec/changes/<name>/ 目录结构中。

这让 OpenSpec 的可靠性极高（没有状态同步问题），但也限制了它——无法支持分布式/云端工作流，也不支持同一变更的并发操作。对于个人开发者和小团队场景，这是完全合理的取舍。

2.5 竞品横向对比

工具定位优势劣势

3 使用方式与执行链路

3.1 适合解决什么问题

✓ 适合的场景

  • 中等规模新功能（超过 3 个文件）
  • 需要在团队中对齐 AI 生成意图
  • 希望 AI 按步骤有节奏地实现，而不是一次性乱改
  • 需要可追溯的决策记录（design.md 存档原因）
  • 跨多个 AI 工具/模型切换使用的开发者
  • 现有项目（brownfield）的增量改造

✗ 不适合的场景

  • 一行代码 bug 修复——完全没必要写 proposal
  • 纯探索性实验代码（还不知道要做什么）
  • 高频小改动（每天 10+ 次 trivial change）
  • 需要多人实时协作同一 Change 的团队
  • 严格要求 CI/CD 与规格绑定的企业流程

3.2 最小可运行路径

# 1. 全局安装
npm install -g @fission-ai/openspec@latest

# 2. 进入项目并初始化（选择你使用的 AI 工具）
cd your-project && openspec init

# 3. 告诉 AI 你要做什么（slash 命令）
/opsx:propose add-dark-mode

# AI 自动创建 openspec/changes/add-dark-mode/
✓ proposal.md ✓ specs/ ✓ design.md ✓ tasks.md

# 4. 让 AI 开始实现
/opsx:apply

# 5. 完成后归档，Delta Spec 合并进主规格库
/opsx:archive

3.3 核心使用模式

🚀 快速提案模式（Propose）
当：有明确功能需求，想一步到位
运行 /opsx:propose <name>，AI 按 DAG 顺序依次生成 proposal → specs → design → tasks，全程无需人工干预，完成后进入 apply-ready 状态。
📝 分步对齐模式（New + Continue）
当：需求复杂，想逐步审阅每个 Artifact
先 /opsx:new <name> 创建 Change，AI 生成 proposal 后停下等人审阅，确认后 /opsx:continue 继续下一步。适合需要对规格有完全掌控的场景。
⚡ 快进模式（FF）
当：已有部分 Artifact，需要补全剩余
/opsx:ff 检查当前 Change 的完成状态，快速补全缺失的 Artifact 直到 apply-ready，跳过已有文件不重复生成。
🔄 验证同步模式（Verify + Sync）
当：实现完成需要验收，或规格与代码需同步
/opsx:verify 运行 UAT 检查列表，/opsx:sync 将代码中的实际变更反向同步到规格文档，保持 specs 与代码一致。

3.4 完整链路示例

以"为电商 App 添加收藏夹功能"为完整示例，展示从需求到归档的全链路：

Step 1 — 提案
用户在 Cursor 中输入 /opsx:propose add-favorites。AI 首先调用 openspec new change "add-favorites" 创建目录，再调用 openspec status --json 获取 DAG 构建顺序。
Step 2 — 生成 proposal.md
AI 调用 openspec instructions proposal --change add-favorites --json，获取 context（项目背景）、template（文档结构）和 instruction（写作规则）。按模板写出 proposal.md：为什么做、改什么、影响范围。
Step 3 — 生成 specs/ & design.md
proposal 完成后，ArtifactGraph 解锁 specs 和 design（它们都只依赖 proposal）。AI 并行创建 specs/favorites/spec.md（需求 + WHEN/THEN 场景）和 design.md（API 设计、数据结构）。Validator 检查规格是否包含 SHALL/MUST 和至少一个场景。
Step 4 — 生成 tasks.md
specs + design 完成后，AI 创建 tasks.md，包含勾选框格式的任务列表（- [ ] 1.1 Add favorites model）。ArtifactGraph 判断 apply.requires: [tasks] 已满足，状态变为 apply-ready。
Step 5 — 实现
/opsx:apply → AI 读取所有 context 文件，逐一完成 tasks.md 中的任务，每完成一个将 - [ ] 改为 - [x]，遇到问题暂停等待指导。
Step 6 — 归档
/opsx:archive → ArchiveCommand 将 specs/favorites/spec.md（Delta）中的 ADDED Requirements 合并进 openspec/specs/favorites/spec.md（主规格），整个 Change 目录移入 archive/2026-04-12-add-favorites/。

4.2 关键模块实现细节

4.2.1 ArtifactGraph — DAG 工作流引擎

核心模块

ArtifactGraph — Kahn 算法驱动的工作流调度

graph.ts · schema.ts · state.ts
要解决的问题

AI 要按正确顺序生成多个相互依赖的文档（proposal 必须先于 specs，specs 和 design 完成后才能写 tasks）。如何精确表达这种依赖关系，并在运行时判断"接下来应该做什么"？

选择了什么 · 为什么

使用有向无环图（DAG）建模 Artifact 依赖，用 Kahn 算法（入度为零的节点队列）计算拓扑排序，获得确定性的构建顺序。状态检测通过文件系统检查（不用数据库），每次查询都重新扫描文件是否存在。

// Kahn 算法核心：确定下一步可以构建的 Artifacts
getNextArtifacts(completed: CompletedSet): string[] {
  const ready: string[] = [];
  for (const artifact of this.artifacts.values()) {
    if (completed.has(artifact.id)) continue;
    const allDepsCompleted = artifact.requires.every(req => completed.has(req));
    if (allDepsCompleted) ready.push(artifact.id);
  }
  return ready.sort(); // 排序保证确定性
}

Schema 加载时还会做三重验证：重复 ID 检测 + 引用合法性检查 + DFS 循环依赖检测，确保工作流定义本身的完整性。

代价与局限

文件系统状态检测让 OpenSpec 在并发场景下完全不可靠——两个 AI 会话同时操作同一 Change 会导致状态错乱。Kahn 算法的确定性排序需要所有 Artifact ID 都已知，无法支持动态生成的 Artifact 数量（如"根据需求多少生成对应数量的 spec 文件"）。

4.2.2 SchemaResolver — 三层解析机制

扩展性

SchemaResolver — 优先级驱动的工作流配置解析

resolver.ts · types.ts
要解决的问题

不同项目、不同团队、不同用户需要不同的工作流（有人需要"极简 proposal+tasks"，企业用户需要"加入安全评审步骤"），但内置的默认工作流也需要对大多数用户开箱即用。如何既不破坏简单用户体验，又保留完全的可定制性？

选择了什么 · 为什么

三层优先级解析（类似 Git config 的 system/global/local 模式）：

  1. 项目本地：<projectRoot>/openspec/schemas/<name>/schema.yaml
  2. 用户全局：$XDG_DATA_HOME/openspec/schemas/<name>/schema.yaml
  3. 内置包：dist/schemas/<name>/schema.yaml（最低优先级）

Schema 文件本身是标准 YAML + Zod 验证，包含 Artifact 定义列表和 apply 阶段配置，完全可读可编辑。

代价与局限

三层解析让 Schema 来源难以追踪——当行为异常时，用户需要知道"到底用的是哪层的 Schema"。目前没有 openspec schema info 命令来展示当前生效的 Schema 来源，调试体验较差。

4.2.3 Validator — 规格质量门禁

质量保证

Validator — 多层次 Spec 格式强制检查

validator.ts · requirement-blocks.ts
要解决的问题

AI 生成的"规格"可能只是漂亮的 Markdown，但没有实质性的约束语言。一个写着"用户可以登录"的规格，跟一个写着"系统 SHALL 在 3 秒内完成认证并返回 JWT"的规格，质量天壤之别。

选择了什么 · 为什么

三层验证叠加：

  1. Zod 结构验证：Spec/Change 对象必须符合 TypeScript 类型定义，包含 Purpose/Requirements 等必要章节
  2. 结构规则检查：每个 Requirement 必须有至少一个场景（WHEN/THEN）；场景标题必须用 #### 而非 ###
  3. 语义规则强制：Requirement 文本必须包含 SHALL 或 MUST（排除 should/may 等弱约束）

Delta Spec 还有额外检查：同一 spec 内 ADDED/MODIFIED/REMOVED 不能有重复 Requirement，不能有跨区块冲突。

代价与局限

严格的格式要求（#### 场景标题、SHALL/MUST 强制）在实际使用中是最常见的踩坑点——AI 如果没有严格按 template 写，就会验证失败，且错误信息对用户来说不够直观。

4.2.4 CommandGeneration — 多工具适配引擎

生态扩展

CommandGeneration — 适配器模式驱动的斜线命令生成

generator.ts · registry.ts · adapters/
要解决的问题

Cursor、Claude Code、GitHub Copilot、Windsurf 等 30+ 工具各有不同的斜线命令文件格式（YAML frontmatter 字段不同、目录结构不同）。如何维护一套命令内容，同时支持所有工具？

选择了什么 · 为什么

经典的适配器模式（Adapter Pattern）：定义 ToolCommandAdapter 接口，每种工具实现两个方法 getFilePath(id) 和 formatFile(content)。命令内容（CommandContent）与格式生成完全解耦。

// 适配器接口 - 每个工具实现这两个方法
interface ToolCommandAdapter {
  toolId: string;
  getFilePath(commandId: string): string;  // 决定写到哪个文件
  formatFile(content: CommandContent): string; // 决定写什么格式
}

// Cursor 写到 .cursor/commands/opsx-*.md
// Claude 写到 .claude/commands/opsx/*.md
// Windsurf 写到 .windsurf/commands/opsx-*.md

CommandAdapterRegistry 用 Map 存储所有注册适配器，支持运行时通过 toolId 查找。

代价与局限

适配器数量随工具增长线性增加（目前 30+，还在增加）。每次工具更新命令格式规范，对应适配器就需要更新，维护成本不低。目前没有适配器的自动化测试覆盖，只能靠手工验证。

4.2.5 ArchiveCommand — Delta Spec 合并器

知识积累

ArchiveCommand — 增量规格归档与主库合并

archive.ts · specs-apply.ts · requirement-blocks.ts
要解决的问题

每次功能变更都会产生 Delta Spec（只写变化部分：ADDED/MODIFIED/REMOVED/RENAMED）。归档时这些增量变化需要精确地 patch 进主规格库（openspec/specs/），就像 Git patch 一样，但操作对象是 Markdown 结构化文档而非代码行。

选择了什么 · 为什么

解析 Delta Spec 的四种操作并逐一执行：

  • ADDED：将新 Requirement block 追加到主 spec 对应章节末尾
  • MODIFIED：用 normalizeRequirementName 做名称匹配，找到原 block 后整体替换
  • REMOVED：按名称找到 block，从主 spec 中删除整个 Requirement
  • RENAMED：找到 FROM 的 Requirement，将其标题改为 TO 的名称

在正式写入前，会用 Validator 对合并后的结果做一遍验证，防止 patch 导致结构破坏。

代价与局限

Delta 合并是基于文本解析的，当主规格库的 Requirement 标题有细微变化（空格、大小写）时可能匹配失败。normalizeRequirementName 做了一些容错，但复杂的重命名链条（A→B 再 B→C）在同一个 Change 中仍可能失败。

5 方案评价

5.1 优势分析

✓ 核心优势

工具无关的生态宽度 30+ 适配器覆盖几乎所有主流 AI 工具，这是 39.5k Star 背后的核心竞争力，也是竞品（Kiro 锁定 AWS IDE）无法复制的。
DAG 工作流精确调度 用有向无环图表达 Artifact 依赖比简单的"顺序步骤"更灵活——specs 和 design 可以并行，这对 AI 的效率有实质提升。
YAML Schema 全面可定制 工作流本身作为数据（schema.yaml），三层解析允许项目/用户/默认三级覆盖，扩展性极强。
零依赖状态追踪 文件系统即状态，不需要数据库，不需要守护进程。离线工作、系统崩溃后恢复都没有任何额外成本。
自食其果（Dogfooding） 项目自身的 openspec/ 目录显示它在用自己管理自己的开发，这是设计合理性的最强证明。

✗ 核心局限

AI 输出质量强依赖 OpenSpec 定义了结构，但具体 Artifact 内容质量完全取决于 AI + 用户提示。Delta Spec 格式复杂，AI 容易写出不符合格式要求（#### 场景标题、SHALL/MUST）的内容。
无法支持云端/团队协作 文件系统状态天然不支持多人并发操作同一 Change，也无法在 CI/CD 管道中自动触发工作流步骤。
适配器维护成本持续增长 每新增一个工具或工具更新命令格式，都需要更新适配器。30+ 适配器已经是不小的维护负担，测试覆盖也不够完整。
Delta Spec 合并脆弱 文本解析的 Requirement 匹配在边界情况下容易失败（标题细微变化、跨 Change 的冲突修改），一旦出错需要手动修复 Markdown。
工作流分裂（propose vs. new/continue） 核心工作流（propose 一步到位）和扩展工作流（new + continue + ff + verify + sync）并存，用户需要学习成本来判断使用哪套。
