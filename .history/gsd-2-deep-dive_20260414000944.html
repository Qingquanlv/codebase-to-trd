GSD 2 深度解读

基于磁盘状态机的自主编程 Agent — TypeScript + Pi SDK
GSD（Get Shit Done）2 是一个运行在终端里的自主编程 Agent，它的核心创新是把整个开发工作流的状态存在磁盘文件而非内存中，从而实现了崩溃恢复、多终端协同和无限长任务执行。读完本文，你将理解它与其他 AI 编程工具的本质差异、如何在项目中落地使用，以及那些让它"聪明"的关键工程决策。

核心结论

  • 磁盘即状态机 — 所有工作流进度存在 .gsd/ 目录，进程崩溃后从磁盘恢复，而不是重头开始
  • 每单元独立上下文 — 每个子任务（plan / execute / verify）都在全新的 200k token 窗口执行，彻底消除上下文污染
  • 声明式调度取代条件分支 — 用规则表（dispatch table）替换 130 行 if-else，"下一步做什么"变得可测试、可扩展
  • 亚毫秒复杂度分级驱动模型路由 — 纯启发式算法在调用 LLM 前判断任务难度，动态分配 Opus / Sonnet / Haiku，平衡质量与成本
  • Rust N-API 加速关键路径 — grep / glob / AST 解析 / git 操作通过 N-API 调用 Rust，绕过 Node.js I/O 瓶颈

1 摘要

1.1 项目背景

随着 LLM 能力提升，"让 AI 写代码"从聊天补全演进为自主 Agent。但现有工具面临两个根本性问题：一是上下文污染——长任务中历史对话越积越多，模型注意力被稀释，代码质量在几十轮后急剧下滑；二是状态易失——所有进度存在内存中，一次网络中断或进程崩溃就意味着一切重来。GSD 2 针对这两个痛点做了系统性设计。

1.2 项目预期

1.2.1 预期一：长任务持续推进能力

传统对话型 AI 工具的有效窗口通常在几十轮内。GSD 的目标是让 Agent 能稳定推进"新增功能模块"这类需要多天工作量的任务，关键实现是：把一个大需求拆解为 Milestone → Slice → Task 三层结构，每个最小单元（Unit）都在独立的全新 LLM 会话中执行，完成后结果写回磁盘，下个单元读磁盘而非历史对话获取上下文。

1.2.2 预期二：进程级可靠性

GSD 的 auto 模式可能持续运行数小时。通过锁文件（crash-recovery）+ JSONL 取证（session-forensics）+ OS 互斥锁（session-lock）三层保护，即使 Node.js 进程崩溃，下次启动时也能精确定位到崩溃位置并提示用户是否继续。

1.2.3 预期三：成本感知的自适应质量

项目内置了一套无需 LLM 调用的复杂度分级器：在调度每个 Unit 前，通过任务元数据启发式地判断 light / standard / heavy，并结合当前 budget 使用率和历史失败率动态调整分配的模型。目标是在预算耗尽前优先完成更多工作，而不是用最贵的模型处理所有任务。

2 项目设计理念

2.1 磁盘驱动状态机（Disk-Driven State Machine）

GSD 最根本的设计选择：所有工作流状态不存在内存里，只存在磁盘上。.gsd/ 目录下的 Markdown 文件（ROADMAP / SLICE / TASK / SUMMARY）加上 SQLite 数据库就是整个系统的唯一真相来源。

这个选择带来了一系列连锁收益：进程可以随时崩溃然后重启；用户可以同时在多个终端查看状态；Git worktree 隔离每个 Milestone 的开发分支；而且整个状态是人类可读的 Markdown，用户随时可以手动编辑干预。代价是磁盘 I/O 变成热路径，这也是引入 Rust 原生引擎的直接原因。

2.2 每单元新鲜会话（Fresh Session Per Unit）

对话型 AI 工具的通病是"历史越长，质量越差"。GSD 的解法是彻底的：每个 Unit（Task / Slice / Milestone 级别）都在一个全新的 LLM 会话（新的 context window）里执行。上一个 Unit 的输出不是通过对话历史传递，而是写成 SUMMARY 文件，下一个 Unit 的 Prompt 里按需 inline 这个文件的内容。

这种设计的精妙之处在于"按需"两字——auto-prompts.ts 里有一套分层上下文注入机制，根据 inline_level（full / references / compressed）和当前 context budget 决定把多少历史内容塞进 Prompt，而不是无脑全部 inline。

2.3 声明式架构理念（竞品横向对比）

GSD 在多个核心模块都选择了声明式（declarative）而非命令式（imperative）的设计：dispatch table 替代 if-else 链、规则化的 TTSR guardrails、schema 驱动的状态类型。这反映了一个设计判断：可观察性和可测试性比"快速写完"更重要。

3 项目使用方式与执行链路

3.1 当前适合解决什么问题

✅ 适合的任务类型

  • 中大型功能开发 — 如"新增用户权限系统"，需要设计、实现、测试多个步骤
  • 无人值守批量推进 — 可以挂在后台运行，完成一个 Slice 后自动继续
  • 需要干净 Git 历史的场景 — 每个 Milestone 一个独立分支，squash-merge 保持主分支整洁
  • 跨终端协同 — 磁盘状态可跨终端查看，多人项目可共享 .gsd/ 目录状态
  • 高可靠性要求的自动化场景 — CI/CD headless 模式、定时任务驱动的自动维护

⚠️ 不太适合的任务类型

  • 五分钟内的小修改 — 创建 Milestone 的开销大于任务本身
  • 完全开放式的探索性研究 — GSD 需要明确的目标和可验证的完成标准
  • 纯设计 / 头脑风暴 — 需要人机高频交互的创意过程不适合自主循环
  • 需要即时视觉反馈的前端调试 — 当前版本对 UI 像素级调整的理解有限

3.2 最小可运行路径

从零开始，用 GSD 完成第一个自主任务只需 5 步：

# 第一步：安装
npm install -g gsd-pi
# 第二步：进入你的项目目录，首次运行会触发 onboarding（配置 LLM API Key）
cd my-project && gsd
# 第三步：创建一个 Milestone（描述你的目标）
/gsd discuss "为用户登录接口添加失败次数限制功能"
# 第四步：启动自主执行
/gsd auto
# 第五步：查看进度（另一个终端或 Web UI）
gsd web # 打开浏览器 Dashboard

3.3 核心使用模式

🪜 Step 模式（交互逐步）
何时用 想要控制每一步的输出，或第一次使用某个新 Milestone 时
输入 /gsd step，每次执行一个 Unit 后等待用户确认
预期输出 一次一个 Task 的代码变更 + SUMMARY，人工审核后继续
🤖 Auto 模式（无人值守）
何时用 任务拆解清晰、测试覆盖好、可以放心跑完整个 Milestone
输入 /gsd auto，Agent 自动循环直至 Milestone 完成或遇到阻塞
预期输出 完整的代码实现 + 测试 + SUMMARY，以及一个新的 git commit
🔧 Headless 模式（CI/CD）
何时用 需要在无 TTY 环境中运行，如 GitHub Actions、定时脚本
输入 gsd --headless auto，配合 stdin JSON 预置答案
预期输出 结构化 JSON 事件流 + 退出码（0 = 成功），可被 CI 消费

3.4 从需求到交付的一次完整链路

以"为现有系统新增用户登录失败次数限制"为例，追踪一次完整的 GSD 执行链路：

① 用户提出目标
用户运行 /gsd discuss "为登录接口添加失败次数限制，5次后锁定15分钟"，GSD 启动一个 discuss-milestone Unit，与用户澄清边界：是 Redis 还是数据库存计数？锁定是软锁还是硬锁？
② Plan Milestone
discuss 完成后，自动触发 plan-milestone Unit。Agent 读取讨论结果，生成 ROADMAP.md：3 个 Slice——"数据模型变更"、"接口逻辑实现"、"集成测试"。状态写入 .gsd/milestones/M01/ROADMAP.md 和 SQLite。
③ Plan Slice（逐个）
对每个 Slice，GSD 分配一个全新会话执行 plan-slice。Agent 读取 ROADMAP inline 到 Prompt，输出 SLICE.md：具体到文件级别的任务列表，如"修改 User 表添加 login_attempt_count 字段"。
④ Execute Tasks（新鲜上下文）
每个 Task 在独立新会话中执行。auto-prompts.ts 根据 inline_level 配置决定把多少 SLICE 内容塞进 Prompt（full / references / compressed），复杂度分级器决定用 Sonnet 还是 Opus 执行。代码变更直接提交到 Milestone 专属的 git worktree。
⑤ Run UAT / 写入 SUMMARY
所有 Task 完成后，run-uat Unit 执行测试命令，写入 verdict（PASS / FAIL）。Slice 完成后生成 SUMMARY.md 作为下个 Slice 的"压缩历史记忆"。
⑥ Worktree Merge + 继续
Slice 全部通过后，Milestone 的 git worktree squash-merge 回主分支。调度器自动推进到下一个 Milestone，或以 info 级别 stop 通知用户所有工作已完成。整个过程用户可以随时打断，下次 /gsd auto 从磁盘状态精确续行。

4.1.2 目录与核心文件说明

【启动与引导层】 ★ src/loader.ts · src/cli.ts · src/resource-loader.ts · src/tool-bootstrap.ts

为什么存在：必须在 Pi SDK 任何模块求值之前完成品牌名称（PI_PACKAGE_DIR）和运行时路径的注入；没有这层，SDK 无法知道自己是 GSD 而不是原始 Pi，工具路径（fd/rg 可执行文件）也会因平台差异而找不到。"两文件加载器"（loader → cli）的架构专门为此设计。

文件职责
★ src/loader.ts快速路径：--version / --help 在任何 import 前处理；设置 PI_PACKAGE_DIR / GSD_VERSION / GSD_BIN_PATH 三个关键环境变量；最后动态 import cli.ts
src/cli.ts主 CLI 入口：解析 argv，检测运行模式（interactive / headless / web / rpc），初始化扩展注册器，启动 TUI 或无头执行器
src/resource-loader.ts初始化并校验 bundled resources（prompt 模板、扩展包），首次运行时同步到 GSD_HOME
src/tool-bootstrap.ts探测 fd / rg 可执行文件是否可用，不可用时回退到内建版本

【工作流引擎层】 ★ src/resources/extensions/gsd/auto.ts · auto-dispatch.ts · auto-prompts.ts · auto-loop.ts · auto-verification.ts

为什么存在：这一层是 GSD 区别于普通 LLM wrapper 的本质所在——它把"问题 → 规划 → 执行 → 验证"变成一个持久的磁盘驱动状态机循环，而不是一次性对话。没有这层，用户的每次交互都是独立的、无记忆的；有了这层，一个跨越数天的复杂需求可以被 Agent 自主推进到完成。

文件职责
★ auto.ts状态机主循环的编排者：读取 GSDState → 调用 dispatch → 分配模型 → 开启新 Pi 会话 → 注入 Prompt → 等待 agent_end → 循环
auto-dispatch.ts声明式规则调度表（替代 130 行 if-else）：每条规则有 name + match 函数，顺序评估第一个匹配的规则决定下一个 Unit 类型
auto-prompts.ts分层上下文注入引擎：按 inline_level 和 context budget 决定把多少历史文档 inline 进 Prompt，内置 30k 字符 preamble 上限
auto-loop.ts单次 auto 迭代的执行体：处理 session 生命周期、Pi 事件流、超时恢复
auto-verification.ts强制验证阶段：每个 Slice 完成后运行 UAT 命令并检查 verdict 文件，防止未验证的代码悄悄通过

【状态存储层】 ★ src/resources/extensions/gsd/state.ts · gsd-db.ts · paths.ts · .gsd/（用户项目目录）

为什么存在：所有工作流进度必须在进程崩溃、终端关闭、机器重启后仍然完整保留。把状态放进磁盘文件（Markdown + SQLite）而不是内存，是实现崩溃恢复、多终端协同和人工干预的基础。这层是整个系统"可靠性"承诺的物理实现。

文件/目录职责
★ state.tsDB 优先的状态派生：从 SQLite 读取结构化状态，对未迁移项目回退到解析 Markdown 文件，输出统一的 GSDState 对象
gsd-db.tsSQLite 封装层：Milestone / Slice / Task 的 CRUD，提供事务性写入，是状态的权威存储
paths.ts所有 .gsd/ 路径的唯一解析器：resolveMilestoneFile / resolveSliceFile / resolveTaskFile 等，防止路径散落各处
.gsd/milestones/每个 Milestone 一个子目录，含 ROADMAP.md / CONTEXT.md / SUMMARY.md 和若干 Slice 子目录
.gsd/runtime/临时运行时文件：锁文件、rewrite 计数器、UAT 计数器，进程间通过文件系统通信

【智能路由层】 src/resources/extensions/gsd/complexity-classifier.ts · auto-model-selection.ts · context-budget.ts · routing-history.ts

为什么存在：大模型 API 成本差异悬殊（Opus vs Haiku 约 15 倍），不能所有任务都用最贵的模型。这层在调用任何 LLM 前完成"任务难度评估 + 预算感知 + 历史失败率学习"，动态分配模型，在质量和成本之间找到最优平衡。

文件职责
complexity-classifier.ts纯启发式复杂度分级（无 LLM 调用）：基于 unit 类型、任务元数据（文件数/依赖数/代码块数）、关键词检测，亚毫秒输出 light/standard/heavy
auto-model-selection.ts综合分级结果、budget 压力（>90% 强降级为 light）、用户偏好，最终选定 LLM provider 和 model ID
context-budget.ts根据 context window 大小计算各部分的 token 预算（preamble / inline context / 任务数量范围）
routing-history.ts自适应学习：记录每个 unit 类型在各 tier 的失败率，高失败率时自动上调分级

【崩溃恢复层】 ★ src/resources/extensions/gsd/crash-recovery.ts · session-forensics.ts · session-lock.ts · interrupted-session.ts

为什么存在：auto 模式的单次执行单元可能持续 5-30 分钟；网络中断、OOM、用户 Ctrl-C 随时可能发生。没有这层，任何崩溃都意味着丢失当前 Unit 的全部进度，必须重新执行。这层将"崩溃"变成了一个可恢复的检查点，而不是灾难。

文件职责
★ crash-recovery.ts锁文件写/读/删除：每次 Unit 开始写入含 PID + sessionFile 路径的锁，正常结束删锁，下次启动检测残锁判断是否崩溃
session-forensics.tsJSONL 取证：从崩溃时的 Pi session 文件中提取已完成的工具调用，重建"崩溃前最后状态"摘要供用户参考
session-lock.tsOS 级互斥锁（flock）：防止同一项目两个 auto 进程并发执行，配合锁文件实现双重保护
interrupted-session.ts综合评估中断类型：区分"崩溃"和"用户主动暂停"，为每种情况提供不同的恢复建议文本

【原生性能层】 native/ · packages/native/ · src/resources/extensions/gsd/native-parser-bridge.ts

为什么存在：GSD 的状态机每轮循环都需要批量解析 .gsd/ 目录下的大量 Markdown 文件；auto-prompts 需要频繁 grep 代码库寻找相关上下文。Node.js 的单线程 I/O 在这些高频场景下成为明显瓶颈。通过 N-API Rust 模块将这些操作下放到原生层，可将关键路径提速 10-50 倍。

【能力扩展层】 src/resources/extensions/ · src/extension-registry.ts

4.2.1 关键模块一：声明式调度管道（auto-dispatch.ts）

📁 src/resources/extensions/gsd/auto-dispatch.ts 🏷️ 规则表模式（Chain of Responsibility 变体）

要解决的问题：GSD 的工作流有 10+ 种 Unit 类型（discuss / plan / execute / uat / complete 等），决定"下一步该执行哪个"需要考虑当前 Milestone 状态、Slice 状态、是否有崩溃残留、预算是否超限……用命令式 if-else 实现会产生一个数百行的怪兽函数，无法单独测试每条规则，添加新规则会引入隐患。

解决方案：把所有调度逻辑提取为一张有序的 DispatchRule[] 数组，每条规则有 name 和 async match 函数。match 返回 DispatchAction（dispatch / stop / skip）或 null（不匹配，继续下一条）。调度引擎只需顺序评估规则表，第一条返回非 null 的规则获胜。

// 文件: src/resources/extensions/gsd/auto-dispatch.ts (核心数据结构)
export type DispatchAction =
  | { action: "dispatch"; unitType: string; unitId: string; prompt: string; matchedRule?: string }
  | { action: "stop"; reason: string; level: "info" | "warning" | "error" }
  | { action: "skip"; matchedRule?: string };

export interface DispatchRule {
  name: string;                                          // 可读名称，用于日志和测试
  match: (ctx: DispatchContext) => Promise<DispatchAction | null>; // null = 不匹配，继续评估
}

// 规则按优先级排列，顺序至关重要
const DISPATCH_RULES: DispatchRule[] = [
  { name: "check-crash-recovery",  match: async (ctx) => { /* 最高优先：检测崩溃残留 */ } },
  { name: "check-budget-ceiling",  match: async (ctx) => { /* 预算超限则 stop */ } },
  { name: "check-active-override", match: async (ctx) => { /* 用户手动覆盖命令 */ } },
  // ... 10+ 条业务规则
  { name: "execute-next-task",     match: async (ctx) => { /* 默认：执行下一个待办 Task */ } },
];

// 调度引擎：极简的顺序评估
export async function evaluateDispatchRules(ctx: DispatchContext): Promise<DispatchAction> {
  for (const rule of DISPATCH_RULES) {
    const result = await rule.match(ctx);  // null = 本规则不适用
    if (result) return { ...result, matchedRule: rule.name };
  }
  return { action: "stop", reason: "no matching rule", level: "warning" };
}

代价与局限：规则的顺序本身就是隐式的业务逻辑，乱序会导致静默错误（如崩溃检测放在预算检测后面会漏掉崩溃场景）。规则条数增多后，调试"为什么选了这条规则"需要靠日志中的 matchedRule 字段追溯。

4.2.2 关键模块二：复杂度分级器（complexity-classifier.ts）

📁 src/resources/extensions/gsd/complexity-classifier.ts 🏷️ 启发式分级 + 自适应学习

要解决的问题：模型路由需要知道"这个 Task 有多难"，但用 LLM 来判断难度有明显的自我矛盾（贵的 LLM 才能判准难度，用了就没省到）。需要一个零 LLM 调用的快速分级器。

解决方案：两阶段判断。第一阶段按 unit 类型给默认分级（plan-milestone = heavy, complete-slice = light）；第二阶段对 execute-task 类型做元数据分析：读取 TASK.md 的文件引用数、依赖数、代码块数量、复杂度关键词，并结合历史失败率做自适应上调；最后用当前 budget 使用率做一次强制降级（90%+ 时 heavy → light）。

// 文件: src/resources/extensions/gsd/complexity-classifier.ts (核心逻辑)
const UNIT_TYPE_TIERS: Record<string, ComplexityTier> = {
  "complete-slice":    "light",   // 收尾工作，不需要深度推理
  "run-uat":           "light",   // 执行验证命令，结果写文件
  "discuss-milestone": "standard",
  "plan-slice":        "heavy",   // 规划需要全局视角
  "execute-task":      "standard",// 默认 standard，元数据分析可上调
  "reassess-roadmap":  "heavy",   // 重新评估整体方向
}

function applyBudgetPressure(result: ClassificationResult, budgetPct?: number): ClassificationResult {
  if (!budgetPct) return result;
  let tier = result.tier;
  // 强制降级：预算超 90% 时即使是 heavy 任务也用最便宜的模型
  if (budgetPct >= 0.90 && tier === "heavy")    tier = "light";
  else if (budgetPct >= 0.75 && tier === "heavy") tier = "standard";
  else if (budgetPct >= 0.50 && tier === "heavy") tier = "standard";
  return { ...result, tier, downgraded: tier !== result.tier };
}

代价与局限：启发式规则会有误判——一个关键词很少但实际上修改面很广的重构任务可能被错误分为 light。自适应学习的粒度是 unit 类型级别，无法区分同一类型内不同任务的差异。

4.2.3 关键模块三：崩溃恢复机制（crash-recovery.ts）

📁 src/resources/extensions/gsd/crash-recovery.ts + session-forensics.ts 🏷️ 心跳锁文件 + JSONL 取证

要解决的问题：auto 模式的单个 Unit 执行可能持续数分钟到数十分钟。网络中断、内存溢出、用户强制关闭都会导致进程非正常退出。如何在下次启动时让用户无缝感知到"上次崩溃了"并提供恢复选项？

解决方案：三层嵌套保护。第一层：每次 Unit 开始前在 .gsd/runtime/auto.lock 写入含 PID + sessionFile 路径的 JSON 锁文件，正常退出时删除；下次启动检测到锁文件且 PID 进程已不存在 → 判定崩溃。第二层：从崩溃时的 Pi session JSONL 文件读取已完成的工具调用，重建崩溃前状态摘要（session-forensics）。第三层：OS 级 flock 互斥锁防止并发执行。

// 文件: src/resources/extensions/gsd/crash-recovery.ts (核心逻辑)
export function isLockProcessAlive(lock: LockData): boolean {
  const pid = lock.pid;
  if (pid === process.pid) return true;  // 同一进程，肯定活着
  try {
    process.kill(pid, 0);  // 信号 0 = 探活（不发送真实信号）
    return true;
  } catch (err) {
    // ESRCH = 进程不存在（崩溃）
    // EPERM = 进程存在但无权限（仍然活着）
    return (err as NodeJS.ErrnoException).code === 'EPERM';
  }
}

// Pi 的 session JSONL 是追加写入的，每行一个事件
// 崩溃后文件在磁盘上完整保留到崩溃点——这是 appendFileSync 的关键价值
export async function synthesizeCrashRecovery(sessionFile: string): Promise<string> {
  const lines = readFileSync(sessionFile, 'utf-8').split('\n').filter(Boolean);
  const toolCalls = lines
    .map(l => JSON.parse(l))
    .filter(e => e.type === 'tool_result');  // 提取已完成的工具调用
  // 从工具调用结果重建"崩溃前做了什么"的摘要
  return formatCrashSummary(toolCalls);
}

代价与局限：锁文件的 PID 检测在分布式环境（多台机器共享 .gsd/ 目录的场景）下不可靠，但 GSD 主要针对单机开发场景，这个限制可接受。Session JSONL 取证的质量依赖 Pi SDK 的日志完整性。

4.2.4 关键模块四：分层上下文注入（auto-prompts.ts）

📁 src/resources/extensions/gsd/auto-prompts.ts 🏷️ 分层预算分配 + 优雅降级

要解决的问题：每个 Unit 在独立新会话执行，但它需要了解当前任务的上下文——所在 Milestone 的目标、所属 Slice 的计划、关联 Task 的详情。把所有历史文档全部 inline 会超出 context window；什么都不 inline 则 Agent 缺乏背景会走弯路。

解决方案：三档 inline_level 配置（full / references / compressed）+ 30k 字符 preamble 上限的自动裁剪。full 模式把完整 Markdown 文件 inline；references 模式只 inline 文件路径（告诉 Agent "去读这个文件"）；compressed 模式只 inline 文件的关键摘要段落。超出 30k 上限时自动按章节边界截断，而不是硬截字符。

// 文件: src/resources/extensions/gsd/auto-prompts.ts (关键逻辑)
const MAX_PREAMBLE_CHARS = 30_000;  // preamble 硬上限，防止挤占 task 内容空间

function capPreamble(preamble: string): string {
  if (preamble.length <= MAX_PREAMBLE_CHARS) return preamble;
  // 按章节边界截断（## 开头的行），而不是硬截字符
  // 避免截断到句子中间产生语义损坏的 Prompt
  return truncateAtSectionBoundary(preamble, MAX_PREAMBLE_CHARS).content;
}

function formatExecutorConstraints(): string {
  const prefs = loadEffectiveGSDPreferences();
  const windowTokens = resolveExecutorContextWindow(undefined, prefs?.preferences);
  const budgets = computeBudgets(windowTokens);
  // 在 Prompt 中明确告知 Agent 它的上下文窗口大小和建议任务数量范围
  // Agent 可以据此自行调整输出密度，避免单个 task 写太多/太少
  return [
    `## Executor Context Constraints`,
    `The agent has a **${Math.round(windowTokens / 1000)}K token** context window.`,
    `- Recommended task count: **${budgets.taskCountRange.min}–${budgets.taskCountRange.max} tasks**`,
    `- Each task gets ~${Math.round(budgets.inlineContextBudgetChars / 1000)}K chars inline context`,
  ].join("\n");
}

代价与局限：inline_level = "references" 要求 Agent 有足够的工具调用能力主动去读文件；如果 Agent 不读或读错，任务质量会下降。compressed 模式的摘要质量依赖于 SUMMARY 文件是否准确，而 SUMMARY 本身是之前 Agent 生成的——存在"错误传播"的风险。

5 方案评价

5.1 优势分析

  • 磁盘状态机是真正的差异化：目前市场上很少有工具能做到跨会话、跨终端、跨崩溃的持久状态，这是 GSD 在"长任务可靠执行"场景的根本护城河
  • 声明式调度的工程严谨性：把调度逻辑变成可测试的规则表，每条规则独立，可以 unit test，这在同类工具中罕见
  • 成本感知是一等公民：复杂度分级 + 预算压力降级的组合，让 GSD 在有限预算下可以完成更多工作，不是简单地"永远用最贵的"
  • Rust 原生引擎的务实选择：没有为了"纯 TS" 而坚持，在性能关键路径引入 Rust 是正确的工程判断
  • 扩展架构良好：24 个扩展相互独立，主引擎不依赖具体扩展，添加新能力无需修改核心代码

5.2 局限分析

  • 依赖闭源 Pi SDK：核心 Agent 运行时（会话管理、工具执行、LLM 路由）封装在 @gsd/pi-coding-agent 里，这是一个闭源包，限制了社区贡献和问题排查深度
  • 初始学习曲线陡峭：Milestone / Slice / Task 三层结构对于"只想快速执行一个命令"的用户来说过重；onboarding 文档对新用户不够友好
  • 启发式分级的天花板：复杂度分级器无法感知代码库的全局状态，"修改一个被 50 个地方引用的核心接口"和"新增一个独立工具函数"可能被分到同一级别
  • SUMMARY 质量的链式风险：每个 Slice 的 SUMMARY 由 Agent 自己生成并作为下个 Slice 的上下文，一次质量差的 SUMMARY 会影响后续所有 Slice 的执行质量
  • Git worktree 的操作复杂性：每个 Milestone 一个 worktree 对不熟悉 git worktree 的用户来说是心智负担，worktree 冲突处理不够自动化
